# Event Template

## 什么时候值得写事件

只有下列情况才值得落 `events/*.md`：

- 财报或业绩会
- 指引变化
- 关键经营指标出现拐点
- 管理层 / 治理变化
- 大客户签约 / 流失
- 资本配置动作
- 行业结构变化
- 出现支持或证伪旧 thesis 的新事实

如果只是重复旧信息，或只是价格波动，不要写事件。

## 事件正文应回答的问题

当前实现建议把事件写成以下结构：

1. `发生了什么`
2. `为什么重要`
3. `影响哪些画像 section`
4. `对 thesis 的影响`
5. `证据与来源`
6. `本轮研究路径`
7. `需要继续跟踪什么`

## 字段写法建议

### 发生了什么

- 客观描述新事实
- 不要在这里混太多结论

### 为什么重要

- 说明为什么这件事值得写进长期画像
- 最好直接指向商业模式、护城河、财务质量、资本配置或估值框架

### 对 thesis 的影响

- 说明它是强化、削弱、混合还是暂时中性
- 如果是 mixed，具体写“哪部分变好，哪部分变差”

### 证据与来源

- 写使用的文件、电话会、财报、新闻、公告
- `refs` 可以留链接、文件 id、标题或 query 命中项
- 如有必要，可补一句“为什么这些来源足以支持当前判断”

### 本轮研究路径

- 简短记录本轮怎么研究出来的
- 可以写：
  - 查询词
  - 看了哪些资料
  - 排除了哪些噪音
  - 哪些信息最终没有写回画像

### 需要继续跟踪什么

- 明确下一步要看什么
- 尽量写成可观察变量，而不是泛泛而谈

## 判断是否回写主画像

写完事件后，再问一次：

- 这次只是新增事实，还是已经改变长期判断？

如果已经改变长期判断，再回写 `profile.md` 的相关 section。

如果还没有改变，只保留事件，不要为了“更新而更新”去重写主画像。
