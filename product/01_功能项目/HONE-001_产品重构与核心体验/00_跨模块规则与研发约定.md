# HONE-001 跨模块规则与研发约定

状态：`draft`
适用模块：投资、洞察、Agent、跟踪、我的
更新时间：2026-07-12

## 1. 文档作用

本文件定义五个模块共同遵守的产品规则。各模块 PRD 只描述本模块特有逻辑；出现冲突时，以本文件和总 PRD为准。

本轮需求描述产品行为和验收口径，不指定接口、表结构、组件库或技术实现。研发可复用现有能力，但最终行为必须满足需求。

## 2. 版本优先级

| 级别 | 含义 |
|---|---|
| P0 | 第一版闭环必需；缺失会导致主要旅程不可用 |
| P1 | 第一版稳定后补充；显著提升效率和完整性 |
| P2 | 后续增强；不阻塞核心闭环 |

每项功能同时标注实现属性：

- `复用`：当前代码已有主体能力，主要做入口或体验重组。
- `改造`：已有部分能力，需要改变数据关系或用户流程。
- `新增`：当前产品没有完整能力，需要新建。

## 3. 全局信息架构

### 3.1 一级导航

移动端固定顺序：

```text
投资　　洞察　　Agent　　跟踪　　我的
```

| 编号 | 要求 | 优先级 |
|---|---|---|
| GLO-001 | 登录后默认进入“投资” | P0 |
| GLO-002 | 移动端使用五项底部导航，Agent 位于中间并突出，但必须保留文字标签 | P0 |
| GLO-003 | Web/macOS 使用分层侧栏：品牌下方优先放投资、洞察、跟踪工作台；其后放新研究、研究记录搜索、最近对话和全部记录；账户头像与设置固定在左下角；不再设置独立 Agent 跳转卡片和“返回官网” | P0 |
| GLO-004 | 切换一级入口后保留各模块最近浏览位置；退出登录后清除本地敏感状态 | P1 |
| GLO-005 | 当前入口、按钮和焦点状态必须有不依赖颜色的可识别反馈 | P0 |

### 3.2 全局辅助入口

| 编号 | 要求 | 优先级 |
|---|---|---|
| GLO-006 | 提供全局搜索，至少支持公司/ticker、洞察；历史对话和研究结果后续接入 | P1 |
| GLO-007 | 提供未读提醒入口；提醒详情统一进入“跟踪” | P0 |
| GLO-008 | 公司、洞察、事件、任务详情均可发起带上下文的 Agent 对话 | P0 |
| GLO-008A | 桌面端进入“我的”时，左下角账户入口需要显示当前状态；侧栏不重复出现“我的”；桌面顶栏右侧只保留全局搜索 | P0 |
| GLO-008B | 桌面侧栏研究记录可搜索、可选择并打开对应 Agent 会话；“新研究”直接重置到 Agent 默认研究态 | P0 |

## 4. 核心对象与唯一标识

| 对象 | 最低要求 | 关联模块 |
|---|---|---|
| 公司 | 稳定公司 ID、ticker、交易所、公司名；ticker 不能单独作为永久主键 | 投资、洞察、Agent、跟踪 |
| 用户投资上下文 | 用户/actor、持仓、自选、投资主线、偏好 | 投资、Agent、跟踪 |
| 洞察 | 内容 ID、类型、作者、发布时间、权限、关联公司/主题 | 洞察、投资、Agent |
| 事件 | 事件 ID、类型、发生/预计时间、来源、关联公司、状态 | 投资、跟踪、Agent |
| Agent 会话 | 会话 ID、用户、上下文来源、消息和附件 | Agent及所有上下文入口 |
| 跟踪任务 | 任务 ID、触发规则、对象、状态、最近/下次执行时间 | Agent、跟踪 |
| 研究结果 | 结果 ID、来源任务/会话、生成时间、关联公司、可见性 | Agent、跟踪、投资 |

### 4.1 关联规则

- 公司是跨模块主连接对象；同一家公司在不同交易所或存在多类证券时必须可区分。
- 洞察、事件、任务和研究结果可以关联零个、一个或多个公司。
- 用户私有数据不得因为公司关联而对其他用户可见。
- 删除自选或持仓不自动删除历史洞察、任务和研究结果；需要单独处理关联。

## 5. 权限与可见性

| 编号 | 规则 | 优先级 |
|---|---|---|
| GLO-009 | 未登录用户只可访问明确公开的官网和公开洞察；投资、Agent、跟踪、我的需要登录 | P0 |
| GLO-010 | 洞察必须支持公开、登录可见、会员可见、内部四种权限 | P0 |
| GLO-011 | 权限不足时展示内容摘要和明确的权限说明，不返回正文数据 | P0 |
| GLO-012 | 持仓、成本、收益、投资主线、会话和任务均为用户私有 | P0 |
| GLO-013 | 管理员可在授权后台查看和维护数据，但普通用户端不得暴露管理员入口 | P0 |
| GLO-014 | 会员状态第一阶段可由后台赋予；支付和自动续费不属于本轮 | P0 |

## 6. 数据与时间规则

### 6.1 时间

- 数据层保存标准时间与时区；界面按用户时区显示。
- 公司财报、市场开收盘等使用对应交易所时区，同时显示用户本地时间。
- 全天事件与具体时刻事件必须区分。
- 日期跨日、夏令时切换和用户更换时区后，日历不得重复或遗漏事件。

### 6.2 行情与财务数据

- 所有行情、估值、财务和事件必须记录来源与更新时间。
- 延迟行情必须明确标注“延迟”及更新时间，不得伪装为实时。
- 数据源失败时保留最近一次成功数据并标记“可能已过期”；不得显示为正常最新值。
- 计算收益时必须明确币种、成本口径和是否包含现金流；第一阶段不处理券商级税费和复杂公司行动回溯。

### 6.3 内容与 AI

- 老王原始观点、编辑加工内容和 AI 生成内容必须可区分。
- AI 对内容的摘要、解释和推断不得覆盖原文。
- 时间敏感回答必须展示数据时间；无法获得最新数据时明确说明。

## 7. 通用页面状态

每个列表和详情页必须实现：

| 状态 | 统一要求 |
|---|---|
| 首次加载 | 展示与页面结构一致的加载状态，不使用无限转圈遮挡整个应用 |
| 空状态 | 解释为什么为空，并提供一个明确下一步操作 |
| 局部失败 | 其他已成功区域继续可用，失败区域可重试 |
| 全页失败 | 说明失败原因类型并提供重试；不得丢失用户尚未提交的输入 |
| 离线 | 可查看已缓存内容时明确标注；写操作进入待重试或明确失败，不假装成功 |
| 权限不足 | 展示可见摘要、所需权限和返回路径 |
| 数据过期 | 展示最后更新时间与过期状态 |

## 8. 写操作与一致性

- 新增/修改持仓、自选、任务、偏好等写操作必须有成功或失败反馈。
- 影响长期数据或主动服务的高风险操作需要确认：删除持仓、删除任务、清空历史、退出全部设备。
- 重复点击、网络重试或客户端恢复不得创建重复持仓、重复任务或重复内容。
- 乐观更新失败时回滚界面，并保留用户输入方便重试。
- 多端同时修改时，服务端版本为最终结果；出现冲突必须提示用户刷新或重新确认，不静默覆盖关键数据。

## 9. Agent 上下文规则

| 来源 | 自动携带的最低上下文 |
|---|---|
| 公司详情 | 公司 ID、ticker、当前页面模块、用户是否持仓/自选 |
| 洞察详情 | 洞察 ID、标题、作者、权限范围、关联公司 |
| 跟踪事件 | 事件/任务 ID、时间、状态、关联公司 |
| 投资首页 | 用户主动选择的组合范围；不得默认把全部精确持仓发给不受信任的外部模型 |

- 界面必须告诉用户当前 Agent 正在基于什么上下文回答，并允许移除。
- 私有上下文只在当前用户权限范围内读取。
- 从受限洞察进入 Agent 时，模型仅能读取用户有权访问的内容。

## 10. 跨端要求

- Web、macOS、iOS 共享业务数据与账号状态，不要求共享所有本地显示设置。
- 移动端优先单列和底部导航；大屏可使用列表+详情+Agent 辅助栏。
- 核心功能不能依赖 hover；触控目标需适合移动端操作。
- macOS/iOS 远程壳离线时必须展示可恢复状态，不承诺离线 Agent。
- 系统级推送在实现前，产品文案只能承诺站内提醒；接入原生推送后再单独验收授权、撤回和到达逻辑。

## 11. 埋点与基础观测

P0 至少记录以下不含正文和精确持仓的产品事件：

- 一级入口访问。
- 公司详情访问。
- 洞察曝光、打开、读完、收藏。
- Agent 发起、成功、失败、终止及上下文来源。
- 持仓/自选新增与删除。
- 跟踪任务创建、触发、送达、打开与失败。
- 登录成功/失败、权限拦截。

不得将对话正文、附件内容、手机号、精确持仓和成本直接写入通用分析埋点。

## 12. 统一完成标准

一项 P0 功能只有同时满足以下条件才算完成：

- 主流程、空状态、失败状态和权限状态均已实现。
- Web、macOS、iOS 目标端行为符合本文件定义。
- 写操作具备幂等或防重复逻辑。
- 关键事件有必要观测，不记录敏感正文。
- 通过产品验收用例，并标明复用、改造或新增的代码范围。
- 不破坏现有 Agent、登录和用户数据；涉及迁移时提供回滚方案。
