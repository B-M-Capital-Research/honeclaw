状态：`draft`

## 1. 需求背景

财报、宏观数据、公司公告、价格异动和研究任务都具有时间属性。当前这些信息分散在聊天、站内推送和后台任务中，用户很难知道“今天该看什么、未来会发生什么、过去错过了什么”。

跟踪模块用时间和关注关系组织主动服务。日历是核心视图，但模块还需要容纳任务、即时提醒和历史结果。

## 2. 解决的问题

- 把与持仓、自选和投资主线相关的事件聚合到今日和日历。
- 统一管理 Agent 创建的提醒、监控和周期任务。
- 让主动结果可回看、可继续研究，而不是一次性通知。
- 让用户控制提醒类型、时间、频率和勿扰规则。

## 3. 用户与范围

### 用户

- 有持仓、自选或长期研究对象的用户。
- 使用 Agent 创建监控和周期研究任务的用户。

### 本期范围

- P0：今日、日历、任务列表/详情、历史结果、站内提醒、偏好/勿扰、Agent 闭环。
- P1：原生推送、多渠道选择、日历订阅导出、任务模板。
- P2：复杂条件组合、团队共享跟踪、外部日历双向同步。

### 不做

- 通用个人日程和会议管理。
- 未经用户同意自动创建长期任务。
- 在未完成原生推送时承诺 App 关闭也一定收到系统通知。

## 4. 信息架构

```text
跟踪
├── 今日
├── 日历
├── 任务
└── 历史
```

- 移动端默认进入“今日”。
- Web/macOS 可同时展示时间线、日历和详情。
- 未读提醒入口最终打开对应跟踪事件或结果。

## 5. 统一事件模型

### 5.1 事件来源

| 来源 | 示例 |
|---|---|
| 市场事件 | 财报、SEC 文件、分红、拆股、价格异动、评级、宏观事件 |
| 内容事件 | 与用户关注公司相关的新洞察 |
| Agent 任务 | 周期复盘、条件监控、定时研究 |
| 用户提醒 | 用户指定时间的一次性提醒 |

### 5.2 事件状态

| 状态 | 含义 |
|---|---|
| upcoming | 已知未来将发生 |
| active | 正在发生或刚触发 |
| completed | 已结束且结果可读 |
| missed | 在有效时间内未被用户查看 |
| expired | 已过保鲜期且不再有行动价值 |
| cancelled | 来源任务取消或事件撤销 |

同一外部事件只能生成一个稳定事件记录；来源更新时更新记录并保留变更时间，不重复轰炸用户。

## 7. 排序与去重逻辑

### 今日排序

1. 用户明确创建且已到期的提醒/任务结果。
2. 持仓相关高重要性事件。
3. 自选相关高重要性事件。
4. 未来时间最近的财报/宏观事件。
5. 普通摘要和相关洞察。

同一事件如果同时命中持仓、自选和任务，只展示一条主记录，并在“为什么收到”中列出多个原因。

### 保鲜期

- 价格异动等微观状态按短保鲜期处理，过期后不再作为待办提醒。
- 财报、SEC、分红等事实事件可以长期保留。
- 保鲜期影响提醒状态，不删除需要审计的任务执行记录。

## 8. 关键流程

### 8.1 查看今日事件

```text
进入跟踪今日
→ 查看按重要性排列的事件
→ 打开详情
→ 进入公司/洞察 或 询问 Agent
→ 标记已读/完成后续动作
```

### 8.2 创建任务

```text
选择模板或从 Agent 发起
→ 设置对象、规则、时间、渠道
→ 预览下一次触发和勿扰影响
→ 用户确认
→ 任务进入运行中
→ 结果进入历史并产生提醒
```

### 8.3 勿扰合并

```text
事件在勿扰期间触发
→ 按用户规则暂存
→ 勿扰结束
→ 丢弃已过保鲜期噪音
→ 合并可读摘要
→ 进入站内记录并按可用渠道投递
```

## 9. 边界与异常

| 场景 | 处理要求 |
|---|---|
| 同一事件多来源 | 根据稳定键去重，保留来源列表和较可信字段 |
| 外部事件时间改变 | 更新日历并标明变更；已读状态保留 |
| 任务错过执行时间 | 在允许补跑窗口内补跑；超出后标记错过，不重复无限补跑 |
| 任务执行中被暂停 | 当前已开始执行按既定策略完成或取消；界面明确结果 |
| App 关闭 | P0 只保证站内持久化；无原生推送时不承诺系统通知 |
| 用户拒绝推送权限 | 保持站内通知，不反复弹系统授权 |
| 时区/夏令时变化 | 重新计算下次执行并向用户展示，不产生双次执行 |
| 无持仓和自选 | 仍展示用户任务和可选宏观事件，并引导添加关注 |
| 数据源中断 | 保留已有日历，标记更新失败；任务结果明确数据缺失 |
| 任务结果为空 | 区分“无事件”成功结果和执行失败 |
| 会员到期 | 已有任务按权益规则暂停或降级，提前说明，不静默丢失 |

## 10. 验收标准

- [ ] 今日能聚合事件、任务和提醒，并按定义排序去重。
- [ ] 月/周日历正确处理全天事件、具体时刻、时区和时间变更。
- [ ] 用户可创建、确认、暂停、编辑、删除和重试任务。
- [ ] Agent 创建的任务立即出现在跟踪，结果可返回 Agent 继续研究。
- [ ] 勿扰、摘要、事件类型和严重度偏好真实影响投递。
- [ ] App 关闭、推送拒绝、数据源失败等场景不作虚假送达承诺。
- [ ] 任务删除不误删历史结果；重复事件和重复触发被正确去重。
- [ ] 现有 cron、Web 站内推送和事件引擎数据可兼容迁移。

## 11. 研发拆分建议

1. 统一事件/任务/结果视图模型。
2. 今日聚合与排序去重。
3. 日历与时区处理。
4. 任务管理和 Agent 创建闭环。
5. 历史结果与跨模块详情。
6. 通知偏好、勿扰、原生/渠道投递增强。

## 12. Live Demo 链路确认（2026-07-12）

本轮已明确所有主要 CTA 需要有可评审的后续状态，而不是停留在按钮层：

- 时间线“详情”进入事件详情，展示为什么收到、事件状态和 Agent 准备清单。
- “查看结果”进入结构化结果，展示结论、指标、证据、待验证项、数据时间和继续追问。
- “继续准备”打开对应事件的准备状态，并能进入带上下文的 Agent。
- “管理全部任务”打开任务总览，可暂停、恢复、编辑和删除；删除后保留历史结果。
- 日历“查看当日详情”和历史“查看”复用统一事件/结果详情模型。

上述交互当前使用模拟数据，用于确认产品链路，不代表事件引擎和真实任务调度已经接入。
