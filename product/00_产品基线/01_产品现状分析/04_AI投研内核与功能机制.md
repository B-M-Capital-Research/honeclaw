# AI 投研内核与功能机制

## 核心不是单个 Prompt，而是四层协作

| 层 | 作用 |
|---|---|
| Persona / Policy | 定义投资纪律、事实边界、风险表达和隐私规则 |
| Skills | 识别用户意图并规定某类任务的工作流 |
| Tools | 获取数据、读写用户上下文、创建任务或启动研究 |
| Runner | 让不同模型/Agent 运行方式在相同权限边界中执行 |

## 投资纪律与回答边界

系统 Prompt 当前明确约束：

- 聚焦金融和投资研究领域。
- 不直接输出无条件的买卖指令；提供依据、假设、风险和验证路径。
- 对时间敏感信息说明时间与来源；无法核验时避免伪装成最新事实。
- 公司名或 ticker 有歧义时先确认。
- 创建任务、修改持仓等有副作用操作需要确认。
- 投资组合与定时任务以结构化工具数据为准，不凭聊天上下文猜测。
- 群聊里隔离个人数据，避免暴露用户组合和私有信息。

仓库的 `soul.md` 进一步定义了长期投资、反情绪化和研究纪律的产品人格。官网的品牌主张与这一执行层相互对应。

## 当前 Skill 地图

| 类型 | Skills | 作用 |
|---|---|---|
| 公司研究 | `stock_research`、`deep_stock_research` | 单股研究、估值、筛选；深度研究仅管理员 |
| 组合决策支持 | `portfolio_management`、`position_advice` | 维护持仓/关注与仓位分析 |
| 公司长期记忆 | `company_portrait` | 创建、更新、查询公司画像 |
| 市场与专题 | `market_analysis`、`gold-analysis` | 市场环境和黄金分析 |
| 主动服务 | `scheduled_task`、`missed`、`notification_preferences` | 建任务、查错过消息、调整通知偏好 |
| 内容呈现 | `chart_visualization`、`image_generation` | 图表和图片生成 |
| 内容理解 | `image_understanding`、`pdf_understanding` | 处理用户上传材料 |
| 系统管理 | `skill_manager`、`hone_admin` | Skill 和管理员运行操作 |

Skill 不是硬编码菜单：系统会在首轮提供可用技能，并根据输入搜索相关 Skill，再由 `skill_tool` 加载具体规则。Slash Skill 也有单独支持。

## Tool 能力

| Tool | 当前作用 |
|---|---|
| `data_fetch` | 从 Financial Modeling Prep 获取行情、公司资料、财务、新闻、榜单、行业表现、ETF、加密资产和财报日历 |
| `web_search` | 补充新闻、近期事件与外部证据 |
| portfolio | actor 范围内维护持仓和关注列表 |
| company profile / files | 在 actor 沙箱中搜索、读取、维护长期材料 |
| cron | 创建和管理用户定时任务 |
| notification preferences / missed | 调整主动消息偏好并查找错过的内容 |
| deep research | 启动长时深度研究任务 |
| skill discovery / execution | 查找、加载和执行 Skill |
| admin restart | 仅管理员可用的服务操作 |

FMP 数据按类型设置缓存：行情约 5 分钟、新闻 15 分钟、财报日历 1 小时、财务 6 小时、公司资料 24 小时。是否能返回真实数据依赖有效的 FMP Key；Web 搜索同样依赖外部服务配置。

## Runner 与安全隔离

当前支持 HONE Cloud、Function Calling、Codex CLI/ACP、OpenCode ACP、Gemini CLI 和 Multi-Agent 等运行形态。关键安全规则是：

- 非管理员若配置了需要主机访问的原生 CLI/ACP runner，会被强制切换到严格的 actor-bound Function Calling 执行。
- 每个 actor 使用独立沙箱、工具白名单和执行预算。
- 管理员部署可保留更强的本地 runner 权限。

这说明“支持多种 Agent 运行时”与“普通用户可任意访问宿主机”是两回事；公共用户路径在代码中有额外收敛。

## 股票研究的实际工作流

普通股票研究把三种意图区分开：

1. 单公司研究：先确认 ticker，拉取 snapshot，再按需要搜索近期事件。
2. 估值分析：拉财务与行情，结合同行和经营假设解释估值。
3. 条件筛选：形成候选集，收敛到 3–5 个标的，再逐个补快照和风险。

只有用户明确要求或信息确实需要时，才进一步生成图表。该设计强调可解释工作流，而不是每次对话都调用全部工具。
