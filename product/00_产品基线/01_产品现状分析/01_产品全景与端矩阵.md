# 产品全景与端矩阵

## 一句话结论

当前 HONE 不是单一聊天网页，而是一套以“投资纪律与长期研究记忆”为核心、由公开用户端、管理员控制台、主动事件服务、多渠道机器人和自托管运行时共同组成的 AI 投研系统。面向普通用户的体验高度集中在聊天界面，后台则显式管理用户、资产、画像、任务、研究和运行状态。

## 产品分层

| 层级 | 当前职责 | 主要载体 |
|---|---|---|
| 品牌与获客层 | 产品定位、功能解释、博客、开源入口 | 官网、路线图、Blog、GitHub |
| 用户交互层 | 登录、问答、附件、投研工具调用、历史、主动消息、投资上下文 | Web Chat、`/portfolio`、`/me` |
| 投研智能层 | 人格与纪律、技能选择、工具调用、数据检索、研究工作流 | Prompt、Skills、Tool Registry、Agent Runner |
| 长期数据层 | 会话、组合、公司画像、用户主线、任务和通知偏好 | 文件/SQLite；Cloud 模式下 PG + OSS |
| 主动服务层 | 定时任务、持仓事件监控、摘要、渠道投递 | Scheduler、Event Engine、Channel Sinks |
| 运营控制层 | 用户、会话、任务、画像、研究、日志和配置管理 | HONE Console |

## 端与运行形态

| 端/形态 | 面向对象 | 当前产品形态 | 状态与边界 |
|---|---|---|---|
| 公开官网 | 访客 | 首页、路线图、博客、条款、隐私 | 已在线验证 |
| Web 用户端 | 受邀用户 | 手机验证码登录；聊天、历史、附件、分享、主动消息、投资上下文 | 登录入口已在线验证；登录后待运行验证 |
| macOS 用户端 | 普通用户 | Tauri 远程 WebView，加载线上 `/chat` | v0.14.0 已发布 Universal DMG；不捆绑本地 AI 运行时 |
| iOS 用户端 | 开发/验证用户 | SwiftUI + WKWebView，加载线上 `/chat` | 发布物是 Simulator App 与 Xcode 工程；真机/App Store 仍需要签名分发 |
| 自托管桌面端 | 管理员/部署者 | Tauri 桌面应用，包含配置、sidecar 与本地运行时管理 | 与“远程用户端”是两种不同桌面产品形态 |
| 管理 Web 控制台 | 管理员/运营/开发 | 用户、会话、Skills、任务、研究、通知、日志、设置 | 已实现（代码），本轮未登录验证 |
| 飞书、Discord、Telegram、iMessage | 社群/个人用户 | 消息接入、群聊上下文、定时任务结果和事件推送 | 各自有独立 binary；是否可用取决于渠道凭证和部署配置 |
| CLI / Console | 开发者/高级用户 | 本地命令行对话与运行维护 | 已实现（代码） |
| MCP | Agent/工具生态 | 将 HONE 能力暴露给 MCP 客户端 | 已实现（代码） |
| OpenAI-compatible API | 外部客户端 | `/v1/chat/completions` | 已实现（代码），由 API Key 鉴权 |

## 两套前端表面

同一套 SolidJS 工程通过构建变量拆成两套产品表面：

- Public：`/`、`/roadmap`、`/blog`、`/chat`、`/me`、`/portfolio`、条款与隐私。
- Admin：Dashboard、Sessions、Users、Skills、Tasks、Notifications、Schedule、Research、LLM Audit、Logs、Task Health、Settings。

这意味着“普通用户产品”和“内部运营系统”共享前端技术栈，但导航、权限和目标完全不同。

## 品牌命名现状

- 用户界面和 v0.14.0 发布已主要使用 `HONE`。
- 仓库名仍是 `honeclaw`，官网域名仍是 `hone-claw.com`，大量 crate、binary、环境变量和历史文档仍保留 `hone-*` / `HONE_*`。
- 因此当前状态是“前台品牌已切换，工程与基础设施命名尚未完全迁移”，不是全栈完成改名。
