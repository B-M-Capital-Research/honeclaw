## Context

The isolated research-control design fixed one provider behavior but divided one reasoning process into two model roles. Business rounds could load evidence, while a separate completion saw only `continue_research` and `finish_research` and was expected to decide whether that evidence was sufficient. If that completion failed, timed out, returned content, or otherwise violated its narrow protocol, ADR 0003 allowed the runtime to interpret the failure as an implicit/degraded finish and enter terminal synthesis.

The first production canary after the MiniMax clean-EOF transport repair proved that boundary unsafe. The exact query `crwv和英伟达有什么关系` completed as one successful stream with no fixed refusal, reset, second answer, or missing-DONE error, so the provider adapter and public stream lifecycle were working. However, the Agent made only `data_fetch(search)` calls for CRWV and NVIDIA. The isolated control completion then ended through an error-labelled implicit finish, and terminal synthesis asserted company-relationship facts that were not present in the current-turn tool evidence. The canary therefore failed answer quality even though transport and DataFetch identity lookup succeeded.

The first production canary after deploying the initial ADR 0004 implementation (`0cd03abb`) proved that one-loop ownership fixed transport and publication but left a last-mile asymmetry. It produced one answer and one successful terminal in about 42.4 seconds with no reset, error flash, refusal, second generation, or visible/persisted mismatch. Its five Agent calls successfully searched CRWV/NVIDIA, ran one relationship Web search, and fetched batched CRWV/NVDA quotes. FMP and Tavily were healthy. Yet DirectFinal used the quote provider time (`2026-07-18 04:00`) as the answer data time instead of the current Session time and expanded secondary snippets into unsupported customer rank and supply-priority claims. Explicit terminal synthesis had a strong final user contract, while DirectFinal had only an earlier system reminder. A first follow-up attempted to make quote/profile coverage a runtime publication prerequisite; independent review showed that this would recreate the exact generic-error/refusal class the design is meant to remove, while also deadlocking crypto and malformed/budget-error paths. The accepted repair therefore keeps entity/quote/asset-route sequencing in the Agent prompt, retains the ledger for structural guidance and telemetry, gives both completion paths one exact-prefix/source-bounded contract, and never uses that ledger to veto a complete natural answer.

The next deployed canary on `c7c5741f` removed the legacy active-phase timeout and completed `crwv和英伟达有什么关系` as exactly one clean answer in about 48.54 seconds, but it exposed a narrower pre-finish routing defect. The Agent searched both CRWV and NVIDIA, then loaded quote/profile only for CRWV; the ledger compared only symbols that had already been quoted and therefore exposed `finish_research` without proving that the successful NVIDIA candidate route had any same-symbol quote/profile attempt. Terminal prose then invented “largest/one of the largest”, “core customer”, “no equity relationship”, preferred-partner, chip-model, contract-size, and role details beyond the current source snippets. This was a content canary failure, not an FMP/Tavily outage, a second frontend run, or evidence that Interactive publication needs a semantic gate.

The first canary after the per-route implementation was deployed as exact build `ac2468f9`. Its transport and publication behavior passed: one start, one assistant delta, one successful finish, no reset/error/refusal, byte-identical persistence, zero remaining active chats, and 46.300993 seconds to the only visible answer. Provider diagnosis again showed healthy FMP and Tavily. Content still failed because the Agent searched both entities but stopped after CRWV quote and one relationship Web query, without CRWV profile or NVDA quote/profile. The active prompt simultaneously asked for more evidence and told the provider that a natural body would be accepted, so it produced an early DirectFinal and expanded snippets into unsupported rank, role, MFN-priority, chip-model, and Forward-PE claims. `D-2026-07-19-04` resolves the contradiction inside the Agent loop: tool-enabled rounds are evidence-only, route gaps are rendered concretely, and Web snippets carry trusted provenance metadata. The service still accepts a provider DirectFinal unchanged as a no-veto compatibility fallback and never adds a post-generation semantic interceptor.

The next exact production build, `3d2474b5`, proved that evidence-only rounds and transport uniqueness were still insufficient when `finish_research` carried only an empty object. A fresh `crwv和英伟达有什么关系` run had one start, one visible answer, one successful terminal, no reset/error/second answer, byte-identical history, and zero remaining active chats. FMP and Tavily were healthy, and the terminal received the complete tool transcript. The Agent nevertheless stopped after one broad relationship search and expanded three basic snippets into unsupported customer rank, GPU-model, bargaining-power, and “no equity relationship” assertions. This was neither stale deployment nor a post-run interceptor: the empty finish signal transferred control without an explicit, mechanically resolvable claim-to-source handoff. The same incident review found an independent browser defect: the shared fetch helper retried non-idempotent chat POSTs, and a stream exception could briefly render an error before authoritative run recovery.

Exact build `ae4d9a8f` then deployed the structured handoff, terminal isolation, and browser one-shot/recovery repair with healthy cloud mode, Postgres, object storage, ports, auth boundaries, and enabled channel processes. A correct fresh direct canary omitted `channel_scope` and completed `crwv和英伟达有什么关系` as one successful, byte-identical stream in about 101.748 seconds with FMP and Tavily healthy. Content nevertheless failed. The Agent submitted `web_search`, `quote`, and `profile` as evidence IDs rather than copying the opaque current-turn call IDs. The ae4 runtime did not recover the generic Web ID, the DataFetch references were malformed as well, and mechanical validation dropped every reference. The unrestricted terminal therefore received an empty fact set, reversed the source's capacity-purchase direction, and invented additional relationship details. The transport and deployment were current. The remaining defect was that the handoff schema described an ID shape but did not bind the Agent to the actual citable IDs already present in its own current turn, while the runtime still allowed unresolved-all evidence to reach terminal synthesis.

`data_fetch(search)` establishes candidate identity; it does not establish a commercial relationship, ownership, customer concentration, contract, GPU deployment, valuation input, or the absence of any such relationship. A service-side semantic checker cannot safely repair this after generation: it would recreate the brittle refusal/rewrite path that previously replaced valid Agent answers with “cannot safely publish” copy. The research Agent that reads the user query and tool results must retain both evidence-selection and finish authority inside one continuous loop.

## Decision

- Keep one `Agent::run`, one `AgentContext`, and one model role for active Interactive research. After finance activation, each active completion sees the actor-bound business tools. There is no separate `continue_research` completion and no service-owned implicit finish decision.
- Carry `AgentRunnerRequest.timeout` and the configured step timeout into the strict FunctionCalling runner. Convert the request/fallback overall value once at `Agent::run` entry into a single absolute deadline shared by initial model discovery, active model rounds, tool execution, tool observers, terminal synthesis, and committed-prefix terminal recovery; no phase may reset or extend it. Apply a fresh configured step deadline to every individual await. Before returning from a tool failure or deadline, retain a failed `ToolCallMade` with `status=failed`, `isError=true`, and an explicit timeout flag. Use the canonical `hone-core` tool-effect classifier in both the Agent and Session so an uncertain persistent failure stops inside the current loop and cannot be replayed before the outer boundary receives its trace.
- Track a request-local structural evidence ledger inside the Agent loop for guidance and telemetry, not publication authority:
  - `data_fetch(search)` is `identity_only`; it may identify or refine a security but cannot by itself make terminal synthesis eligible. The Agent assigns every named security a stable, distinct, case-sensitive `entity_route`, explicitly declares each search's call-scoped `identity_match=exact_symbol|name_or_alias` from the complete query, and reuses that route verbatim on refinement, quote, profile/snapshot, and later DataFetch calls. A later search that omits, empties, mistypes, or corrupts its match/link metadata cannot migrate or replace an earlier route merely because that route once had a valid declaration. Any parseable DataFetch call that mentions an explicit route immediately registers it as pending; that route accepts no route-bound non-search evidence until its own valid search declaration and result, including within a correctly ordered parallel batch. The ledger retains one independent candidate set per declared route, even when provider result sets overlap. Before the explicit completion signal becomes available, each route needs one candidate symbol present in both its route-bound quote attempts and profile/asset-route attempts. An unrelated extra quote, pre-search quote/profile, a profile for a different candidate, or coverage of only one named entity cannot satisfy another route;
  - the service does not infer entity boundaries or match mode from user-language separators, ticker shape/case/length, or positional result differences. It never splits commas, `and`, `和`, `&`, or `/`, so legal inputs such as `AT&T`, `S&P Global`, `M&T Bank`, `H&R Block`, `BRK/B`, and `Berkshire Hathaway, Class B` remain one Agent-understood entity unless the Agent declares otherwise. Each named entity still receives its own search call, and calls may run in parallel. The first Agent-declared exact-symbol search fixes a persistent route constraint; later name refinement cannot widen it to a product such as CWY whose name merely embeds CRWV. Exact equivalence uses the shared audited `hone-core::provider_symbol` dialects rather than local punctuation replacement, covering only bounded share-class, exchange-suffix, Hong Kong padding, index, crypto-pair, and cashtag variants. For `name_or_alias`, word-boundary grounding preserves short company names such as Ford, Apple, and Tesla without falling through to FORD, Appleseed, or derivative products. A later successful refinement under the same `entity_route` replaces that route's prior empty/noisy candidates and resets the current empty-attempt streak. Legacy untagged searches remain provisional routes until an explicit search names one exact, case-sensitive old query through the actual `query`, `refines_query`, or `supersedes_query`. `refines_query` and `supersedes_query` are strictly mutually exclusive; if both appear, no provisional route migrates and the explicit route remains pending. A valid single-source migration carries only search/empty attempt history, verbatim query aliases, and an already Agent-declared exact-symbol constraint; it never imports candidates, quote symbols, profile/asset-route symbols, or post-identity attempts. If a route has no candidate after two current consecutive unsuccessful search attempts, explicit finish still requires a real route-correct non-search attempt after the latest empty result; an unscoped no-symbol follow-up is attributable only when exactly one active route exists. The Agent may then disclose the gap rather than refuse the whole question;
  - include crypto-specific `search → crypto_quote` and separate broad-market routes. Unsearched/pre-search symbol calls do not satisfy the structural floor;
  - this structural state controls only whether the internal `finish_research` schema is exposed and whether the active request transitions from business-tool `Required` guidance to business-tools-plus-finish `Auto`. It does not participate in DirectFinal acceptance, Interactive publication, post-run response review, or answer persistence. The Agent prompt—not a server-side answer gate—still owns semantic evidence selection. Genuine no-coverage, malformed calls, budget exhaustion, crypto routing, and unusual wording remain answerable with bounded disclosure instead of a generic runtime failure.
- The initial Agent round rereads the complete query and, if companies or securities are named, returns only one separate DataFetch search call per named entity with its own stable route and call-scoped match mode. After finance activation, active rounds are evidence-only. Before the structural floor they use business-tool `Required` and return only real tool calls; after the floor, `finish_research` appears beside business tools under `Auto`, but the prompt permits only more business calls or one logical finish-only structured handoff containing `answer_scope`, `facts`, `inferences`, and `gaps`. Before that schema is sent, the runtime builds a bounded source catalog from matching current-turn invocation/result pairs: successful non-search DataFetch results with usable scalar data and successful Web results with an original URL plus excerpt-capable text. Search, failures, missing invocations, unsupported tools, empty data, and non-citable Web rows are excluded. When the catalog is nonempty, the flat schema enumerates its exact opaque IDs in `tool_call_id`; an empty catalog emits no empty enum. The accompanying prompt lists only each ID and structural source description, not excerpts, URLs, errors, or a raw evidence transcript. Facts contain no free-text claim: each has only a stable id plus current-turn references. Web references identify one result and exact excerpt; DataFetch references identify an exact call ID and scalar RFC 6901 JSON Pointer. Inferences point to submitted fact IDs, while unresolved attempted dimensions remain gaps and never become negative facts.
- Treat the handoff as a tolerant pre-terminal protocol transition, never an answer gate. Resolve each reference independently and mechanically against current-turn tool results. Exact IDs are authoritative. A generic Web tool name may be mapped to an actual ID only when its submitted result number and verbatim excerpt match exactly one current-turn Web invocation; ambiguity or no match authorizes nothing, and generic DataFetch names are never guessed. Preserve resolved excerpts/fields and valid sibling facts. If the current turn has citable sources but the handoff contains an unresolved locator or contains neither a mechanically resolved related fact nor scoped fallback, the same Agent receives at most one hidden locator-only correction containing the exact catalog before terminal synthesis. This bounded correction does not replay business tools, introduce a second research role, review answer content, or modify already visible prose. After that correction, remaining invalid siblings may be omitted only when at least one valid related fact or scoped fallback survives; otherwise fail before terminal. A direct gaps-only handoff is legal only when the citable catalog is empty because the attempted real sources produced no citable result, and it must disclose the concrete missing coverage. When the catalog is nonempty, gaps alone never authorize terminal. Syntax-broken/truncated controls have no whole-turn authority; with citable sources they receive the same one correction, otherwise they fail. Fallback remains strictly Agent-selected: one Web reference can recover only that result number, and one DataFetch scalar/missing-leaf reference can recover only its exact direct object. DataFetch `search`, a result without its matching invocation, unselected results, neighboring batched entities or financial periods, root/array/null targets, and `error/errors/isError/*_error` paths contribute nothing. Duplicate invalid handoffs never union scopes. Mixed finish plus business calls executes only business calls and ignores every accompanying finish control; finish controls never enter registry, observer, progress, persistence, or budget.
- Keep complete `Stop + Done` DirectFinal acceptance as a provider compatibility/no-veto fallback, even before the advisory floor. If a provider ignores the evidence-only prompt and returns a nonempty natural body, publish it unchanged through the deferred Interactive boundary without terminal synthesis, retry, rejection, replacement, or a second model generation. Do not advertise or invite this path in active prompts. Empty output, provider error, timeout, protocol failure, or iteration exhaustion remain genuine failures and never synthesize terminal authority.
- Keep tool-capable rounds speculative and hidden from the public answer. The terminal completion remains tool-free and is built from the current system contract, the current user request, and the mechanically resolved handoff evidence only. It receives no raw tool transcript, prior assistant/tool facts, tool-round prose, provider reasoning, or earlier conversational evidence. Provider reasoning signatures needed for MiniMax/Mimo tool-result replay remain only on active tool follow-ups and are non-evidence/non-persistent. The canonical first-line commit, exact suffix publication, one buffered terminal-only committed-prefix recovery, honest partial failure, and provider finish/DONE rules from ADR 0003 remain unchanged.
- The normal explicit-finish terminal owns the final ephemeral user message, exact first-line prefix, and evidence-limited answer contract. Extract and repeat the prefix already generated for this runtime turn rather than asking the model to choose a timestamp. The Session query time comes from one structured clock value shared with `Session 上下文`, the answer suffix, and context-overflow recovery; quote provider time belongs only in `行情口径` and must itself come from resolved quote fields. Only resolved excerpts/fields, labeled inferences/gaps, and bounded fallback evidence cross the terminal boundary. Global/runtime prompt layers still guide a compatibility DirectFinal, but it must not be documented as receiving the terminal-only resolved handoff.
- Keep evidence selection open-ended. The complete user query—not a phrase grammar, ticker exception table, or server-side intent classifier—determines which business tools the Agent calls. Relationship questions require exact quote/profile evidence for the named entities plus current Web/news/filing evidence for the claimed relationship; valuation questions require the inputs needed for the stated valuation frame. The structural ledger does not claim that any one non-search call is semantically sufficient.
- Require evidence-limited language in the Agent prompt. “This run did not find evidence” is not proof that a relationship, holding, customer, contract, or event does not exist. Negative factual assertions such as “no equity relationship”, “not a customer”, or “no contract exists” require explicit current evidence; otherwise the Agent may only describe what the current search did or did not establish.
- Describe and annotate actual Web-search capability: it is low-bandwidth basic search that returns at most three titles, original URLs, and snippets, not page bodies. Cap successful responses locally as well as at request time. A trusted root contract states snippet scope and that result order/score is not real-world rank; locally overwritten per-result metadata marks citation scope and missing-URL results as non-citable. Provider content cannot spoof those fields. This envelope is evidence context for the Agent, never a service-side output judge.
- Give the tool-free terminal the resolved structured handoff as its external-evidence boundary. Each relationship number, direction, rank, role, right/obligation, product model, and valuation label must be supported by one resolved excerpt/field; Web facts cite the runtime-injected title and original URL inline. A URL only locates the source. The terminal Agent derives factual wording directly from resolved evidence because no free-text fact claim crosses the boundary. A judgment beyond literal evidence must come from a submitted inference and appear as a separate sentence beginning `推断：`; otherwise the Agent omits it. None of these instructions authorize a service-side final-answer filter, semantic validator, rewrite, retry, or refusal.
- Keep Interactive post-run reconstruction observational. The service may record entity/evidence coverage and protocol transitions, but it may not prepend, normalize, semantically approve, retry, repair, reject, replace, or append a fixed completeness refusal to the successful Agent answer. Completed Interactive bodies use a dedicated security-only finalizer: it may remove system/internal protocol material, redact unsafe local paths, and stabilize local media references, but it may not rewrite market copy, veto planning-like prose, reconstruct an answer from tool outcomes, or run the scheduled/heartbeat completeness guard. The exact finalized body emitted to the user is also the body persisted in session history. Caller-supplied typed scheduled/heartbeat contracts retain their separate deterministic validation and fail-closed behavior.

## Consequences

- The model that chooses the evidence also decides when it is ready to synthesize, with all business-tool results still in the same context. There is no second control role that can lose the user question, mistake identity lookup for research completion, or add an extra latency-only completion.
- The prompt and structural ledger make missing search/quote/profile/crypto routing concrete on every active round and delay only the explicit finish schema. Their state is explicitly described as structural attempts rather than successful evidence or complete entity discovery. DirectFinal and Interactive publication remain independent, so an imperfect tool trace cannot become a whole-answer ban.
- The normal path now pays one tool-free terminal round in exchange for a single unambiguous final stage with exact time and mechanically resolved evidence. A provider that ignores the tool-only protocol and returns a complete natural answer still cannot be replaced by service-authored failure copy; that compatibility fallback remains unchanged even though it does not receive the terminal-only handoff.
- A provider that copies the dynamically enumerated IDs pays no additional completion. If it still emits an unresolved locator, the bounded same-Agent correction may add one hidden model call before the terminal. That cost is preferable to granting an unrestricted terminal empty evidence, but it makes production TTFT measurement mandatory and does not by itself solve latency.
- Genuine provider gaps still permit a useful answer after a real evidence attempt. The Agent narrows claims and labels the gap; the runtime does not turn missing fields into a whole-answer ban.
- The service remains responsible for structural safety—tool registration, current-run evidence isolation, stream lifecycle, terminal uniqueness, security-only Interactive cleanup, exact visible/persisted bytes, and typed noninteractive contracts—but not for open-ended semantic publication review.
- Removing the isolated control request also removes one model round from the critical path. Actual first-visible latency still depends on business-tool and terminal generation time and must be measured in production rather than inferred from the architecture.
- One absolute request deadline bounds total FunctionCalling latency instead of multiplying an overall timeout by phases, and the configured step deadline bounds any single stalled model/tool/observer await. A timed-out or failed persistent tool leaves an auditable uncertain trace and fails immediately inside the same loop; failed observer progress is emitted as failure rather than completion.
- No durable-data migration is required. The evidence ledger, completion-signal availability, stable turn clock, and terminal transition are request-local.

## Risks

- Entity-route quote/profile coverage intentionally controls only explicit-finish availability and `Required → Auto` progression. Route identity and per-search `identity_match` are declared by the same Agent that reads the full query; malformed declarations are ignored rather than guessed. If that Agent omits an entity or reuses a route, the service will not second-guess natural language or turn the omission into a publication ban. The initial tool-only instruction and concrete route guidance reduce that risk, but a provider can still ignore the prompt and return DirectFinal; behavioral canaries must verify question-specific evidence without expanding the runtime into a semantic veto.
- A provider may reject or ignore `Required` or fail to copy a schema-enumerated call ID. Existing narrow provider-capability `Required → Auto` compatibility telemetry remains necessary. Before the entity-route floor, the reminder asks only for more business calls; after the floor, it asks only for more calls or a logical finish-only handoff. Provider duplication never adds authority. Malformed or unresolved locator framing may add exactly one hidden same-Agent locator-only correction before terminal, but never an unbounded loop, business-tool replay, or answer rewrite. Any complete DirectFinal body is preserved; the structural ledger never turns it into a publication veto.
- Tool categorization must stay structural and generic. Adding company-specific symbols, question phrases, or hard-coded relationship routes would recreate the architecture this decision removes.
- Iteration limits remain necessary. Reaching one is a normal failed run, not permission to fabricate an answer, publish a fixed “research incomplete” response, or enter terminal synthesis without explicit finish.
- Tolerant handoff resolution may omit an invalid reference after one bounded locator correction. That unresolved state must remain visible in internal telemetry. With a nonempty citable catalog, at least one mechanically resolved related fact or scoped fallback must survive; gaps alone and an all-invalid handoff fail before terminal. A direct gaps-only handoff remains usable only when the catalog is empty because real attempted sources were unavailable. Compact fallback evidence is deliberately mechanical and bounded; the terminal Agent still owns relevance, wording, and disclosure, and no post-answer semantic gate is introduced.
