# Bug: Web 直聊把投研过程句当成最终回复，用户需要二次追问才拿到正式答案

- **发现时间**: 2026-04-22 22:02 CST
- **Bug Type**: Business Error
- **严重等级**: P3
- **状态**: Fixed
- **证据来源**:
  - `data/sessions.sqlite3` -> `session_messages`
  - `session_id=Actor_web__direct__web-user-e05f5e5f74a3`
  - `2026-04-22T21:18:41.820223+08:00` 用户提问：`最近BABA值不值得加一些`
  - `2026-04-22T21:19:26.154724+08:00` assistant final 只返回：`已确认你当前组合里没有记录的持仓，且本地没有 BABA 公司画像；这次我会按“是否新加一笔观察/底仓”的框架来判断，而不是按已有仓位加仓。下一步补财务和估值质量，避免只看跌了多少。`
  - 这条回复没有回答“值不值得加一些”，也没有给出 BABA 的价格、估值、财务质量、风险或动作建议；用户在 `2026-04-22T21:20:35.993210+08:00` 只能追问 `不需要，直接告诉我吧`。
  - `2026-04-22T21:21:03.944666+08:00` assistant 才给出正式判断：`BABA 现在可以考虑“小幅分批观察”，但不适合一次性大幅加仓...`
  - `data/runtime/logs/acp-events.log` 对应 Web 会话后续只看到正式回答流和 `stopReason=end_turn`，没有证据显示第一条过程句之后系统自动补发正式答案；正式答案依赖用户二次追问触发。

## 端到端链路

1. Web 直聊用户提出一个明确的投资判断问题：`最近BABA值不值得加一些`。
2. 系统返回一条“已确认无持仓/无画像，下一步补财务和估值质量”的过程性说明，并将其作为 assistant final 落库。
3. 该 final 没有回答用户问题，也没有标记为仍在处理中。
4. 用户追加 `不需要，直接告诉我吧` 后，系统才再次运行并输出正式投研判断。

## 期望效果

- 用户明确问“值不值得加一些”时，Web 直聊应直接给出结论、关键事实、风险与动作边界。
- 如果系统确实需要更多工具或数据，应继续完成后再发送最终回复，或明确以可理解的等待/失败状态收口。
- 过程性文本不应被当成本轮最终答案。

## 当前实现效果

- 第一条 assistant final 只说明“下一步补财务和估值质量”，没有回答买入/加仓判断。
- 用户需要二次追问才能拿到正式答案，说明第一轮用户目标未完成。
- 该样本发生在 Web direct 链路，不是 Feishu 渠道专属问题；但当前证据只有单次 Web 会话样本，暂不升级为跨渠道 P2。

## 用户影响

- 用户收到的是内部研究计划句，而不是投研结论，增加了一次不必要的交互。
- 不影响消息投递、会话落库、工具可用性或定时任务配置，也未造成错误投递或数据损坏。
- 因此本缺陷主要属于 AI 返回质量/结构问题：功能链路仍可通过用户二次追问完成，所以定级为 P3。

## 根因判断

- Web direct 的 final 判定可能没有过滤“下一步补...”这类过渡句，导致研究计划被当成可见最终答复。
- 该形态与 Feishu 直聊已有 `feishu_direct_partial_reply_before_tool_completion.md` 同类，但影响渠道、证据会话和用户入口不同；当前先按 Web 独立受影响链路建档，后续若确认共享收口逻辑，可合并根因修复。

## 修复情况（2026-04-27）

- 已确认根因位于共享 ACP Answer 收口，而不是 Web 渠道专有出站逻辑：`stopReason=end_turn` 过去不会检查是否仍有未完成工具，导致过程句在 Web 直聊里也可能被记成 `success=true` final。
- 本轮已在 `crates/hone-channels/src/runners/acp_common/ingest.rs` 增加完整性门槛，并同步落到 `codex_acp`、`opencode_acp`、`gemini_acp` 三个 runner；若 tool 尚未完成，整轮改判失败，不再把过渡句持久化成成功答复。
- 由于 Web 出站本来就不会保留失败时的 partial stream，这个共享修复会直接把同类坏态从“半成品成功”收敛为“明确失败提示”，避免用户必须靠二次追问才能拿到真正的下一轮执行。

## 下一步建议

- 将这类样本纳入质量巡检：用户问题包含 `值不值得`、`能不能买`、`加一些`，但 final 没有明确判断或动作边界时标记异常。
- 继续观察是否还有“tool 已完成但模型只产出过渡句”的新样本；若仍存在，那将是不同于本轮 `unfinished tool` 的剩余根因。
