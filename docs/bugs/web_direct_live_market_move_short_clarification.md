## 端到端链路

1. Web direct 用户询问某公司“今天为什么涨涨跌跌”。
2. 当前轮应被识别为强时效日内涨跌归因问题。
3. runner / model 没有先调用可用的实时网页、行情或新闻核验工具。
4. assistant 直接收口为短答澄清，要求用户确认上市地和成本价。
5. 用户的核心问题没有被回答。

## 期望效果

- 当公司实体可以高置信识别时，应默认使用主上市地或明确声明分析口径，先研究并解释当日走势。
- 最终答案至少应覆盖一句话结论、日期 / 市场 / 时段、价格路径或数据限制、按重要性排序的催化、上涨段与回落段原因、事实与推断边界、板块 / 同业背景，以及下一步可观察催化。
- 如果确实需要澄清上市地，应在给出默认口径下的初步解释后再说明 ADR / 韩国主板差异，而不是完全阻塞回答。

## 当前实现效果

- 真实用户同题只收到“需要确认哪个交易标的”和“你买在什么价格”的短答。
- 回答没有使用本轮可审计工具证据，也没有解释当日涨跌原因。
- 后续探针显示系统可以产出更完整答案，但该能力没有稳定覆盖普通 Web direct 真实会话。

## 用户影响

- 用户明确要求“为什么涨涨跌跌”，收到的是澄清和成本价追问，任务完成度低。
- 不影响消息投递、会话收口、权限边界、数据写入或定时任务执行；没有错投、敏感信息泄露、系统级未回复或批量失败证据。
- 因此这是质量性缺陷，且不影响功能链路，定级为 `P3`。

## 根因判断

- 初步判断是 Web direct 的强时效日内涨跌归因意图没有稳定触发“先核验、再解释”的质量门槛，模型可以在无工具证据时把问题改写成上市地 / 持仓成本澄清。
- 该问题与“强时效金融来源不可审计”相邻，但根因更偏向回答契约缺失：即使上市地有歧义，也应先默认主上市地并说明范围，而不是停止在澄清问题。

## 下一步建议

- 为 live-market move 类问题增加强制 evidence / answer contract：高置信实体先默认主上市地，至少一次当前来源核验，短答不得直接收口。
- 增加回归样本：`今天Hynix为什么涨涨跌跌的` 应产出有时点、价格路径、催化和事实 / 推断边界的答案。
- 确认该 contract 不影响简单问候、非金融常识解释或真正高歧义 ticker 的澄清流程。
