## 端到端链路

1. 用户在 Feishu 直聊里询问 Cerebras 的业务本质。
2. 系统正常生成了一条“公司业务/护城河/财务拆解/Bull vs Bear/动作建议”的长结构化分析。
3. 这条分析没有把“业务介绍”和“投资风险”有效去重，同一组风险点在多个板块被重复改写后再次输出。
4. 用户在后续真实会话里直接指出“很多信息数据都是重复的”。
5. assistant 随后承认冗余集中在 `G42 客户集中度` 与 `ORCL 关联敞口` 两个点上。

## 期望效果

- 当用户问“公司做什么业务”时，系统应优先给出清晰的业务介绍，再在必要处补一轮风险提示，而不是把同一风险点在多个段落反复改写。
- 结构化长答中的每个板块应提供增量信息，避免 `财务拆解 / 多空逻辑 / 动作建议` 只是重复同一组结论。
- 若系统要保留投资建议模板，也应在生成前做一次跨段去重，减少冗余句子占据正文。

## 当前实现效果

- 当前链路能够正常接收、生成并送达回答，没有出现错投、无回复、工具失败或持久化异常。
- 但在真实会话中，Cerebras 这轮答案把同一风险锚点在多个结构化板块重复展开，导致用户必须额外指出“信息数据重复”。
- 到 `2026-04-19 15:31 CST`，同一用户在 ASTS 主题的后续对话里再次直接指出“重复性内容信息还是太多”，说明这不是单次个股样本偶发波动，而是长答组织问题仍在当前真实链路持续影响阅读体验。
- assistant 后续的自我承认说明，这不是单纯的用户主观偏好，而是本轮答案内部已出现明显的重复组织问题。
- 这是质量类缺陷。之所以定级为 `P3`，是因为它没有破坏主功能链路，用户仍收到了可读、基本正确的分析；当前伤害主要是信息密度下降与阅读体验退化，而不是系统不可用。

## 用户影响

- 用户难以快速抓住真正的新增信息，因为正文被同一风险点的重复改写稀释。
- 在投研分析场景里，这会降低长答可信度，让用户怀疑系统是在“堆字数”而不是提供增量判断。
- 当这种冗余出现在多轮个股分析中，用户会更频繁打断并要求压缩，影响会话效率。

## 根因判断

- 当前 answer 生成更像是套用固定的投资分析模板，但缺少“跨章节去重”约束，导致同一个风险点在不同章节被重复消费。
- 这轮日志显示 search/answer 都没有工具调用，说明问题主要发生在最终文本组织阶段，而不是工具结果冲突或外部检索噪声。
- 会话里已经存在 `ORCL` 持仓背景与用户给出的 `G42` 信息，模型倾向把这些高权重锚点在每个子章节都重复展开，缺少“同一逻辑只说一次”的收口机制。
- 最近一小时的 ASTS 样本还说明，问题不只局限于“风险点重复”；当回答模板切到“流程拆解 + 动作建议 + 催化条件”时，也会继续输出过多铺垫与框架性套话，最终被用户视为重复冗余。

## 下一步建议

- 为 Feishu 直聊的个股/IPO 分析补一条质量回归样本，检查同一风险点是否跨章节重复出现超过一次。
- 在长结构化答案出站前增加轻量去重审查，优先压缩“同一实体 + 同一风险判断”在多个章节的重复表述。
- 对“业务介绍类”问题强化回答模板，先回答业务本身，再限制风险提示段落长度，避免模板性投资论证压过用户原问题。
