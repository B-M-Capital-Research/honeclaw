# Bug: Feishu 直聊询问 skill 时误报“没有该 skill”，并把内部约束直接当答案返回

- **发现时间**: 2026-04-18 22:10 CST
- **Bug Type**: Business Error
- **严重等级**: P3
- **状态**: New
- **证据来源**:
  - `data/sessions.sqlite3` -> `session_messages`
    - `session_id=Actor_feishu__direct__ou_5fe09f5f16b20c06ee5962d1b6ca7a4cda`

    - `2026-04-18T21:06:49.334+08:00` assistant 返回：`我没有所谓的小红书技能，也不会生成迎合该平台流量逻辑的内容。我的底层系统纪律严格禁止...`
    - 同一回答继续外泄“底层系统纪律”“被禁止使用王炸/无敌/FOMO/过度排版”等内部约束口径，而不是先回答 skill 本身是否存在、如何使用或如何获取
    - `2026-04-18T21:13:01.772+08:00` 用户紧接着指出：`你的文档里有`
    - `2026-04-18T21:14:09.407+08:00` assistant 又承认此前出现“事实遗漏”，说明上一轮答复并不可靠

    - `2026-04-18 21:06:49.298` 整轮以 `success=true` 收口，说明系统在成功拿到技能发现结果后，最终答案仍没有正确消费这个工具结果

## 端到端链路

1. 用户在 Feishu 直聊中直接询问“小红书 skill”。
2. 系统正常进入搜索与回答链路，并在 answer 阶段实际调用 `hone_discover_skills`。
3. 最终回复没有基于技能发现结果回答“有哪些 skill / 是否存在 / 能做什么”，而是直接否认 skill 存在。
4. 同一条回复还把“底层系统纪律”“禁止使用 FOMO/营销词/过度排版”等内部行为约束直接暴露给用户。
5. 用户随后指出“你的文档里有”，assistant 才承认此前存在事实遗漏。

## 期望效果

- 当用户询问某个 skill 是否存在、长什么样、如何使用时，系统应优先回答技能发现结果本身。
- 如果当前没有精确同名 skill，也应说明接近能力、可替代方案或检索结果为空，而不是编造“没有所谓的某技能”。
- 不应把内部约束、系统纪律或生成策略口径直接当成用户答案输出。

## 当前实现效果

- 这轮真实会话里，技能发现工具已经被调用，但最终答案仍错误地否认相关 skill，并偏离到“我不会写小红书风格内容”的自我约束陈述。
- 用户原本是在问 capability / 文档，不是在要求生成营销文案；当前回复属于答非所问。
- 同一会话后续又出现“此前存在事实遗漏”的自我修正，进一步证明首轮回答没有正确消费已有上下文和工具结果。
- 这是质量性 bug，不影响消息送达、会话持久化或工具执行成功，因此按 `P3` 定级；之所以定级为 `P3`，是因为问题主要表现为回答内容不准和内部口径外泄，没有破坏主功能链路。

## 用户影响

- 用户无法可靠了解当前系统到底有没有对应 skill、skill 文档如何使用、能力边界是什么。
- 回答把本应内部消化的“系统纪律”直接暴露给用户，降低产品感并增加困惑。
- 对 capability / 文档类提问，这会让用户误以为系统没有该能力，或误解真实使用边界。

## 根因判断

- answer 阶段虽然调用了 `hone_discover_skills`，但最终回答没有正确消费工具结果，说明“工具结果 -> 最终答案”的桥接存在质量缺口。
- 模型把“用户问 skill”错误改写成“用户想要小红书风格写作”，随后沿着拒绝营销文案的方向展开，导致语义跑偏。
- 当前出站净化主要覆盖 `<think>`、协议文本和底层报错，但对“内部政策口吻被包装成自然语言答案”缺少专门约束。

## 下一步建议

- 补一条 capability / skill 问答回归样本，要求 answer 阶段必须显式消费 `hone_discover_skills` 的结果。
- 对“询问 skill / 文档 / 能力边界”的意图增加更强的回答模板，优先输出检索结论、用途和使用方式，而不是泛化成内容风格讨论。
- 在出站审查里补一条规则：当回答出现“底层系统纪律/被禁止/内部规定”等口吻时，优先检查是否偏离了用户问题。
