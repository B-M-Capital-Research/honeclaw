## 端到端链路

1. Web scheduler 到点触发用户配置的英文 X 帖草稿任务。
2. prompt 中带入权威触发配置和用户的输出格式约束。
3. agent 完成网页搜索 / 行情核验并生成 final。
4. final 成功写入 Web 会话，但没有遵守短 X 帖格式，而是生成接近市场长评的多段正文。

## 期望效果

- 对短帖、卡片、表格、JSON、邮件草稿等明确格式约束，scheduler final 应优先满足用户给出的结构和长度要求。
- 本任务应输出 5-8 条英文短帖草稿，每条接近 140 英文字符，保留明确观点，避免长解释和长来源尾注。
- 如证据不足，应该压缩为带 uncertainty 的短帖，而不是把核验过程扩写成完整长文。

## 当前实现效果

- final 虽然使用英文并覆盖了 AI、半导体、云计算、能源、地缘等主题，但每条都是长段落。
- 输出包含标题、分隔线、解释段和尾部 sources，不适合直接发 X，也没有满足“每条约 140 个英文字符以内”。
- 任务已成功收口和落库，没有 send_failed、空回复、原始工具报错、`<think>`、本机路径或 JSON/Markdown 结构损坏证据。

## 用户影响

- 用户需要手工大幅改写才能发布，定时任务没有交付可直接使用的 X 帖草稿。
- 这会降低内容生产类 scheduler 的可用性，尤其是用户依赖固定格式自动生成的场景。
- 该问题不影响 Web scheduler 的触发、生成、落库或投递主链路，也没有造成错投、数据安全或系统失败；主要是用户显式格式约束未被遵守，因此按质量性问题定级为 `P3`。

## 根因判断

- 初步判断是 scheduler / agent final 阶段缺少对显式格式约束的强校验或压缩改写。
- 模型在完成“核验当天最新新闻”的子目标后，把内容扩展为完整市场评论，没有在最终阶段回到用户要求的 X 帖短格式。
- 该问题不同于 heartbeat 结构化 JSON 退化，也不同于普通金融复盘日期口径错误；本轮证据显示核心问题是输出形态与用户任务契约不一致。

## 下一步建议

- 为 scheduler final 增加轻量格式契约检查：当用户明确要求每条字符数、条数、语言和“不写长解释”时，最终回复应按该结构自检并压缩。
- 对短帖类任务增加回归样本：输入要求 5-8 条、每条约 140 英文字符，输出不应生成长分析正文或长来源尾注。
- 修复时避免简单截断；应保留每条观点和必要 uncertainty，同时把来源信息压缩为内部证据或极短口径。
