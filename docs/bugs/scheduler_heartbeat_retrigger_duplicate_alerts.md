## 端到端链路

1. Heartbeat 任务在 `08:30` 首次识别到 `BlueBird 7` 发射计划，正常命中 `triggered` 分支并向用户发送提醒。
2. 到 `09:00`、`09:30`、`10:00` 的后续轮询窗口，任务再次读取到同一条“今日发射”的旧事实。
3. 当前链路没有判断“这条事件是否已经提醒过”或“这次是否存在新增增量事实”，于是继续把同一催化当作新的 `triggered` 结果。
4. 用户在没有新增信息的情况下，会每 30 分钟收到一次相似的 ASTS 提醒。
5. 最近一小时同类根因已扩散到 `TEM大事件心跳监控`：对“会议进行中”这种持续性事件也会在相邻窗口重复送达，说明问题是 heartbeat 的通用增量检测缺口，而不是 ASTS 单任务特例。

## 期望效果

- Heartbeat 对已触发事件应具备“首报一次”或“仅在新增增量出现时再报”的抑制逻辑。
- 如果只是同一条发射计划在后续轮询里继续存在，而没有新增结果、延期、失败、完成或价格阈值变化，不应重复推送。
- 历史投递去重应覆盖“跨窗口的同一事实重复提醒”，而不只覆盖同一调度窗口内的重复发送。

  - `15:00` 又再次送达同一旧事件，只是把措辞改成“已成功发射”
- 这说明系统并不是稳定地“只首报一次”，而是会在不同窗口间对同一旧事实反复切换 `noop/triggered` 判定。
- 这些提醒的核心事实没有本质变化，都是围绕“今日发射/发射任务已确认”展开，只是措辞略有变化。
- 最新两次重复提醒之间，事件状态措辞还出现了“`已成功发射`”与“`计划于今日发射`”的前后摇摆，说明系统在重报同一旧事件时还会放大事实表达的不稳定性。
- 同期价格字段也基本保持同一组数字，说明不是因为价格阈值新增触发，而是重复消费同一旧事件。
- `17:30 -> 18:00` 的最新窗口进一步说明，这条根因已经不只影响 ASTS：`TEM大事件心跳监控` 也会对同一场 `AACR` 进行中事件连续两轮送达，只是改写摘要措辞而没有新增独立触发事实。
- `18:31 -> 19:01` 的更新窗口则确认 ASTS 旧事件仍在持续重复：同一 `BlueBird 7` 事件在半小时内再次送达两轮，只是从“今日发射窗口”改写成“已完成”。
- `19:31 -> 20:01` 的最新窗口说明，这种重复并不会稳定止血，而是会在 `noop` 与 `triggered` 之间来回摇摆：
  - `TEM大事件心跳监控` 在 `19:31` 先回落成 `noop`
  - `20:01` 又围绕同一 `AACR 进行中` 事件重新送达
  - `ASTS 重大异动心跳监控` 也在 `19:31` 短暂停发后，于 `20:01` 再次送达同一 `BlueBird 7` 旧事件
- `21:30 -> 22:00` 的最新窗口进一步说明，问题已经同时覆盖“同一事件连续半小时重报”和“旧事件在多轮 `noop/failed` 后重新触发”两种形态：
  - `ASTS` 在 `21:30` 与 `22:00` 连续两轮又重报同一 `BlueBird 7` 旧事件
  - `RKLB异动监控` 则把 `2026-04-16` 已完成的 Mynaric 收购在 `22:00` 重新当成当前新提醒送达
- 这与已修复的“同一窗口重复投递”不同；当前是任务本身每次轮询都会重新把旧事件当成新触发。

## 用户影响

- 这是质量与业务语义缺陷。Heartbeat 主功能仍可运行，用户也确实收到了提醒。
- 但同一催化在无新增进展时被反复推送，会制造噪音并削弱提醒系统的可信度，用户难以分辨“真正的新进展”与“重复播报”。
- 之所以定级为 `P3`，是因为它没有造成漏报、错投、系统失败或数据错误；当前主要伤害是提醒语义失真与消息噪音增加，而非主链路不可用。

## 根因判断

- Heartbeat 触发链路目前看起来只判断“当前是否满足触发条件”，没有持久化“这条触发事实是否已在近几轮送达过”。
- 对事件型 heartbeat，系统缺少“增量检测”能力，无法区分“旧事件仍然存在”与“出现了新的事件进展”。
- 对“会议进行中”“发射窗口日”这类持续型条件，系统也没有稳定的去重键或状态机，导致同一事实会随着措辞改写被反复当成新触发。
- 已修复的发送幂等只覆盖同一调度窗口内的重复发送，不能抑制跨窗口的同事实重报。

## 下一步建议

- 为 heartbeat 的 `triggered` 分支补充最近一次已发送事件摘要或事件哈希，避免相同事件在短时间窗口内重复推送。
- 对事件型任务增加“新增进展”判定，例如只有当事件状态从“计划中”变为“已发射/延期/失败”或出现新的公告时间点时才再次提醒。
- 在回归样本里补一条“同一事件跨四个轮询窗口重复出现”的测试，验证不会因旧事实常驻搜索结果而持续重发。
