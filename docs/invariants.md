# Invariants

Last updated: 2026-07-22

## Source of Truth and Document Priority

- Code and tests take priority over explanatory documents
- `README.md`, workspace manifests, and `config.example.yaml` are the primary implementation-level docs
- `docs/repo-map.md`, `docs/decisions.md`, and `docs/adr/*.md` hold long-lived context
- `docs/current-plan.md` is the dynamic index, and `docs/current-plans/*.md` carries the state of individual tasks that still need tracking; neither file should carry long-term rules

## Definition of Done

- Relevant verification must be completed before a task is closed
- Any affected context documents must be updated before a task is closed
- Cross-module long-lived behavior changes or architecture decisions must be recorded in `docs/decisions.md`, with ADRs added when needed
- At the end of a medium-or-greater task, if the task needs handoff, pause-and-resume support, or explicit retention of follow-up risk, a handoff must be left behind

## Planning and Handoff Constraints

- Parallel tasks must not share the same detailed plan; each task should use its own `docs/current-plans/*.md`
- `docs/current-plan.md` only tracks active task links, status, and links
- `docs/current-plan.md` and `docs/current-plans/*.md` are opt-in: they only record cross-turn, cross-module, behavior / structure / workflow changes, or tasks that need parallel collaboration, handoff, or blocker management
- Low-value one-off tasks such as a single commit / sync / rebase, small script or config fixes, no-behavior-change patches, and pure copy or formatting changes should not be mechanically written into the dynamic plan docs
- For the same topic, prefer reusing the original handoff instead of adding fragmented files on the same day or in the same phase
- A handoff is not a logbook; keep only the goal, result, verification, risk, and unfinished items needed by the next person
- Small pure-execution tasks do not require a new handoff unless the user asks for one, the task needs asynchronous follow-up, or the task changes the workflow / structure / risk surface

## Testing and Script Constraints

- Rust unit tests should live next to the implementation in `#[cfg(test)] mod tests`
- Rust integration tests should live under `tests/integration/`
- CI-safe regression scripts should live under `tests/regression/ci/`
- Regression scripts that depend on an external account, an external CLI, or local machine state should live under `tests/regression/manual/`
- One-off troubleshooting scripts should live under `scripts/tmp/` and must not enter CI
- Open-source collaboration uses layered test coverage: prioritize core Rust logic, key cross-module paths, frontend state / data logic, and keep external integration verification split between local contract tests and manual regression
- Coverage numbers are secondary to behavioral proof; do not introduce a repo-wide `90%+` hard gate or optimize for static UI line coverage
- Default CI proof must cover Rust tests, frontend unit tests, and CI-safe regression scripts
- High-risk logic changes must keep success-path, failure-path, and boundary-condition verification in automated tests whenever the behavior is CI-safe

## Investment Response Prompt Contract

- `soul.md` remains the complete investment-reasoning and answer-composition baseline restored from the pre-`71a4498e` large prompt. Refactors may move structural safety or caller-supplied typed scheduled/heartbeat enforcement into runtime code, but Interactive business prose remains Agent-owned and must not acquire a runtime semantic publication gate. No refactor may silently delete the prompt's time, entity, evidence, valuation, scenario, response-template, or anti-fabrication requirements.
- Every security, broad-market, and sector answer begins with a Beijing data-time and quote-basis line; no plan, greeting, heading, delimiter, or second time line may precede it. For open-ended Interactive turns, one live clock read creates both the Session context and the exact natural-DirectFinal answer-time anchor. On Web finance only, the service may own exactly one typed, non-market-claim first line using that Session time and the neutral basis `本轮仅使用可核验资料，具体报价时间与数据缺口在正文逐项披露`. A deterministic explicit-security fast path may ACK it before the first model call. A name-only finance turn may ACK it only after a budget-accepted batch contains at least one registered DataFetch call with a supported `data_type`, every required security target, and a valid identity shape where applicable, while every call in the batch is registered, structurally valid, and known read-only. Unsupported/missing-target DataFetch, malformed arguments, invalid identity declarations, and budget/route rejection never activate finance or deferred ACK. The Agent's final body must reproduce that prefix byte-for-byte. Provider-only leading Unicode whitespace left after hidden-reasoning stripping may be removed solely when the first non-whitespace content begins with the exact ACKed prefix; any non-whitespace preamble or real prefix mismatch still fails closed. The service may not synthesize a price, exchange, relationship, candidate, valuation conclusion, or any other open-ended market prose in the prefix, and may not semantically approve, repair, reject, or replace the completed body. Prefer `hone_quote_time.beijing` for quote-source time in the body; it must never replace the Session query time. `market_date_new_york` and generic `new_york` values prove only a market date/timezone, not NYSE, a close, or regular-session close. Exchange wording must come from structured `exchange` / `exchangeShortName` fields. Caller-supplied typed scheduled/heartbeat contracts retain their separate server-owned time/entity/quote prefix.
- Every nonempty interactive turn enters `AgentToolDiscovery`; the empty turn alone is `PassThrough`. This includes portfolio/watchlist wording: when membership matters, the main Agent calls actor-scoped `portfolio(view)` inside the same tool loop rather than a server phrase parser selecting a preflight route. Scheduled and heartbeat work may retain deterministic `Securities` / `Portfolio` / `Broad` preparation because their task body and execution contract are typed separately. Interactive requests must never stop at a server-side `confirmed no entity`, `needs clarification`, or phrase-grammar outcome before the main runner reads the complete current query.
- In the main agent loop, every named security discovered from the complete current request should receive its own current-turn `data_fetch(search)` call, including ordinary explicit tickers. Lowercase or mixed-case ticker spelling in a securities context still uses normalized exact-symbol lookup rather than alias ambiguity. The same Agent assigns a stable, distinct, case-sensitive `entity_route` to each named security, declares that individual search's call-scoped `identity_match=exact_symbol|name_or_alias`, and reuses the route verbatim on refinement, quote, profile/snapshot, and later DataFetch calls. Any identity search carrying an explicit route but missing a valid call-scoped `identity_match` is rejected before observer notification, registry execution, and provider/network access; it cannot inherit an older declaration or create/pollute ledger state. Independent calls may run in one parallel tool round, but multiple named entities must never be concatenated into one search query. Only a syntactically valid identity search may register a route; malformed, out-of-order, budget-rejected, or non-search calls never create or poison ledger state. An unknown alias whose returned symbol uniquely matches one accepted route is merged into that route instead of opening another. Interactive finance admits at most six identity routes for the complete turn, applies that limit while admitting the first batch, and rejects every later new route before execution. It instructs the Agent to prioritize user-named anchors before at most four screening candidates. Because the service does not infer semantic entity ownership from model-authored routes, this order-based ceiling cannot mechanically guarantee coverage when the user names more than six securities; the bounded final must disclose omitted coverage instead of expanding the crawl. Route-bound quote/profile/snapshot or other follow-up evidence is accepted only after that route's own valid search declaration and result. Before the structural floor permits `Auto`, every accepted route with candidates must have one same selected symbol in its route-bound quote and profile/asset-route evidence. Overlapping provider candidates, pre-search evidence, an unrelated quote, a mismatched profile, history, model memory, ranked-first results, scheduler metadata, another route, or partial provider output cannot establish it. This floor guides tool choice only and never gates a complete natural answer.
- The lexical scanner owns audited bare/cashtag/exchange-qualified/share-class/index/crypto-pair spans and syntax and delegates provider dialect/canonical equivalence to the shared `hone-core::provider_symbol` truth source, but its Interactive output is only a non-factual candidate seed. It cannot declare the entity set complete, trigger clarification, or prevent the main Agent from finding names, aliases, mixed identifiers, or unusual wording. The service must not infer entity boundaries or match mode by splitting commas, `and`, `和`, `&`, `/`, share-class text, case, length, or ticker shape; AT&T, S&P Global, M&T Bank, H&R Block, BRK/B, and `Berkshire Hathaway, Class B` remain for the Agent to understand from full context. The Agent's first `exact_symbol` declaration fixes a persistent route constraint, using only audited provider equivalence; later name refinement cannot replace CRWV with an embedded-name product such as CWY. `name_or_alias` grounding preserves word boundaries for short names such as Ford, Apple, and Tesla. A later successful search under the same `entity_route` replaces earlier empty/noisy candidates and resets the current empty-attempt streak. Untagged legacy searches form provisional per-query routes and remain active alongside tagged routes until an explicit search names one exact, case-sensitive old query through the actual `query`, `refines_query`, or `supersedes_query`. `refines_query` and `supersedes_query` are strictly mutually exclusive; if both appear, no provisional route migrates and the explicit route remains pending. A valid single-source migration carries only search/empty attempt history, verbatim query aliases, and an already Agent-declared exact-symbol constraint; it never imports candidates, quote symbols, profile/asset-route symbols, or post-identity attempts. One tagged entity must never hide all untagged entities. A route still empty after two current consecutive search attempts requires a real route-correct non-search attempt after the latest empty result before bounded disclosure; an unscoped no-symbol follow-up may be assigned only when exactly one active route exists. Amounts, years, percentages, cadence, target prices, and metric/technology acronyms remain excluded as lexical securities. The blocking auxiliary-LLM JSON extractor and its timeout/failure result are not part of current-turn entity discovery.
- For interactive turns, the Agent reads the complete query and its observed tool trace—not a service-side keyword list—determines which evidence is loaded into context. The service must not infer “quote-only”, “valuation”, “recent analysis”, “extended hours”, or a required response template from user wording. It may observe deterministic current-turn boundaries for telemetry, but those observations cannot authorize publication or change a successful body. It must not parse arbitrary Agent-authored financial, holding, news, event, or valuation chapters, prepend facts, normalize prose, continue omitted seeds, retry/repair the answer, rewrite content, or reject success. One canonical answer/session-history/SSE terminal remains mandatory; typed scheduled/heartbeat contracts retain their separate deterministic validation and fallback path.
- Interactive function-calling output keeps model-authored tool-capable prose speculative behind `DeferredUserOutputEmitter`. Every nonempty request may use the open Agent tool loop, but only a budget-accepted current-turn DataFetch call with a registered tool name, supported `data_type`, all required targets, valid identity shape, and parseable arguments activates finance guidance. Unsupported/missing-target/malformed/rejected DataFetch retains ordinary answer semantics and never activates deferred ACK. Active finance research stays in one `Agent::run`, one context, and one model role across Web, Feishu, Discord, and Telegram; its request flag is independent of the Web-only prefix publication policy. Provider requests use the composed current input plus at most four prior user utterances / 4,000 characters, spending that budget newest-first and labelling it as reference-only; they exclude historical assistant/tool/reasoning facts. Production exposes only real actor-bound business tools: it has no `finish_research`, completion handoff, source catalog, locator correction, separate terminal synthesizer, terminal audit, post-answer semantic review, or second generation. Finance research has intrinsic ceilings of three tool-call batches, 24 total calls, 20 DataFetch calls, and six Web calls even when the caller supplies no tighter budget. A T0 ACK begins this accounting immediately, so Web-only batches before DataFetch activation still consume the batch, total-call, and Web-call budgets. At the first exhausted ceiling, the next request in the same Agent loop has `tools=[]` and asks that same Agent to form one natural evidence-bounded final from the existing context; this is the ordinary next iteration, not the retired second terminal generator.
- The request-local structural ledger records only accepted Agent-declared identity routes, their current candidates, bounded empty attempts, route-bound quote/profile/snapshot, broad-market, symbol-scoped, and crypto `search → crypto_quote` paths. Before the structural floor, business tools use `Required`; afterward the same real tools use `Auto`, allowing another tool call or one natural nonempty `Stop + Done` DirectFinal. The ledger, route ceiling, and tool ceilings constrain research work and produce telemetry; they do not semantically approve, replace, suppress, or rewrite a complete answer. After a service-owned prefix is ACKed, every call in every later batch must name a tool in the active registry, have parseable structurally valid arguments (including supported DataFetch type/target/identity shape), and be classified as known read-only. The whole batch is rejected before assistant tool framing, observer notification, registry execution, or network access if any call fails that conjunction; an unknown but read-only-looking alias is still unknown and cannot enter the registry.
- Keep all matching current-turn tool results in the same Agent context. DataFetch search remains identity-only; quote/profile/financial/relationship facts require their own current evidence. Web output is bounded snippet evidence, not a page body or ranking. Relationship direction and strength must stay within current evidence; absent direct support, use neutral wording and do not elevate a relationship to core/largest/major customer or supplier, high dependence, locked/exclusive/priority status, multiple binding, ownership/non-ownership, GPU model, contract value, or equivalent detail. Genuine missing coverage is disclosed narrowly and never widened into a fixed whole-answer refusal.
- Any complete nonempty `Stop + Done` natural body is the same Agent's DirectFinal. Publish it once after security/path/media finalization and persist byte-identical bytes. Empty response, provider error, timeout, lifecycle failure, or iteration exhaustion remains a genuine internal failed run and never synthesizes completion authority. If no prefix was delivered, the normal error boundary applies. If the typed service prefix was actually ACKed, the public transcript appends the fixed suffix `本轮研究未能完成，暂未形成可供参考的标的结论。`, persists exactly those visible bytes with partial/failure metadata, and closes once without an error-card flash; this suffix is an explicit failure disclosure, not a security conclusion or substitute answer. Exactly one typed finish and one internal DONE still validate provider lifecycle; compatible clean-EOF normalization remains limited to one framed finish with no later payload.
- Public-chat history restore is fenced from a newer local send. Each restore captures sync and send generations; before creating the next optimistic user/assistant pair, a send advances both generations, aborts the prior restore, and cancels its retry. An older bootstrap may not reconcile over that pair, drop its first deltas, reset its timer, or recreate a completed run. Chat POST remains one-shot; recovery observes authoritative server state.
- The actor-scoped portfolio store is the only truth source for requests about “我的持仓 / 我的关注 / 我的组合” and must be read by the main Agent through `portfolio(view)` in the same interactive tool loop. Stored symbols, quantities, and costs may establish portfolio membership, but not current market facts: requested current-price/performance analysis must continue through current-turn DataFetch search and same-symbol quote verification for relevant tickers. The answer must disclose limited coverage rather than present a subset as the whole portfolio. A read call never substitutes for an explicitly requested portfolio mutation. Typed scheduled/heartbeat preparation may still use the service-owned bounded portfolio preflight.
- Broad-market turns resolve each requested scope and representative benchmark. Sector turns discover representatives from current theme evidence and exact-resolve/quote at least three related securities. Mixed scopes retain independent dates and evidence rather than overwriting one another.
- Current prices and changes come from same-symbol DataFetch quotes with provider timestamps; an explicit US premarket/postmarket question additionally requests the exact symbol's bounded extended-hours minute bar and accepts it only when session and freshness match. Every factual extended-hours price sentence must preserve the verified entity, requested session, price, and, when stated, currency even when the session marker follows the number or English/Chinese modifiers intervene. A failed/stale/mismatched extended-hours lookup may retain the regular quote only with an explicit regular-session label and an extended-hours-unverified disclosure; it must never relabel the regular quote as premarket/postmarket, and crypto/night-session requests must not be mislabeled as US extended trading. Dated web search supplies news, causes, policy, and current-event context; the Agent's answer-composition contract still requires a real verified absolute date and matching full source domain for event facts, or an explicit inference/hypothesis/scenario label. The interactive service does not attempt to prove that requirement by parsing arbitrary prose clauses. Exact per-claim runtime enforcement for financials, holdings, valuation derivations, or events requires structured claim provenance tied to tool results. A successful quote forbids user-facing claims that current/realtime market data was not requested or is unavailable; describe it as latest available and non-tick-by-tick.
- Provider transport/auth/quota failures, successful empty exact coverage, genuine multi-market ambiguity, and stale/missing quote timestamps are distinct states. A provider error object is never equivalent to an empty result, and neither state may be rewritten as “the ticker does not exist” or generic “entity recognition incomplete.” Interactive post-run investment reconstruction is observational only: success, failure, completeness, or absence of the observation may be logged but may never change the successful Agent response. Completed Interactive output uses only system/internal-protocol and unsafe-local-path cleanup plus media-reference stabilization; it must not use legacy market-copy normalization, planning-sentence vetoes, tool-result reconstruction, or the typed investment completeness guard. No service-authored `UnsafeIncomplete`/“cannot safely publish” fixed refusal may replace it. The finalized body emitted to the user and the assistant body persisted in history must be byte-identical. Bounded exact-quote batches and reusable probes remain the typed scheduled/heartbeat preparation path, whose typed fail-closed behavior is unchanged.
- The established equity, fund, crypto, broad-market, sector, valuation, scenario, and action patterns remain the prompt baseline, but the interactive Agent chooses and adapts them from the complete request. Runtime code may not maintain a phrase vocabulary to decide that a turn is “deep” or force numbered headings onto a dynamic answer. Typed scheduled/heartbeat contracts may retain deterministic templates. Company financial-data absence is disclosed specifically as unverified for the current turn and may not be widened into a false quote outage or filled from memory.
- `tests/regression/ci/test_finance_automation_contracts.sh` must statically prove the canonical prompt/skill markers above so a future prompt simplification cannot repeat the `71a4498e` regression unnoticed.
- Company portraits are document-first assets: `profile.md` plus `events/*.md` is the source of truth, while any parsed metadata or API projections are derived views and must not silently diverge from the Markdown files
- Company portrait docs must live inside the current actor sandbox under `company_profiles/`; do not reintroduce a shared/public portrait directory outside actor-scoped user space
- Company portrait reads/imports must treat frontmatter as optional. If `profile.md` or `events/*.md` omit YAML frontmatter, readers and transfer preview/apply flows must still infer minimal metadata from the title, filename, file mtime, or bundle manifest instead of failing hard on `缺少 frontmatter`
- Default PR / push CI excludes `hone-desktop` from workspace-wide `cargo check` and `cargo test`; desktop sidecar resources and packaging checks belong to dedicated desktop build or release flows instead of the generic logic gate
- Local IDE / dev Rust checks may set `HONE_SKIP_BUNDLED_RESOURCE_CHECK=1` to bypass Tauri bundled sidecar validation while still type-checking `hone-desktop`; this flag is only for development syntax checking and must not replace real desktop packaging validation

  - `sccache` is the compiler object cache layer
  - `main` branch prewarms release-target caches, and tag releases should primarily restore from that warmed cache rather than create fresh tag-scoped caches

## Security and Environment Constraints

- Do not hardcode secrets in docs, scripts, or tests
- Do not add flows that depend on external account credentials to the default CI gate
- Diagnostic scripts should avoid mutating business data
- Treat iMessage features as local privileged capabilities; do not assume they can run in generic environments
- Tool calls must go through a security guard that blocks risky command fragments by default
- 用户可见的长期研究记忆目前只保留 company portraits；不要重新暴露 KB 页面、KB API 或 `kb_search` 类记忆入口
- Company portraits are long-term research assets, not trade execution artifacts: they may store investment mainline, moat, management, financial quality, capital allocation, valuation frame, risks, and dated event deltas, but they must not evolve into implicit day-trading or automatic recommendation logs
- Company portrait mutations must stay agent-mediated: the Web console may render portrait documents and event timelines, while create / update / append flows should still go through the agent's native file operations inside the actor sandbox. The only direct UI mutations allowed are full portrait deletion plus actor-scoped company-profile bundle export / import for explicit sharing and restore workflows; UI import must stay directory-level (`skip` / `replace`) and must not grow into section-level merge editing
- Company portraits should preserve not only the current conclusion but also enough rationale to make future review possible: `profile.md` should keep the current investment mainline, key operating metrics, valuation frame, risk ledger, and disconfirming conditions, while event docs should retain why the event mattered, supporting evidence / refs, and a compact research trail when no standalone research-notes layer exists yet
- Before analyzing a company, if the current actor sandbox already contains portraits for similar companies, Hone should inspect those related portraits first and reuse the same macro / industry narrative unless a company-specific variable clearly justifies a different conclusion. Cross-company differences should be explained by company-level facts, not by silently rewriting the shared macro investment mainline.
- Storage mode must be explicit. `cloud.mode=local` is the default and must remain a fully supported runtime path; PG / OSS env variables on a developer machine must not silently force local users onto cloud storage. `cloud.mode=cloud` is the only mode that claims PG / OSS authority, and strict cloud mode may only report durable local dependency count as zero after the relevant hot-path stores are actually backed by PG / OSS.
- Distributed multi-machine deployments use PG for state, indexes, locks, claims, fencing, and quota semantics, and OSS for durable file/object bodies. OSS alone must not be used as the source of truth for quota, cron claims, session write fencing, auth sessions, or worker leadership because it does not provide the required transaction and locking guarantees.
- Public-community PG rows remain the canonical archive. Edge feed pages, resource descriptors, and the active resource index are derived delivery artifacts only. `community-publish --apply` must materialize a repeatable-read/read-only PG snapshot on the dedicated advisory-lock session, fully verify eligible managed R2 bytes before any publication write, recheck that lock before writing, refuse empty/conflicting/oversized output, publish descriptors then mutable `active.json` then cursor pages then `latest.json`, read back mutable writes, and treat explicit unlock failure as failure. Dry-run may use HEAD existence checks but must label that weaker proof accurately. Redis is not a substitute for this durable source or for the R2 binary hot path; it may be reconsidered only for measured metadata/state contention.
- Community edge storage must remain private. Do not enable an R2 `r2.dev` URL or bucket custom domain for this path, do not expose object keys as unauthenticated URLs, and do not put source cookies, signed source URLs, public-session tokens, edge tokens, or signing secrets into PG projections, R2 objects, logs, committed config, or frontend JSON.
- The community Worker is fail-closed: an absent/empty/unknown/true `EDGE_DISABLED`, a trimmed HMAC secret outside 32..1024 UTF-8 bytes, a missing R2 binding, an invalid prefix, or an invalid/expired/duplicate edge cookie must never reach Cache API or R2. Only the explicit values `false`, `0`, `no`, or `off` enable the negatively named switch; production procedure should use exactly `EDGE_DISABLED=false`. The backend grant defaults to `cloud.community_delivery.mode=off`, applies the same secret byte boundary, clamps token lifetime to 60..3600 seconds, stores the token only in an HttpOnly/Secure/SameSite=Strict cookie scoped to `/_community/v1/`, and clears it on off-mode, invalid secret, failed auth, or logout.
- Community edge authentication must happen before every shared-cache read and write. Every resource request must additionally validate its id/version against an uncached, fail-closed active index before shared-byte cache lookup; cached bytes can never bypass later revocation, and resource shared-cache TTL must not exceed one hour. Canonical cache keys may omit user identity only because published feed/resource bytes are non-personal and the Worker has already validated a current actor-bound grant. Browser responses remain private and vary on Cookie; personal unread/seen state stays on the authenticated backend API. Worker errors, invalid descriptors/integrity results, and legacy-origin responses must never populate the shared cache.
- Community edge compatibility remains mandatory. The frontend may discover the edge path only when its build flag is enabled and the backend returns an enabled `prefer` grant for the exact `/_community/v1` path; `off`, `shadow`, malformed/expired grants, and all edge feed/resource failures return immediately to the existing API. The Worker may fall back only to the fixed `https://origin.hone-claw.com` legacy mapping, must request identity encoding and enforce the same 8MiB feed/128MiB resource response ceilings, may forward only the single validated `hone_web_session` cookie, and must not forward `hone_community_edge` or unrelated browser cookies. Resource HEAD failures, inactive versions, invalid/missing active indexes, invalid descriptors, and integrity failures must not internally fall back. Existing `/api/public/community`, `/seen`, and `/resources/:resource_id` behavior must remain usable throughout rollout and rollback.
- Skill runtime activation comes from the global skill registry override layer: local mode uses `data/runtime/skill_registry.json`, cloud mode uses PG `cloud_skill_registry`. Web/admin skill toggles and runtime skill loading must observe the same backend in a given storage mode.
- Notification preferences are runtime state, not static seed files: local mode uses per-actor JSON files in `data/notif_prefs`, while cloud mode uses PG `cloud_notification_prefs` so tool edits, Web edits, event-engine routing, schedule overview, and mainline distill see the same actor preference row.
- Portfolio state must remain actor-scoped and backend-consistent: local mode uses `data/portfolio/portfolio_*.json`, while cloud mode uses PG `cloud_portfolios` so portfolio tool edits, Web portfolio views, event-engine subscription refresh, and digest/admin reads see the same actor portfolio row.
- LLM audit is append-oriented runtime evidence: local mode uses SQLite at `storage.llm_audit_db_path`, while cloud mode uses PG `cloud_llm_audit_records` so runtime writes and Web audit list/detail reads observe the same audit backend. Do not treat the SQLite file as a blob substitute once PG audit import is available.
- Company profiles are actor-scoped long-term research memory: local mode uses actor sandbox `company_profiles/<profile_id>/profile.md` and `events/*.md`, while cloud mode uses PG `cloud_company_profile_files` for the same logical files. Existing native-file runner writes must remain compatible by syncing successful-turn sandbox Markdown into PG; Web/API/event-engine reads must go through `CompanyProfileStorage` rather than direct filesystem scans.
- Generated images are local cache artifacts in local mode but must be OSS-addressable in cloud mode: response finalization must upload generated images from actor sandboxes or `storage.gen_images_dir` and return `oss://...` markers when OSS is configured. Cloud strictness must not depend on durable `file://` image paths.
- iMessage remains a local worker/all-mode integration. `HONE_RUNTIME_ROLE=web` must not start scheduler, event-engine, channel listeners, or iMessage sidecars; `worker`/`all` are the only roles allowed to run local privileged integrations.
- Non-local Web console deployments must enable a Bearer token
- Public web auth must use server-side sessions backed by an HttpOnly `hone_web_session` cookie. New session tokens must be high-entropy random values and only their hash may be persisted; legacy plaintext session rows may be accepted only as a compatibility path until they expire or are rotated. Auth rejection logs must not include raw cookie or session token values.
- Public user login must be phone number + SMS verification as the primary path. The active, non-revoked admin-created web invite user list is the invite-list admission source; phones outside that list must fail closed before SMS send/check. When Aliyun Captcha 2.0 prefix and scene id are configured, every public SMS send must pass server-side `VerifyIntelligentCaptcha` before calling the SMS provider. Aliyun SMS and Captcha credentials must come from environment/configuration, never committed source.
- `ChatMode` describes only the message shape (`direct` / `group`) and does not determine session ownership; shared group context must explicitly go through `SessionIdentity`
- `ActorIdentity` and `SessionIdentity` must stay separate: the former is for permissions, quota, sandbox, and private-data isolation, while the latter is for context recovery and session persistence
- Global finance-domain constraints are injected at runtime by `crates/hone-channels/src/prompt.rs`: no stock-picking recommendations, reject non-finance questions, warn users not to blindly follow buy or sell advice, keep greetings short, and require macro / market narrative analysis to distinguish noise from investment mainline-changing evidence. Do not flip between conflicting narratives on a few days of price action or a single headline unless the prior hypoinvestment mainline has been explicitly falsified, and do not override these core rules only in a single channel or in a local config.
- Runtime prompt time anchoring is a core behavior contract: Hone must keep the session-provided current time as the source of truth for macro / news / event-driven analysis, must state the current time first on clearly time-sensitive macro answers, and must rewrite relative-time macro searches into absolute-date queries before calling search tools.
- `config.yaml` is the only long-lived user-writable config source
- LLM credentials are config-only: runtime LLM provider/profile/auxiliary/agent paths must not read `*_API_KEY`, `*_BASE_URL`, or `api_key_env` as a fallback. User-facing setup should write inline `api_key` / `api_keys` under `config.yaml` and mask those fields in UI/log output.
- `data/runtime/effective-config.yaml` is the generated runtime input for child processes, and deleting `data/runtime/` must be a safe runtime reset that does not remove user config
- No steady-state runtime path should read or write legacy `data/runtime/config_runtime.yaml` or sibling `.overrides.yaml` files anymore; the only allowed exception is one-way startup migration that promotes still-missing user settings into canonical `config.yaml`
- `storage.session_runtime_backend` decides the production session read path:
  - `json`: `data/sessions/*.json` is the source of truth
  - `sqlite`: `storage.session_sqlite_db_path` is the source of truth
- During the current rollout, even when `session_runtime_backend=sqlite`, JSON should continue to dual-write as a rollback mirror until SQLite stability is proven over time
- Desktop runtime config materialization must normalize historical canonical configs so `storage.session_sqlite_shadow_write_enabled` remains `true`; old generated runtime snapshots or seeded configs must not silently disable JSON -> SQLite session mirror writes
- `SessionStorage` should best-effort backfill existing JSON sessions into the SQLite index during startup whenever an index is configured, including `session_runtime_backend=sqlite` even if `session_sqlite_shadow_write_enabled=false`; this keeps mirror lag recoverable after any historical window where shadow write was disabled, while preserving the configured runtime read authority
- In cloud-authoritative mode, PG `cloud_sessions` remains the session truth source, but if `storage.session_sqlite_shadow_write_enabled=true` and `storage.session_sqlite_db_path` is configured, successful session writes should still best-effort dual-write the local `sessions.sqlite3` mirror and allow synthetic `cloud_sessions/<id>.json` source paths so bug triage / recovery tooling does not silently go stale

## Agent Runtime Constraints

- Use `agent.runner` as the single source for runner selection; channels and the Web UI should not branch `gemini_cli` or `codex_cli` execution paths on their own
- `AgentSession` exposes `run()` as the only public entry point; its responsibilities should stay limited to session orchestration, persistence, and listener dispatch. When adding a new execution path, prefer extending the unified runner contract instead of adding a new `run_xxx` branch.
- Shared execution preparation belongs in `crates/hone-channels/src/execution.rs`; session flows and transient task flows should reuse it instead of each path rebuilding tool registry / runner / sandbox wiring on its own
- Runner selection and provider / CLI differences belong in `HoneBotCore::create_runner()` and `crates/hone-channels/src/runners/`; do not reintroduce provider-specific checks in channel entrypoints or `AgentSession`
- Heartbeat and other transient scheduler tasks may bypass transcript persistence, but they must still reuse the same execution-preparation path instead of directly instantiating concrete runner types inside `scheduler.rs`
- A channel actor's local file visibility must stay inside a repo-external actor sandbox. The default root lives under `hone-agent-sandboxes/` in the system temp directory; `HONE_AGENT_SANDBOX_DIR` may override it only when the resolved path is outside the current source checkout / git worktree.
- Actor sandboxes must not contain portfolio storage files. Local portfolio truth remains `storage.portfolio_dir` through `PortfolioStorage`, and cloud portfolio truth remains PG `cloud_portfolios`; if legacy `portfolio_*.json`, `portfolio/`, or `portfolios/` entries appear inside an actor sandbox, sandbox initialization must remove them before handing the directory to a native-file runner.
- Channel attachments must be written to `uploads/<session_id>/` inside the actor sandbox; do not point the underlying runner `cwd` back at the repo root or at a shared upload directory inside the repo
- 用户可见的运行进度允许保留执行细节，但如果文案中包含 actor sandbox 内的绝对路径，必须改写为相对 sandbox 根目录的路径；sandbox 外绝对路径不得原样透出
- Runner timeout config must stay converged at `agent.step_timeout_seconds` and `agent.overall_timeout_seconds`; do not reintroduce runner-specific timeout knobs in channel/runtime config.
- The strict FunctionCalling runner must honor both converged timeout knobs: `AgentRunnerRequest.timeout` (falling back to configured overall only when omitted) and configured step timeout. It creates one absolute deadline at Agent entry and shares it across all initial/active model calls, tool execution, and tool observers; phases must not reset the overall budget, while each individual await receives a fresh step bound. Every attempted tool failure must remain in `ToolCallMade` as `status=failed` / `isError=true`, with timeout marked explicitly before the Agent stops. Tool side-effect classification is canonical in `hone-core`; an uncertain persistent failure must stop inside the current Agent loop, while failure progress must never be rendered as successful completion.
- `function_calling` and `multi-agent` are retired. Their old config strings must fail explicitly rather than silently falling back. `gemini_acp` stays deserializable only for legacy migration/reference; route new users to `codex_acp`, `opencode_acp`, `hone_cloud`, `gemini_cli`, or `codex_cli`.
- `gemini_cli` in channel runtime must default to sandboxed execution and `approval-mode=plan`; it must no longer default to `yolo`
- `codex_acp` uses `@agentclientprotocol/codex-acp` over stdio / JSON-RPC and sets `CODEX_PATH` to the configured Codex binary. Startup must verify `@openai/codex >= 0.144.1` and `@agentclientprotocol/codex-acp >= 1.1.2`, otherwise fail fast with a clear upgrade command.
- The canonical default local runner is `codex_acp` with `model=gpt-5.6-sol` and `variant=xhigh`; `config.example.yaml`, Rust defaults, Desktop/Web defaults, CLI onboarding, and active Codex automations must stay converged on that combination.
- `codex_acp` must create a fresh ACP session for each Hone turn and seed it from Hone's restored local transcript/context. Do not use old `codex_acp_session_id` metadata to call ACP `session/load`; historical `session/update` replay can reintroduce prior prompt/tool events into the current user-visible stream.
- Native CLI/ACP runners (`gemini_cli`, `codex_cli`, `codex_acp`, and `opencode_acp`) are trusted-host capabilities because their subprocesses may inspect files or process state outside Hone's actor-bound tool registry. Only explicit channel/CLI administrators may use them. When a non-admin actor cannot use the configured trusted-host runner, the selected strict in-process function-calling fallback must retain the actor-bound tool registry; absence of that fallback's LLM configuration fails closed. Restore, compaction, and context-ownership decisions must use this effective per-actor runner rather than the globally configured runner name.
- Actor sandbox roots and runtime storage directories must be owner-only (`0700` on Unix), while canonical/effective config files and SQLite database files must be owner-only (`0600`). Startup code must re-apply these modes instead of relying on the caller's umask.
- Executable skill scripts are trusted repository code, but their subprocesses must start from a cleared environment and receive only the small runtime allowlist needed for execution/artifact output. Database, object-store, channel, and LLM credentials must not be inherited.
- Credentialed public CORS must use the HONE production-origin/localhost allowlist plus optional exact origins from `HONE_PUBLIC_ALLOWED_ORIGINS`; it must never mirror an arbitrary request origin.
- `opencode_acp` currently uses `opencode acp` over stdio / JSON-RPC; like `codex_acp`, it must seed each fresh ACP session from Hone's restored local transcript/context rather than relying on remote session replay for continuity.
- When `agent.opencode.model` / `api_base_url` / `api_key` are empty, Hone must inherit the user's local OpenCode config instead of shadowing `~/.config/opencode/opencode.json` via a separate config home
- If `agent.opencode.model` is non-empty, Hone must call ACP `session/set_model` before `session/prompt`; `agent.opencode.variant` should be appended to `modelId` through the same call (for example `openrouter/openai/gpt-5.6-sol/xhigh`) instead of relying on temporary selection state in the local opencode UI
- Session compression may use the auxiliary LLM route, but heartbeat execution must reuse the configured runner through `ExecutionService` with its own narrow tool allowlist and per-tool budgets. Prefer `llm.auxiliary_profile` / `llm.profiles` for background compression; direct `llm.auxiliary` remains the OpenAI-compatible fallback when no profile is configured, and `llm.openrouter.sub_model` remains only as the final legacy fallback.
- Before Hone has its own ACP permission negotiation layer, `opencode_acp` must deny one `session/request_permission` request by default and must not silently allow file writes or terminal execution; the channel runtime may inject a minimal custom `OPENCODE_CONFIG`, but it must be a narrow permission overlay rather than a full replacement for the user's local OpenCode config
- The system prompt must stay layered:
  - `soul.md` is the complete investment reasoning and response-format contract; do not compact away task routing, analysis order, valuation discipline, Bull/Bear/Base framing, or answer-structure rules
  - Detailed live-data, privacy, storage, channel, cron, and security enforcement remains in Rust; intentional overlap with high-value `soul.md` behavior is allowed, and the more specific current runtime rule wins on conflict
  - Canonical `soul.md` is authoritative; effective-config generation must refresh, not preserve, a stale generated runtime copy
  - Static system instructions live in the prefix
  - Session-fixed context is concatenated separately
  - Mutable content such as summaries must not be written back into the static system prefix
- Do not inject the complete skill catalog into every system prompt. Keep generic skill instructions compact, attach only query-relevant skill summaries to the current turn, and use `discover_skills` when the initial match is insufficient.
- Pre-compact cache stability is a runtime contract: before the next compaction boundary, Hone must not introduce avoidable cache misses by shrinking its own restore window below the active compaction threshold or by injecting turn-specific related-skill hints into the static system prefix. Turn-specific guidance belongs in the current turn input; a post-compact prefix change is expected and acceptable.
- ACP runners must receive the Hone-assembled system prompt explicitly; they must not rely on the underlying CLI discovering `AGENTS.md` or `GEMINI.md` from the repo `cwd`
- Interactive entity discovery, evidence loading, and answer generation stay inside one main Agent loop. For a company/security request, the first model round identifies covered named entities from the complete query and returns separate DataFetch search calls with Agent-declared stable `entity_route` and call-scoped `identity_match`; explicit-route searches lacking a valid match mode and routes above the first-batch six-route ceiling fail before registry/network access without ledger pollution. Later finance rounds use only real actor-bound business tools. Finance requests may retain at most four prior user utterances / 4,000 characters for follow-up references, but must exclude historical assistant/tool/reasoning claims and treat the retained user wording as neither current facts nor authority to add unreferenced entities. The structural floor drives `Required → Auto` guidance, while runtime independently caps the turn at six valid identity routes, three finance tool batches, 24 total calls, 20 DataFetch calls, and six Web calls. A typed-prefix ACK starts those counters immediately, including Web-only pre-DataFetch batches; exhaustion moves the same Agent's next iteration to `tools=[]` for one natural evidence-bounded final. Production must not expose `finish_research` or run a handoff, locator correction, separate terminal synthesizer, audit, semantic review, or second generation. After the loop, current-run reconstruction is observation/audit only and must never normalize, retry, repair, rewrite, reject, or otherwise mutate a successful Interactive body. Completed bodies cross the dedicated security-only finalizer and the exact finalized bytes are emitted and persisted. Model-authored draft content, resets, thoughts, attempt-local errors, and progress for unregistered tools stay behind `DeferredUserOutputEmitter`; the sole pre-final exception is the typed Web finance prefix defined below. Users receive one byte-consistent answer and one terminal result without reset/replay. Scheduler and heartbeat keep their typed origin, original task body, deterministic preparation, strict validation, and separate fail-closed rules.
- Only caller-supplied typed scheduled/heartbeat `InvestmentResponseContract` values make normalized entity, exact current quote, price change, and quote source time server-owned publication facts. Those typed facts must precede the model-authored body and survive typed format failure. Profile snapshot-price fields, entity-mismatched news, and semantically ambiguous financial fields must be removed or explicitly labeled unavailable before typed evidence reaches the model. The Web Interactive prefix is a narrower structural exception: it owns only the current Session answer time and a neutral evidence-policy disclosure, never an entity, price, quote time, exchange, relationship, candidate, valuation, or completeness fact. An Agent-discovered Interactive observation cannot authorize any other server-generated prefix or body change.
- After a successful but incomplete typed scheduled/heartbeat draft, supported quote/equity/fund/crypto/market scopes may use a deterministic server fallback built only from the prepared typed contract; the fallback must be sanitized, pass the same final validator, and discard the rejected draft's tool/transcript metadata. Interactive turns may not use that fallback or a second whole-run/repair attempt for dynamic observation. Runner failures, persistent/uncertain side effects, and unsupported typed fallback scopes stay fail-closed; specialized comparison/sector repair must not replay the original user operation. Automatic retry, context-overflow recovery, typed fallback, or typed contract repair is permitted only when every observed tool call is on the explicit read-only allowlist; unknown tools fail closed. `deep_research` is always persistent, while only known actions such as `portfolio view`, `cron_job list`, and `notification_prefs get/get_overview` are read-only. Any request that may persist, send, schedule, delete, complete a trade, start research, or otherwise mutate state is execute-once in both Chinese and English when runner traces are absent or uncertain. The no-trace intent classifier must catch those operations before execution without treating ordinary investment-advice questions as mutations. A hidden attempt cannot justify duplicate execution.
- Current-quote validation must recognize common Chinese and English aliases without allowing a second conflicting price. Historical, open, close, high, or low prices have no verified field in the current contract and therefore fail closed even when accompanied by a date/domain, marked as inference, or numerically equal to the verified current quote. Historical/OHLC semantics in a Markdown table header apply to the numeric rows below it; target/scenario tables and same-symbol current-price tables without date, historical, or OHLC semantics remain valid. Event inference/condition subheadings may govern only their immediately following Markdown list items; ordinary prose or unknown headings reset that mode.
- Web chat activity is server-authoritative and actor/session-scoped. `run_id`, original server start time, phase, and safe status text describe a live runner; daily quota `in_flight` is only a reservation count and must never drive pending UI or elapsed time. Refresh may resume observation of the same detached run but must not replay the POST or reset elapsed time. Shared browser API retry is limited to idempotent `GET` and `HEAD`; `POST /api/public/chat` and every other non-idempotent request are one-shot at the client layer even after a transient status or transport exception. A missing active run plus an unanswered persisted interactive user turn is terminal interruption, not a new local thinking timer. Raw tool/provider/reasoning fields must never be rendered as public progress, and controlled runtime shutdown must drain registered active turns before terminating the backend.
- Web `run_finished` is unique and terminal. Once emitted or received, the SSE producer and client observer must close and ignore late assistant/progress frames; no fallback path may synthesize a second finish event or reset a completed run into thinking state. A non-Abort stream interruption stays in the same pending assistant card and enters authoritative bootstrap/history recovery before any error is shown; only exhausted recovery becomes an error. Explicit abort/stop closes immediately and does not enter disconnect recovery.
- DataFetch semantic-empty results for identity, quote, profile, financials, or holdings must not be cached as durable evidence. FMP key rotation is restricted to authentication, quota, and rate-limit failures; transport, parse, server, and ordinary provider failures must stop without credential fan-out.
- `web_search` is basic snippet evidence, not fetched page content. Successful output must be capped again to three results, overwrite any provider-supplied `hone_search_contract` / `hone_evidence`, mark citation scope and original-URL availability, and state that order/score is not a real-world rank. Current-turn Web results stay in the same Agent context and support only facts directly present in the selected title/content/snippet and original URL. A URL, result order, score, generic cooperation phrase, or model memory cannot prove relationship rank, role, right/obligation, ownership/non-ownership, exclusivity, priority, concentration, product model, or contract value. Any additional judgment must be explicit and bounded; when strength is not established, use neutral wording. This is Agent evidence discipline, never a final-answer parser, semantic veto, post-answer rewrite, retry, or refusal.
- Native CLI/ACP remains administrator-only. Non-admin actors use the strict in-process function-calling fallback with the actor-bound registry; absence of its LLM configuration fails closed.
- Web model context must follow the runner that will actually execute the actor's turn, not only the globally configured runner. Admin/public Web routes must keep a bounded direct-message restore window; before an Interactive runner request is assembled, complete prior scheduler/heartbeat/job groups and prior `run_failed=true` groups are removed from runtime context. This pruning must not delete or rewrite durable session history, bootstrap/history projections, or scheduler records.
- A strict Interactive main-Agent finance request may skip synchronous compaction and compact-summary loading only when the current input is self-contained and contains a deterministic explicit security seed. That fast attempt derives active context plus at most four earlier user-only reference utterances from one durable Session snapshot. Reference recovery is ordered by durable user rows, stops strictly before the current row, and excludes automation, failed groups, slash commands, and compact summaries even when historical text equals the current input. Invoked-skill prompts restore separately from metadata under the current activation registry and do not consume this user-reference budget. Referential follow-ups keep full history. Context overflow still performs forced compaction and retries with the new summary; manual `/compact` and noninteractive compaction semantics remain unchanged.
- Interactive finance may commit early only on Web. The preferred path is the exact typed line `数据时间：北京时间 {Session answer time}；行情口径：本轮仅使用可核验资料，具体报价时间与数据缺口在正文逐项披露`: deterministic explicit-security requests may request ACK before the first model call, while other finance requests must wait for a budget-accepted batch containing a registered, supported/target-complete, structurally valid DataFetch call and only registered, structurally valid, known-read-only siblings. Unsupported/missing-target/malformed/rejected DataFetch and ordinary non-finance never activate deferred ACK. A canonical header from an eligible natural final remains a fallback when the typed commit was not accepted. Every visible delta becomes durable state only after exactly one downstream publication sink accepts it. Cancellation, a rejected/closed SSE receiver, ambiguous sinks, observer failure, or backpressure must not create phantom visible/persisted bytes. After a typed prefix commits, a whole batch executes only when every call remains registered, structurally valid, and known read-only; otherwise it fails before assistant tool-frame creation, observer notification, registry execution, or network access. Success publishes only the exact final suffix; failure appends the fixed research-failure disclosure and persists the exact visible partial bytes.
- Independent finance discovery should be requested in one model tool-call batch: per-entity DataFetch search and ticker-independent Web/news/filing/industry candidate discovery may be selected together. The current executor still awaits those calls serially and the intrinsic batch/call/route ceilings remain authoritative. Quote/profile/financials and other symbol-dependent calls must wait for route-bound search results and must never guess the ticker.
- Session summaries and compacted restore materials may stay session-scoped, but the displayed current time in prompt session context must be recalculated from the live current Beijing time on every turn; do not reuse stale session creation time as "当前时间"
- Session summaries must be stored in the explicit `summary` field instead of relying on a fake `system` summary message
- SQLite-backed session persistence must preserve the original `session_id` and source JSON semantics; do not silently normalize historical `Actor_*`, `Session_*`, or `User_*` identities during mirror writes or cutover reads
- Heartbeat tasks are first-class cron jobs: they must stay visible in the normal cron list, carry an explicit heartbeat marker, and poll on 30-minute slots without pretending to be a fixed daily time
- Cron jobs created from a channel session must preserve that session's origin `channel` and `channel_target`; scheduled or proactive delivery should return to the same target instead of falling back to a platform-level default destination. Missing targets must be rejected at creation time or recorded as `target_missing` execution failures at runtime, never silently replaced with a platform-level home/default channel.
- Skill runtime stays two-phase:
  - The current turn may expose only query-relevant compact skill summaries; catalog-wide discovery happens through `discover_skills`
  - Full `SKILL.md` bodies must be injected only when `skill_tool(...)` or a user slash skill actually invokes that skill
- Registered skills and enabled skills are separate truth domains:
  - Registration still comes from `skills/`, `data/custom_skills/`, or `.hone/skills/`
  - Runtime activation comes from the global `data/runtime/skill_registry.json` override layer
  - When a skill is disabled there, it must disappear from discover/list/search surfaces and be hard-rejected by slash invocation and `skill_tool(...)` across all channels and runners
- Skill disclosure and stage execution must stay aligned:
  - A skill that is shown in discovery, turn-0 skill summaries, related-skill hints, slash resolution, or `available_skills` must be usable in the current stage
  - If the current stage lacks one of the skill's required tools (for example `cron_job`) or the MCP stage tool allowlist excludes it, that skill must disappear from those visible surfaces and fail with an explicit stage-unavailable error instead of looking available first and failing later
- User slash skills and model `skill_tool(...)` calls must share the same prompt-expansion source of truth; do not let the Web/CLI path hand-maintain a divergent skill prompt template
- Invoked skills that materially change the current turn prompt must be recoverable from session metadata after compaction / resume; do not rely only on persisted tool results for skill restoration
- Session restore must respect current skill activation state. If a previously invoked skill is now disabled, its historical prompt may stay in stored metadata, but it must not be re-injected into the live runtime context on restore/compaction recovery
- Skill frontmatter fields such as `allowed-tools`, `model`, `effort`, `context`, `paths`, `hooks`, `arguments`, `script`, and `shell` should be parsed from `SKILL.md` as the source of truth. If runner infrastructure does not yet enforce one of these fields, document that gap instead of silently dropping or misrepresenting it
- Skill script execution must stay opt-in. Loading or disclosing a skill may expand its prompt, but running a declared `script` must only happen through an explicit execution path such as `skill_tool(..., execute_script=true)`
- Invoked skill context and per-turn user input must not be conflated. Only the stable invoked skill prompt may be persisted for restore/compaction; one-shot user supplements after a slash invocation belong only to the current turn
- Session compaction must preserve an explicit boundary model. Post-compact restore should derive context from the last compact boundary forward, with the compact summary stored as part of the message stream rather than treated only as an out-of-band prompt field
- Manual `/compact` must reuse the same boundary-based compaction path as automatic compression, must not consume daily conversation quota, and must not persist the slash command text as a normal user message
- Compact summaries and compact skill snapshots may stay in the stored message stream as restore materials, but primary chat history views should treat them as transcript-only rather than ordinary user-visible dialogue

## Change Constraints

- When directory responsibilities, entrypoints, or major data flows change, update `docs/repo-map.md`
- When workflows, testing contracts, or collaboration rules change, update `AGENTS.md` or this file
- When release or runtime strategy changes, update the matching docs, installer notes, and formula/tap metadata together
- Canonical config and runtime state must stay split:
  - `config.yaml` is the only long-lived user-writable config source
  - `data/runtime/effective-config.yaml` is generated runtime input for child processes
  - deleting `data/runtime/` must be a safe runtime reset that does not remove user config
