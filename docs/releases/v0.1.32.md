# v0.1.32

发布日期：2026-04-17

English readers: please scroll down for the English version below.

## 用户应该知道的

- 用户每日成功对话额度不再固定写死为 `12`，现在可以通过配置项 `agent.daily_conversation_limit` 调整。
- 如果把这个值设为 `0`，Hone 会关闭每日对话数限制，不再因为当天回复次数达到上限而拒绝新的用户消息。
- 这次改动同时保留了原有语义：管理员仍然不受该额度限制，定时任务也仍然不会消耗用户对话额度。
- 除了这项运行时改动，这个 release 还包含了自 `v0.1.31` 以来累计的一轮 bug 台账与排障文档刷新，集中补充了 heartbeat、Feishu scheduler / 直聊、desktop runner 配置和 company portrait 路径等问题的最新证据与状态。

## 谁应该尽快更新

- 通过 Discord、Feishu、Telegram 等消息渠道高频使用 Hone，已经碰到“已达到今日对话上限”提示的用户。
- 需要根据部署环境灵活调整额度，而不是接受固定 `12` 次上限的维护者。
- 希望在自托管环境里直接关闭每日用户对话额度限制的仓库维护者。
- 依赖 `docs/bugs/README.md` 和各缺陷文档持续追踪线上问题状态的维护者，因为这次 release 也把最近一轮 bug ledger 更新一起带进来了。

## 升级后会有什么变化

- `agent.daily_conversation_limit` 成为正式配置项，默认值仍保持 `12`，因此现有部署不会因为升级而被动改变配额策略。
- 运行时 quota 判定改为读取配置值；当值为 `0` 时会直接跳过用户对话额度拦截。
- `config.example.yaml` 和仓库上下文文档已经同步说明 `0 = unlimited` 的配置语义，并补了对应回归测试。
- `docs/bugs/README.md` 与多份缺陷文档会一起前进到新的基线，覆盖 heartbeat unknown-status、Feishu scheduler `HTTP 400` 投递失败、Feishu quota 拒绝文案、desktop runner 设置写入/覆盖、以及 `company_profiles` 路径问题等最近排障结论。

## 安装与更新

- `curl | bash` 用户：重新运行安装脚本即可切到新版本
- Homebrew 用户：`brew update && brew upgrade honeclaw`

## 已知注意事项

- 已经运行中的旧进程仍需要一次有意图的重启，才能加载新的 `agent.daily_conversation_limit` 配置并真正生效。
- 如果你在本地 repo 里把 `config.yaml` 设成 `0`，这个设置属于本地运行态配置，不会随着 Git 提交一起分发给其它环境。
- 除了 `agent.daily_conversation_limit` 之外，这个 release 里的其它“自 `v0.1.31` 以来改动”主要是缺陷文档与台账更新；它们会改进后续排障入口，但不会额外引入新的运行时行为开关。

## English

Release date: 2026-04-17

### What Users Should Know

- The daily successful-reply quota is no longer hard-coded to `12`; it is now controlled by the new `agent.daily_conversation_limit` config field.
- Setting that value to `0` disables the daily user-conversation cap entirely, so Hone no longer rejects new user messages just because the day has already hit the quota ceiling.
- Existing bypass behavior is preserved: admin actors are still exempt, and scheduled tasks still do not consume user conversation quota.
- In addition to that runtime change, this release also carries the accumulated bug-ledger and debugging-doc refreshes since `v0.1.31`, especially around heartbeat behavior, Feishu scheduler/direct-chat issues, desktop runner settings, and company portrait path diagnostics.

### Who Should Update Soon

- Users who rely on Hone heavily through Discord, Feishu, Telegram, or other chat channels and have already hit the daily quota message.
- Maintainers who need to tune the quota per deployment instead of accepting a fixed `12`-reply ceiling.
- Self-hosted operators who want to disable the daily user conversation limit entirely.
- Maintainers who rely on `docs/bugs/README.md` and the linked bug docs as the current operational triage entrypoint, because this release also bundles the latest ledger refresh.

### What Will Change After Upgrading

- `agent.daily_conversation_limit` is now a first-class config field, with the default kept at `12` so existing deployments do not change behavior unexpectedly after upgrade.
- Runtime quota checks now read that config value, and a value of `0` bypasses user conversation quota enforcement completely.
- `config.example.yaml` and repo context docs now document the `0 = unlimited` contract, and regression coverage was added for the unlimited path.
- `docs/bugs/README.md` and multiple bug writeups move forward to a new baseline as part of this release, covering recent triage outcomes for heartbeat unknown-status behavior, Feishu scheduler `HTTP 400` delivery failures, Feishu quota rejection UX, desktop runner setting races/overwrites, and the `company_profiles` path issue.

### Install and Update

- `curl | bash` users: rerun the install script to move to this version
- Homebrew users: `brew update && brew upgrade honeclaw`

### Known Notes

- Already-running old processes still need an intentional restart before the new `agent.daily_conversation_limit` setting is actually loaded.
- If you set `config.yaml` to `0` in a local checkout, that remains a local runtime setting and is not distributed through Git commits.
- Beyond `agent.daily_conversation_limit`, the other changes included since `v0.1.31` are primarily bug-ledger and debugging-document updates; they improve the operational trail but do not add additional runtime feature toggles.
