# v0.1.19

发布日期：2026-04-13

## 用户应该知道的

- 公司画像现在会更主动地沉淀长期研究结论，不再只等用户明确要求“帮我记下来”
- 画像更新规则收紧为“不要记流水账”：如果长期判断、偏好、共识逻辑或估值结论已经变化，系统应直接改 `profile.md` 正文，而不是只在事件里补备注
- Telegram 长回复的 HTML 渲染错误已修复，同类分段保护也扩展到了 Discord 和 Feishu，避免代码块、引用或 HTML tag 在长回复分段时被切坏

## 谁应该尽快更新

- 已经在 Hone 里长期做公司研究、希望系统稳定记住自己判断框架和估值口味的用户
- 经常通过 Telegram、Discord、Feishu 接收长篇分析回复的用户
- 维护多渠道消息发送、公司画像长期记忆或 prompt/skill 规则的开发者

## 升级后会有什么变化

- 只要用户在系统研究某家公司，本轮产出了值得长期复用的内容，系统就会优先沉淀到公司画像，重点包括用户自己的看法、偏好或约束、双方已经达成一致的判断逻辑，以及估值判断、估值区间或估值锚点
- 公司画像的主文档会更明确地保持“当前最佳判断”语义；事件文件继续只承担 thesis change log，不再鼓励把研究过程写成日记式流水账
- 共享分段层新增 HTML / Markdown 两套 format-aware segmenter：Telegram 在跨段时会自动补全并重开 HTML tag，Discord / Feishu 在 Markdown 代码块跨段时会自动补全并重开 code fence
- 这意味着长回复即使被拆成多段发送，也更不容易触发 Telegram `can't parse entities`、Discord 代码块断裂或 Feishu Markdown 结构失真

## 安装与更新

- `curl | bash` 用户：重新运行安装脚本即可切到新版本
- Homebrew 用户：`brew update && brew upgrade honeclaw`

## 已知注意事项

- 这次强化的是“长期研究记忆如何写回”与“长回复如何安全分段发送”，不包含新的公司画像 UI 或新的存储层
- Markdown 感知分段当前优先保护代码 fence；如果后续发现表格或其它复杂块结构在特定渠道仍有边界问题，还会继续补充规则
