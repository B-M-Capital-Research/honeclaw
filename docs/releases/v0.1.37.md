# v0.1.37

发布日期：2026-04-19

English readers: please scroll down for the English version below.

## 用户应该知道的

- ACP runner 现在会把 JSON-RPC 原始收发 payload 单独写进 `data/runtime/logs/acp-events.log`，不再需要从普通运行日志里手工拼接调试线索。
- 这份新日志记录的是 raw trace，而不是二次加工后的 `progress` / `tool` 状态文本；`session/prompt`、`session/update`、`session/request_permission` 等 ACP 消息都会按单行 JSON 追加落盘。
- 每条 ACP 记录都会带上当前 `identity`、`session_id`、actor channel / user / scope，排查某个用户、某个渠道或某条会话时可以直接 `grep`。
- `codex_acp`、`gemini_acp` 和 `opencode_acp` 三条 ACP 链路现在都使用同一套 raw trace 落盘约定；其中 `opencode_acp` 原来那段自定义 prompt 读循环也已经补齐，不会再漏掉它自己的原始事件。

## 谁应该尽快更新

- 正在排查 `codex_acp`、`gemini_acp` 或 `opencode_acp` 偶发异常、超时、空回复、权限请求或工具调用问题的维护者。
- 需要按 `identity`、`session_id` 或具体 JSON-RPC method 回放某次 ACP 会话的开发者。
- 希望把“用户到底问了什么、runner 收到了什么、返回了什么”从普通业务日志里拆出来单独看的团队。

## 升级后会有什么变化

- ACP 原始消息会额外写入独立文件 `data/runtime/logs/acp-events.log`；普通 runtime 日志仍然保留，但不再是唯一排查入口。
- 新日志按单行 JSON 追加，关键字段包括：`runner`、`direction`、`event_kind`、`method`、`session_id`、`identity`、`actor_channel`、`actor_user_id`、`actor_channel_scope`、`payload`。
- 如果某条 ACP 输出不是合法 JSON，也会额外写入 `parse_error` 记录，保留原始行内容，方便定位上游协议异常。
- 这次改动不改变 ACP runner 的业务行为、权限决策或用户可见回答，主要提升的是可观测性和追踪精度。

## 安装与更新

- `curl | bash` 用户：重新运行安装脚本即可切到新版本
- Homebrew 用户：`brew update && brew upgrade honeclaw`

## 已知注意事项

- `acp-events.log` 记录的是原始 payload，可能会比普通日志更大；如果你长期高频使用 ACP runner，需要把它纳入常规日志清理策略。
- 这份日志默认会保留 prompt / tool raw payload，因此更适合作为维护与排障资产，而不是直接暴露给普通终端用户。
- 本次发布主要收口 ACP raw trace 可观测性；它不会自动替你做日志脱敏、轮转或长期归档策略设计。

## English

Release date: 2026-04-19

### What Users Should Know

- ACP runners now write raw JSON-RPC request/response traces into a dedicated file at `data/runtime/logs/acp-events.log`, so ACP debugging no longer depends on reconstructing flows from general runtime logs.
- This new file stores raw traces rather than rendered `progress` / `tool` status messages. ACP traffic such as `session/prompt`, `session/update`, and `session/request_permission` is appended as newline-delimited JSON.
- Every ACP record now includes the current `identity`, `session_id`, and actor channel / user / scope, which makes it practical to `grep` one user, one channel, or one specific conversation directly.
- The same raw-trace contract now covers `codex_acp`, `gemini_acp`, and `opencode_acp`; the custom prompt-read loop in `opencode_acp` is also covered now, so its own raw events are no longer missing.

### Who Should Update Soon

- Maintainers debugging intermittent `codex_acp`, `gemini_acp`, or `opencode_acp` issues such as timeouts, empty replies, permission prompts, or tool-call failures.
- Developers who need to replay one ACP session by `identity`, `session_id`, or JSON-RPC method.
- Teams that want a dedicated ACP trace surface instead of mixing “what the user asked, what the runner received, and what came back” into general business logs.

### What Will Change After Upgrading

- Raw ACP messages are now also written to `data/runtime/logs/acp-events.log`; the general runtime logs still exist, but they are no longer the only debugging entrypoint.
- The new file is newline-delimited JSON. Key fields include `runner`, `direction`, `event_kind`, `method`, `session_id`, `identity`, `actor_channel`, `actor_user_id`, `actor_channel_scope`, and `payload`.
- If an ACP output line is not valid JSON, a dedicated `parse_error` record is also written and preserves the raw line for upstream protocol debugging.
- This release does not change ACP runner business behavior, permission outcomes, or user-facing answers. The main improvement is observability and trace precision.

### Install and Update

- `curl | bash` users: rerun the install script to move to this version
- Homebrew users: `brew update && brew upgrade honeclaw`

### Known Notes

- Because `acp-events.log` stores raw payloads, it can grow faster than general runtime logs. If you run ACP runners heavily, include it in your routine log-cleanup strategy.
- This log can include raw prompt and tool payloads by design, so it should be treated as an operator/debugging asset rather than something exposed directly to ordinary end users.
- This release tightens ACP raw-trace observability only; it does not automatically design log redaction, rotation, or long-term archival policies for you.
