# v0.1.33

发布日期：2026-04-17

English readers: please scroll down for the English version below.

## 用户应该知道的

- Discord bot token 相关的 CLI 交互现在更稳了：`hone-cli onboard` 和 `hone-cli configure --section channels` 会先检查 token 的三段结构、字符集和异常长度，再决定是接受、警告还是拒绝。
- onboarding 里此前那个“输入了非空 token，却仍然误弹空值恢复菜单”的问题已经修掉；现在只有真的空值场景才会进入“必填项为空”的恢复流程。
- 对于非法但非空的 Discord token，CLI 会显示专门的“格式不合法，下一步？”恢复提示，不再错误复用“为空”语义。
- `hone-cli start` 现在会在任一子进程异常退出时 fail-fast，并在 Discord 子进程疑似 token 认证失败时给出更直接的重配提示；`hone-discord` 自身也补了 invalid-auth 检测与引导信息。
- 安装脚本还顺手修掉了一处 CI 暴露出来的归档检测问题，避免 `tar: stdout: write error` 这类管道 `SIGPIPE` 异常干扰安装回归。

## 谁应该尽快更新

- 通过 `hone-cli onboard` 或 `hone-cli configure --section channels` 配置 Discord 渠道，并且已经遇到过 token 输入体验混乱、提示语义不准或误判为空的用户。
- 依赖 `hone-cli start` 在本地快速发现 Discord 配置错误，并希望收到更明确 fail-fast 诊断的维护者。
- 通过安装脚本或 CI 回归脚本验证 `hone-cli` 安装路径与 bundle 解压流程的维护者。

## 升级后会有什么变化

- Discord token 的 optional / required 两条 CLI 路径都会在保存前做 `trim`，避免把首尾空白写进配置文件。
- token 校验不再只看是否为空，而是会检查三段结构、base64url 字符集，以及异常偏短 / 偏长的长度信号，并在 warning 场景下让用户显式确认。
- `hone-cli start` 对子进程退出的监控从“只盯 backend”改成“任一子进程退出即报错”，因此 Discord sidecar 异常时会更早暴露。
- CI-safe 安装回归现在不会再因为 `tar -tzf | head` 这一类管道提前截断而在部分 runner 上误报失败。

## 安装与更新

- `curl | bash` 用户：重新运行安装脚本即可切到新版本
- Homebrew 用户：`brew update && brew upgrade honeclaw`

## 已知注意事项

- 这次 release 主要修的是 CLI 交互、启动期诊断和安装回归；已经运行中的旧进程仍需要一次有意图的重启，才能真正带上新的 fail-fast 和提示行为。
- token 结构校验是“前置防呆”，不是 Discord 官方鉴权的完全替代；格式看起来合法的 token 仍然可能因为过期、撤销或权限问题在运行时失败。
- 自 `v0.1.32` 以来，`main` 上还有几条 bug ledger 文档刷新提交一起进入 compare 范围，但它们不是本次用户向发布的核心运行时变化。

## English

Release date: 2026-04-17

### What Users Should Know

- The Discord bot token CLI flow is now much more robust. Both `hone-cli onboard` and `hone-cli configure --section channels` validate the token’s three-part shape, character set, and suspicious length before deciding whether to accept, warn, or reject it.
- The onboarding bug where a non-empty token could still trigger the empty-field recovery menu is fixed. Only truly empty input now goes through the “required field is empty” recovery path.
- Invalid but non-empty Discord tokens now get a dedicated “invalid format, what next?” recovery prompt instead of incorrectly reusing the “empty value” wording.
- `hone-cli start` now fails fast when any child process exits unexpectedly, and both the CLI and `hone-discord` surface clearer remediation hints when the Discord sidecar appears to be failing on token authentication.
- The installer also includes a small but important regression fix uncovered by CI, avoiding `tar: stdout: write error` style failures caused by a pipe `SIGPIPE` during archive inspection.

### Who Should Update Soon

- Users who configure Discord through `hone-cli onboard` or `hone-cli configure --section channels` and have already seen confusing token prompts, inaccurate wording, or false empty-input handling.
- Maintainers who rely on `hone-cli start` to surface Discord configuration problems quickly and want clearer fail-fast diagnostics.
- Maintainers who validate the installer or bundle extraction path through local scripts or CI-safe regressions.

### What Will Change After Upgrading

- Both the optional and required Discord token flows now trim whitespace before saving, so leading and trailing spaces no longer get persisted into config.
- Token validation no longer only checks for emptiness; it also checks the three-part structure, base64url character set, and abnormal short/long length signals, with explicit confirmation for warning-level cases.
- `hone-cli start` no longer watches only the backend; it now reports an error when any managed child exits, which makes Discord sidecar failures visible much earlier.
- The CI-safe installer regression no longer depends on a fragile `tar -tzf | head` pipeline that could fail on some runners due to premature pipe termination.

### Install and Update

- `curl | bash` users: rerun the install script to move to this version
- Homebrew users: `brew update && brew upgrade honeclaw`

### Known Notes

- This release mainly fixes CLI interaction, startup diagnostics, and installer regression behavior. Already-running old processes still need an intentional restart before the new fail-fast and hinting behavior actually applies.
- The token structure validation is a guardrail, not a full substitute for Discord’s own runtime authentication. A token that looks structurally valid may still fail later if it is expired, revoked, or missing permissions.
- A few bug-ledger documentation refresh commits on `main` also fall into the compare range since `v0.1.32`, but they are not the primary user-facing runtime changes in this release.
