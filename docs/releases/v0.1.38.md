# v0.1.38

发布日期：2026-04-19

English readers: please scroll down for the English version below.

## 用户应该知道的

- Hone 的 Web 管理端和用户端端口现在固定下来了：管理端默认使用 `8077`，面向终端用户的公网聊天端默认使用 `8088`。桌面端打包后的内置服务也会按这组固定端口启动，不再随机漂移。
- 邀请码体系现在和手机号强绑定。管理端生成邀请码时必须填写手机号，邀请码会和该手机号成对绑定；用户从用户端登录时，也必须同时填写邀请码和手机号才能进入聊天页。
- 这次还补了一轮公网暴露面的安全加固，重点收紧了用户端 `8088` 上与邀请码相关的访问路径，降低被批量探测、暴力刷邀请码和越权访问管理能力的风险。

## 谁应该尽快更新

- 已经把 Hone 桌面端或自托管 Web 入口给真实用户使用，并且希望管理端与用户端端口固定、可稳定做反向代理和放行规则的团队。
- 计划把 `8088` 暴露到公网、需要邀请码登录链路更稳妥一些的维护者。
- 需要把邀请码发放流程和手机号实名或运营侧信息绑定起来的团队。

## 升级后会有什么变化

- 管理端端口固定为 `8077`，用户端端口固定为 `8088`；桌面端构建出的 APP 启动后，也会按这组端口拉起对应服务，便于代理、监控和运维脚本固化配置。
- 管理端在创建邀请码时会新增手机号必填校验；服务端会把邀请码和手机号一并保存，避免只靠邀请码单因子进入用户端。
- 用户端登录表单会要求同时填写邀请码和手机号；只有匹配成功后才会发放会话并进入聊天页面。
- 用户端公开路由上与邀请码相关的校验、错误返回和暴力尝试面做了进一步收口，默认更适合放在公网入口后面运行。

## 安装与更新

- `curl | bash` 用户：重新运行安装脚本即可切到新版本
- Homebrew 用户：`brew update && brew upgrade honeclaw`

## 已知注意事项

- 端口固定后，如果本机或服务器上已有其它进程占用了 `8077` 或 `8088`，启动前需要先释放端口或调整你的部署环境。
- 这次发布把邀请码认证从“单邀请码”升级成了“邀请码 + 手机号”双因子绑定；如果你有旧的发码流程、客服文案或自动化脚本，需要同步更新。
- 用户端公网面虽然做了额外收紧，但它依然是一个暴露在网络上的服务。生产环境下仍建议配合反向代理、TLS、基础限流和日志审计一起使用。

## English

Release date: 2026-04-19

### What Users Should Know

- Hone now pins its web ports: the admin surface uses `8077` and the end-user public chat surface uses `8088`. The bundled desktop app also starts its embedded services on these fixed ports instead of choosing random ones.
- Invite codes are now bound to phone numbers. Admins must enter a phone number when issuing an invite, and end users must provide both the invite code and the matching phone number before entering the chat surface.
- This release also tightens the public-facing web surface, especially the invite-related paths on `8088`, to reduce the risk of bulk probing, invite-code brute force, and accidental access beyond the intended user surface.

### Who Should Update Soon

- Teams already using Hone through the desktop app or a self-hosted web entrypoint who want stable admin and public ports for reverse proxies, firewalls, and monitoring.
- Maintainers planning to expose `8088` to the public internet and wanting a safer invite-auth flow.
- Teams that need invite issuance to stay paired with a user phone number or operator-side identity record.

### What Will Change After Upgrading

- The admin surface is fixed to `8077` and the public user surface is fixed to `8088`. The packaged desktop app now starts the corresponding services on those same ports, which makes reverse-proxy rules and operational scripts more predictable.
- The admin invite flow now requires a phone number and persists that binding alongside the invite code, so the public login flow no longer depends on an invite code alone.
- The public login form now requires both invite code and phone number; only a matching pair can create a session and enter the chat page.
- Validation and error handling around invite-auth on the public surface were tightened so the default deployment is better suited for internet exposure behind a proxy.

### Install and Update

- `curl | bash` users: rerun the install script to move to this version
- Homebrew users: `brew update && brew upgrade honeclaw`

### Known Notes

- Because ports are now fixed, make sure nothing else is already listening on `8077` or `8088` before startup.
- This release upgrades invite auth from “invite code only” to a paired “invite code + phone number” flow. If you have existing onboarding docs, support playbooks, or automation around invite issuance, update them accordingly.
- The public web surface is more locked down now, but it is still a network-facing service. In production, you should still put it behind TLS, a reverse proxy, basic rate limiting, and normal access logging.
