# v0.2.0

发布日期:2026-04-21

English readers: please scroll down for the English version below.

> v0.1.46 把事件引擎的 cron-aligned 抓取 + read-time 倒计时做完了,但有个没公开承认的 gap:`OutboundSink` 在运行时只有 `LogSink` 一家。所有"主动事件推送"其实只落 tracing 日志,从不真投递。v0.2.0 把这一层补齐:事件引擎现在会真的把 digest 和 High 级事件发到 Telegram / Discord / Feishu / iMessage。

## 用户应该知道的

### 主动事件推送 **真的会推**了

如果你 v0.1.43 ~ v0.1.46 期间启用了 `event_engine.enabled: true`、设好了 Telegram bot token、配好了 `notif_prefs/`,本来应该每天 08:30 / 09:00 收到 pre/post-market digest 以及 High 级的 price_alert / news_critical。实际上之前你只会在后端日志里看到 `digest truncated ...` 这种行,Telegram 一条消息都没有。

v0.2.0 起,`hone-web-api` 启动 event engine 时会按 `HoneConfig` 的渠道 enabled + 凭据就绪情况逐个 attach 真实 sink:

- **TelegramSink**:直接打 Bot API `sendMessage`,私聊用 `user_id`,群聊剥 `chat_` 前缀
- **DiscordSink**:`POST /users/@me/channels` 解 DM channel(带进程内缓存),再 `POST /channels/{id}/messages`
- **FeishuSink**:缓存 `tenant_access_token`(提前 5 分钟过期),打 `/open-apis/im/v1/messages` 发 text
- **IMessageSink**:走 `osascript` + Messages.app,macOS 专用

未注册的渠道或 sink 抛错都会 fall back 到 `LogSink`,不会让 engine 挂掉。`event_engine.dryrun: true` 仍然一律走 LogSink(想验证引擎行为却不真骚扰人时的 escape hatch)。

### e2e 已在 pre_market 触发下验证

验证路径:把 `digest.pre_market` 临时调到当前时刻 +3 分钟 → 重启 hone-console-page → 到点 digest 触发 → TelegramSink 真实投递到用户。日志记录 `digest truncated to avoid info flooding actor=telegram::::8039067465 dropped=2 kept=20`。

### Config 迁移

**无需改动**。`channel.enabled` + `channel.bot_token` / `app_id+secret` 已经是老字段;v0.2.0 只是把 engine 侧的 sink 装配串起来。dryrun 默认 `false`,如果你之前明确设了 `true`(譬如为了不被轮询结果轰炸),行为与之前完全一致。

## 升级步骤

1. `git pull && cargo build -p hone-console-page`
2. 检查 `config.yaml`:`event_engine.enabled: true` 且 `event_engine.dryrun: false`(后者是默认,如果你之前明确调过可以确认一下)
3. 检查至少一个渠道 enabled + 凭据就位(不然会回退到 LogSink,相当于老行为)
4. 重启 hone-console-page —— 启动日志会有 `event engine sink: MultiChannelSink 已装配 channels=["telegram", ...]`

## 已知的局限

- **渲染格式统一用 Plain**:所有 sink 的 `format()` 返回 `RenderFormat::Plain`,避免 renderer 需要知道 actor → format 映射;副作用是 Telegram 不会有 HTML 加粗等富格式
- **不做长消息分段**:Telegram 4096 字符上限 / Feishu 5000 字符上限,超过会直接 API 报错;目前靠 `digest.max_items_per_batch` 限制条数间接控制
- **iMessage sink 仅在 macOS 有效**:非 macOS 下直接 bail 并走 fallback

# v0.2.0 (English)

Release date: 2026-04-21

> v0.1.46 finished the event engine's cron-aligned polling and read-time countdown work, but there was an unacknowledged gap: at runtime the only `OutboundSink` was `LogSink`. Every "proactive event push" only wrote to tracing logs — nothing was ever delivered. v0.2.0 closes this gap: the event engine now actually pushes digests and High-severity events to Telegram / Discord / Feishu / iMessage.

## What users should know

### Proactive pushes **actually push** now

If you enabled `event_engine.enabled: true` during v0.1.43–v0.1.46, configured a Telegram bot token, and set up `notif_prefs/`, you should have been receiving 08:30 / 09:00 pre/post-market digests and High-severity price/news alerts. In reality you only saw `digest truncated ...` lines in backend logs; Telegram got zero messages.

Starting v0.2.0, `hone-web-api` attaches real sinks based on each channel's `enabled` flag and credential readiness in `HoneConfig`:

- **TelegramSink**: hits the Bot API `sendMessage` directly; DM uses `user_id`, groups strip the `chat_` prefix
- **DiscordSink**: `POST /users/@me/channels` to resolve DM channel (process-cached), then `POST /channels/{id}/messages`
- **FeishuSink**: caches `tenant_access_token` (expires 5 min early), hits `/open-apis/im/v1/messages` with a text body
- **IMessageSink**: uses `osascript` + Messages.app, macOS only

Unregistered channels or sink errors fall back to `LogSink` so the engine doesn't crash. `event_engine.dryrun: true` still routes everything to `LogSink` (escape hatch when you want to exercise the engine without spamming users).

### End-to-end verified via pre_market trigger

Verified by temporarily bumping `digest.pre_market` to "current time + 3 minutes", restarting `hone-console-page`, and confirming TelegramSink delivery when the digest fired. Log line: `digest truncated to avoid info flooding actor=telegram::::8039067465 dropped=2 kept=20`.

### Config migration

**No changes required**. `channel.enabled` + `channel.bot_token` / `app_id+secret` are existing fields; v0.2.0 just wires the engine-side sink factory. `dryrun` still defaults to `false`; if you explicitly set `true` before, behavior is unchanged.

## Upgrade steps

1. `git pull && cargo build -p hone-console-page`
2. Verify `config.yaml`: `event_engine.enabled: true` and `event_engine.dryrun: false` (the latter is the default; worth double-checking if you've ever flipped it)
3. Verify at least one channel is enabled with credentials (otherwise you fall back to LogSink, which matches the old behavior)
4. Restart `hone-console-page` — startup log should include `event engine sink: MultiChannelSink 已装配 channels=["telegram", ...]`

## Known limitations

- **Uniform Plain rendering**: all sinks return `RenderFormat::Plain` from `format()` to avoid requiring the renderer to know actor → format mappings; side effect is no HTML bold / rich formatting on Telegram
- **No message chunking**: Telegram's 4096-char and Feishu's 5000-char limits will bubble up as API errors; mitigated indirectly via `digest.max_items_per_batch`
- **IMessageSink is macOS-only**: bails on other OSes and falls back automatically
