# v0.1.43

发布日期:2026-04-21

English readers: please scroll down for the English version below.

> 本次是 Hone 的第一个"主动事件引擎"版本:Hone 第一次能在你不主动提问的前提下,把真正重要的市场变化按你的节奏推到你身边,而不是等你打开聊天框。

## 用户应该知道的

### 全新模块:主动事件引擎 `hone-event-engine`

- 7 个纯 Rust、无 LLM 的 poller 默认装配:News、PressRelease、Price、Earnings(日历)、CorpAction(含 SEC 8-K)、Macro、AnalystGrade、EarningsSurprise,支持独立轮询间隔与独立关停
- `EventStore` 落盘 SQLite(`./data/events.db`)+ JSONL 冷备,严格按 `event.id` 去重,天然幂等,不会因 poller 重启造成重复推送
- `Router` 做按持仓 / 按订阅的细粒度分发;重大级别(High)实时推送,中低级别进入 pre-market / post-market digest 聚合,避免小波动把你淹没
- `Digest` 调度按你配置的时区触发(默认 `Asia/Shanghai`,`08:30` 早盘,`09:00` 收盘复盘);可配 `max_items_per_batch` 防止尾部爆量
- 内置 cooldown / daily cap / `llm_polish_for` 文案打磨模板,High 级消息每日封顶、同标的 60 秒内不重复告警、可选 LLM 润色
- `DailyReport` 每日对事件流做汇总日志(仅落盘,不当通知推给用户,避免自我推销)
- 所有 poller / renderer / sink 都有 trait 抽象,便于接私有数据源或自建出站

### 三层推送过滤:部署方 × 终端用户 × 管理员

- **部署方**:`config.yaml` 里的 `event_engine.sources.fmp_*` 真正生效 —— 关掉 `fmp_news` 就直接 skip tick,省 FMP 请求、省数据库写入
- **部署方**:新增 `event_engine.disabled_kinds: []` 全局 kind 黑名单,router 在用户过滤之前先过一遍;入库仍然发生(便于日报统计),只是不分发
- **终端用户**:新 skill `notification_preferences` + tool `notification_prefs`,直接在 iMessage / Telegram / Feishu / Discord 说"先别推新闻了""只看重要的""只推我持仓相关的",自动映射成 enable / disable / min_severity / portfolio_only / allow_kinds / blocked_kinds
- **管理员**:设置页新增 `<NotificationPreferencesCard />`,roster 列表勾选即保存、详情区有完整 `min_severity` / `portfolio_only` / `allow_kinds` / `blocked_kinds` 表单
- 三条入口写同一份 `<data>/notif_prefs/{channel}__{scope}__{user_id}.json`;router 每次 dispatch 无缓存重读,任何一条路改完下一条事件立即生效
- 合法 kind tag 由 `ALL_KIND_TAGS` 常量维护,API / tool 双端校验,非法值直接 400 并回显合法清单

### 新的 HTTP API / 管理台

- `GET /api/notification-prefs?channel=&user_id=&channel_scope=` —— 读取某 actor 的偏好,文件缺失返默认不报 404
- `PUT /api/notification-prefs` —— 管理员代任意 actor 改偏好(带 kind tag 白名单校验)
- `GET /api/meta` 响应追加 `notification_kind_tags`,前端不再需要硬编码 16 个 tag
- 设置页 roster 并发拉取所有 actor 的 prefs,勾选 → 即保存;详情区保留手动 "保存" 按钮避免误改

### E2E 验收示例

- 新增 `cargo run --example push_smoke -p hone-event-engine`:扫描 `./data/notif_prefs`,对每个 actor 跑一次 `should_deliver`,对通过过滤的 telegram DM actor 真调一次 Bot API `sendMessage`,一条命令走通 `FilePrefsStorage + should_deliver + Telegram` 全链路
- 输出同时打印每个 actor 的 `enabled / allow / block` 状态与判决结果(FILTERED / SENT / PASS / FAIL),作为上线验收报告

### 新 Skill

- `skills/notification_preferences/SKILL.md`:把中文自然语言("静音""只看高优""不要分析师评级"…)映射到 tool action,带合法 kind tag 清单、常见意图表、工作流三件套

## 升级后会有什么变化

- `event_engine.enabled` 默认仍为 `false`,升级本身不会立刻开始推送;需要显式 enable 并确认你的 FMP 配额
- `event_engine.sources.fmp_*` 从"声明但无效"变成"真正门控",如果你之前写过 `false` 的字段,本次起会真的停掉对应 poller
- 新增字段 `event_engine.disabled_kinds: []` 已写进 `config.example.yaml`,空数组等价于旧行为,不升级配置也兼容
- 个人 prefs 落盘复用既有目录结构 `<data>/notif_prefs/{channel}__{scope}__{user_id}.json`,现存 JSON 文件零迁移
- 设置页 roster 首次加载会一次性并发拉取所有 actor 的 prefs,体感略慢;后续 toggle 全部本地即时
- `<data>/events.db`、`<data>/events.jsonl`、`<data>/digest_buffer/`、`<data>/daily_reports/`、`<data>/notif_prefs/` 是新增的持久化目录,请确保你的磁盘 / 备份策略覆盖到它们

## English

Release date: 2026-04-21

> This is Hone's first "proactive event engine" release: Hone can finally push the market changes that actually matter — on your cadence, into your surface — without you having to open a chat window.

### What Users Should Know

#### New module: the proactive event engine `hone-event-engine`

- Seven pure-Rust, no-LLM pollers ship by default: News, PressRelease, Price, Earnings (calendar), CorpAction (incl. SEC 8-K), Macro, AnalystGrade, EarningsSurprise — each with an independent interval and kill-switch
- `EventStore` persists to SQLite (`./data/events.db`) + JSONL cold archive and dedupes strictly by `event.id`, so restarting a poller never re-sends an event
- `Router` does fine-grained per-holding / per-subscription dispatch; High-severity events push immediately, Medium/Low roll into pre-market / post-market digests so small moves can't drown you
- `Digest` scheduler fires at the timezone you configure (defaults to `Asia/Shanghai`, `08:30` pre-market and `09:00` post-market wrap); a `max_items_per_batch` cap keeps a long tail from exploding
- Built-in cooldown / daily cap / optional `llm_polish_for` — High events are capped per day, the same ticker can't re-alert within 60 s, and template output is optionally polished by LLM
- `DailyReport` aggregates the event stream into a daily log file (disk only — never pushed back to users, no self-marketing)
- Every poller / renderer / sink is behind a trait, ready for private data sources or custom outbound integrations

#### Three-tier push filtering: operator × end user × admin

- **Operator**: `event_engine.sources.fmp_*` in `config.yaml` now really gates pollers — disabling `fmp_news` truly skips ticks, saving FMP quota and DB writes
- **Operator**: new `event_engine.disabled_kinds: []` is a global kind blacklist; router filters against it before per-user prefs. Events still land in the DB (for reporting) but never dispatch
- **End user**: new skill `notification_preferences` + tool `notification_prefs` — users can say "mute news", "high severity only", "only my holdings" directly in iMessage / Telegram / Feishu / Discord; the skill maps natural language to `enable` / `disable` / `min_severity` / `portfolio_only` / `allow_kinds` / `blocked_kinds`
- **Admin**: settings page gains `<NotificationPreferencesCard />` — a roster list with auto-save-on-toggle plus a detail form for `min_severity` / `portfolio_only` / `allow_kinds` / `blocked_kinds`
- All three entry points write to the same `<data>/notif_prefs/{channel}__{scope}__{user_id}.json`; router re-reads on every dispatch with no cache, so any change takes effect on the next event
- Legal kind tags are centralized in the `ALL_KIND_TAGS` constant and validated on both the API and the tool — unknown tags get a 400 with the legal list echoed back

#### New HTTP API / admin surface

- `GET /api/notification-prefs?channel=&user_id=&channel_scope=` reads an actor's prefs; a missing file returns defaults, not 404
- `PUT /api/notification-prefs` lets admins edit any actor's prefs (validated against the kind tag whitelist)
- `GET /api/meta` response now includes `notification_kind_tags`, so the frontend no longer needs to hard-code the 16 tags
- The settings roster fetches every actor's prefs concurrently; the checkbox auto-saves; the detail editor keeps a manual Save button to prevent accidental edits

#### End-to-end smoke test

- New `cargo run --example push_smoke -p hone-event-engine`: scans `./data/notif_prefs`, runs `should_deliver` per actor, and for Telegram DM actors that pass the filter actually calls the Bot API `sendMessage` — one command exercises `FilePrefsStorage + should_deliver + Telegram` end to end
- The output prints each actor's `enabled / allow / block` state and verdict (FILTERED / SENT / PASS / FAIL), doubling as a rollout acceptance report

#### New skill

- `skills/notification_preferences/SKILL.md` maps natural-language intents ("mute", "only high severity", "drop analyst grades"…) to tool actions; ships with the legal kind tag list, a common-intent table, and a three-step workflow

### Who Should Update Soon

- Any operator wanting to upgrade Hone from "question-answering assistant" to "proactive research assistant"
- Ops / support teams worried that default pushes are too noisy, or needing different cadences per user
- Teams already wired into iMessage / Telegram / Feishu / Discord who want users to tweak subscriptions in natural language
- Users relying on portfolio-based alerts who want to silence specific sources (e.g. press-release noise)

### What Will Change After Upgrading

- `event_engine.enabled` still defaults to `false`; this release does not start pushing on its own — you need to explicitly enable it and sanity-check your FMP quota
- `event_engine.sources.fmp_*` moves from "declared but ignored" to a real gate; any `false` you had will now actually stop the corresponding poller
- New `event_engine.disabled_kinds: []` field appears in `config.example.yaml`; empty list matches old behavior, no config change needed
- Per-user prefs reuse the existing layout `<data>/notif_prefs/{channel}__{scope}__{user_id}.json` — no migration for existing JSON files
- The settings roster fetches every actor's prefs concurrently on first load; initial render is slightly slower but subsequent toggles are instant
- `<data>/events.db`, `<data>/events.jsonl`, `<data>/digest_buffer/`, `<data>/daily_reports/`, and `<data>/notif_prefs/` are new persistent directories — make sure your disk / backup policy covers them

### Install and Update

- `curl | bash` users: rerun the install script to move to this version
- Homebrew users: `brew update && brew upgrade honeclaw`
- Before enabling the event engine for the first time: fill in `fmp.api_key`, flip `event_engine.enabled: true`, and tune `poll_intervals` / `digest` / `thresholds` as needed

### Known Notes

- After the tag is pushed, the remote release workflow still needs to finish binary packaging, GitHub Release publication, and Homebrew formula publication
- The Price / Macro / Analyst / EarningsSurprise pollers are not yet wired to per-source `sources.*` toggles — to silence them use `disabled_kinds` or disable the whole event engine
- `allow_kinds` / `blocked_kinds` must use one of the 16 legal kind tags; invalid values are rejected by the API and tool, which echo the full legal list back
- `LogSink` is still the MVP default outbound; real iMessage / Telegram / Feishu / Discord sinks need to be wired per operator channel
- `llm_polish_for`, if enabled, requires `llm.openrouter` to be configured; otherwise it falls back to template-only output
