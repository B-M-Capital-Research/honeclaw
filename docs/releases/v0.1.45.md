# v0.1.45

发布日期:2026-04-21

English readers: please scroll down for the English version below.

> 把财报从"一次性日历条目"升级成一条"有节奏的生命周期":关注起点可配、T-3/T-2/T-1 每日倒计时、发布当日直接附 press release 链接。用户对 `earnings_upcoming` 的静音偏好对整条生命周期都生效。

## 用户应该知道的

### 财报生命周期:T-14 预告 → T-3/T-2/T-1 倒计时 → 发布日链接

此前 `EarningsPoller` 只会在看到一条新日历条目时发一次 `EarningsUpcoming`(id = `earnings:{SYMBOL}:{date}`)。用户实际体感:一周前收到过一次提醒,到了财报前一天晚上早就忘了 —— 而盘前正是最需要知道"今天这只票要出财报"的时候。

v0.1.45 把它改造成一条完整的 lifecycle:

1. **初次预告**(T-14 ~ T-3 进入日历时):一次性 Medium 事件,`earnings:{SYMBOL}:{date}`,和以前一样
2. **T-3 / T-2 倒计时**:每日 Medium 事件,id = `earnings:{SYMBOL}:{date}:countdown:{N}`,title 形如 `AAPL earnings in 3 days (2026-05-02)`
3. **T-1 倒计时**:升级为 High,绕过 digest 立即推送,title 形如 `AAPL earnings tomorrow (2026-05-02)` —— 确保前一晚/盘前必能看到
4. **发布当日**:`EarningsSurprisePoller` 命中后,事件 `url` 不再为空,指向 Yahoo `https://finance.yahoo.com/quote/{TICKER}/press-releases/` 作为通用新闻稿入口(免费、无登录、永远指向最新 PR)

倒计时事件 id 每天不同,不会被 EventStore 的稳定 id dedup 折叠;初次预告的 id 保持原样,不会每次 poll 都重发。

### 关注起点可配:`event_engine.earnings.window_days`

此前 14 天窗口硬编码在 `EarningsPoller::new`,用户无法调。现在暴露到 YAML:

```yaml
event_engine:
  earnings:
    window_days: 14   # 从今天起向前看多少天的财报。典型 7~21 天
```

- 数字越大,越早开始跟这只票的财报(对深度持仓有意义);数字越小,越接近"只看下周"
- 默认 14;未在 YAML 里出现该字段的旧 config 无感升级,继续用 14
- 窗口外的财报既不会发初次预告,也不会生成倒计时 —— `window_days` 真正控制"从什么时候开始关注一只票的财报"

### 尊重用户静音偏好

整条财报 lifecycle(初次预告 + T-3/T-2/T-1 倒计时)复用同一个 `earnings_upcoming` kind,所以:

- 用户在 Telegram/iMessage/Feishu/Discord 上对 Hone 说"别再推财报了",agent 调 `notification_prefs_tool` 把 `earnings_upcoming` 加进 `blocked_kinds` —— **整条 lifecycle 立即静音**,不需要分别屏蔽倒计时和预告
- 发布当日的 `earnings_released` 是另一个 kind(带 surprise% + Yahoo 链接),用户想保留"只看实际财报结果、不要提前提醒"的人可以单独控制
- 部署方仍可用 `event_engine.disabled_kinds: ["earnings_upcoming"]` 做全局黑名单

# v0.1.45 (English)

Release date: 2026-04-21

> Turns earnings from a one-shot calendar entry into a paced lifecycle: configurable attention window, daily T-3/T-2/T-1 countdowns, and a press-release link attached on release day. The user's `earnings_upcoming` mute preference silences the whole lifecycle in one shot.

## What users should know

### Earnings lifecycle: T-14 teaser → T-3/T-2/T-1 countdowns → release-day link

Previously `EarningsPoller` fired a single `EarningsUpcoming` event (id = `earnings:{SYMBOL}:{date}`) when a new calendar entry showed up. The UX gap: users saw it once a week out, then forgot by the night before — exactly when they most wanted a heads-up.

v0.1.45 reshapes it into a full lifecycle:

1. **Initial teaser** (when the entry first enters the window): one-shot Medium event, id = `earnings:{SYMBOL}:{date}`, unchanged from before
2. **T-3 / T-2 countdowns**: daily Medium events, id = `earnings:{SYMBOL}:{date}:countdown:{N}`, title like `AAPL earnings in 3 days (2026-05-02)`
3. **T-1 countdown**: upgraded to High severity — bypasses digest and pushes immediately, title like `AAPL earnings tomorrow (2026-05-02)`
4. **Release day**: `EarningsSurprisePoller` now attaches `url = Some(https://finance.yahoo.com/quote/{TICKER}/press-releases/)` — a free, login-less, always-fresh press-release page

Each countdown id differs by day so EventStore's stable-id dedup doesn't collapse them; the initial teaser id stays the same so it doesn't re-fire on every poll.

### Configurable attention window: `event_engine.earnings.window_days`

Previously the 14-day window was hardcoded in `EarningsPoller::new` and users couldn't tune it. Now it's a YAML field:

```yaml
event_engine:
  earnings:
    window_days: 14   # days ahead to watch. Typical: 7–21
```

- Higher = start tracking earlier (useful for deep positions); lower = closer to "just next week"
- Default 14; configs without this field upgrade silently and keep 14
- Entries outside the window get neither an initial teaser nor any countdowns — `window_days` literally controls "when Hone starts paying attention to a ticker's next print"

### Respects user mute preference

The whole lifecycle (initial teaser + T-3/T-2/T-1 countdowns) shares the `earnings_upcoming` kind, so:

- A user telling Hone on Telegram/iMessage/Feishu/Discord "stop pushing earnings stuff" — agent invokes `notification_prefs_tool` to add `earnings_upcoming` to `blocked_kinds` — **silences the whole lifecycle** in one shot; no need to block countdowns and teasers separately
- Release-day `earnings_released` events (with surprise% and the Yahoo link) are a separate kind, so "I only want the actual result, skip the pre-warnings" is expressible
- Deployers can still use `event_engine.disabled_kinds: ["earnings_upcoming"]` as a global kill switch

## Developer / ops notes

## Upgrade guide

1. Pull latest `main`, run `cargo check --workspace --exclude hone-desktop` to confirm the build
2. Open your `config.yaml`:
   - Want the default window → nothing to change
   - Want to tune the attention window → `event_engine.earnings.window_days: 7` (next week only) or `21` (three weeks out)
3. Restart Hone (or `launch.sh --web`) — the engine starts fresh against `data/events.db`, prior delivery records gone; starting the next day, calendar entries in-window will begin their daily countdown
4. If you're the "just tell me the actual result, skip the pre-warnings" type, say "mute earnings_upcoming" on Telegram / iMessage / Feishu / Discord — agent writes it into your prefs and the whole lifecycle goes silent
