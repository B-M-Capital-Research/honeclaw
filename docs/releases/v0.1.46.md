# v0.1.46

发布日期:2026-04-21

English readers: please scroll down for the English version below.

> 推送时间和数据抓取时间的 gap 是 digest 场景的核心痛点。v0.1.46 做了架构级重构:24h 节奏的 poller 改成 **cron-aligned**,在 flush 前 N 分钟才拉;财报倒计时改成 **read-time derive**,在每次 flush 时刻从 EventStore 现算。结果就是 pre_market 08:30 推到你手机上的那条摘要,数据是 08:00 刚拉的,不再是昨晚你重启 Hone 时那个时刻的快照。

## 用户应该知道的

### 推送数据永远是"刚拉的"

之前 5 个 24h 节奏的 poller(earnings_calendar / corp_action / macro / analyst_grade / earnings_surprise)用的是固定 `interval` 轮询,每 24 小时跑一次。这意味着如果你昨天下午 3 点重启了 Hone,今天早上 08:30 pre_market digest 触发时,用的数据是昨天下午 3 点拉的 —— 和推送时刻差了 17 小时。

v0.1.46 把这 5 个 poller 改成 **cron-aligned**:

- 在 `digest.pre_market` / `digest.post_market` 两个 flush 窗口**前 N 分钟**各跑一次
- N 由新配置 `digest.prefetch_offset_mins` 控制,默认 **30**
- 冷启动时每个 poller 仍然立即跑一次(老用户首次打开 Hone 就能看到昨天没消化的事件)

也就是说,08:30 的早 digest 用的是 08:00 刚拉的数据;09:00 的盘后 digest 用的是 08:30 刚拉的数据。推送和抓取的 gap 从 "最差 24 小时" 降到 "恒定 30 分钟"。

### 财报倒计时不再受 poller 漂移影响

v0.1.45 的 T-3/T-2/T-1 倒计时是 `EarningsPoller` 在拉日历的时候顺手产出的。问题是:poller tick 的时间和 flush 时间是两个时间点 —— 如果 poller 碰巧在 08:29 跑完,08:30 能推;但如果 poller tick 漂到 08:31,那一天的倒计时就可能 off-by-one。

v0.1.46 的处理:

- `EarningsPoller` 只产出稳定 id 的 `earnings:{SYM}:{DATE}` teaser(Medium 进 digest)
- T-3 / T-2 / T-1 倒计时由 `DigestScheduler` 在**每次 flush 时刻**从 EventStore 的未来财报列表现算
- 倒计时事件是临时 `synth:` 前缀,不入库、不走 dedup,纯粹是 flush 那一瞬间叠加到每个订阅该 symbol 的 actor 的 digest 里

用户可见的行为变化:

- T-1 不再单独升 High:早 08:30 的 pre_market 窗口本身就是"盘前提醒",升 High 绕过 digest 反而多一条 noise。想要更早的提醒,调小 `digest.pre_market`(比如 `07:30`)即可
- 整条 earnings lifecycle 仍然共享 `earnings_upcoming` kind,把它塞进 `blocked_kinds` 就能一次性静音 teaser + 所有倒计时

### `poll_intervals` 大幅瘦身(破坏性,但 tolerant)

v0.1.46 起,`event_engine.poll_intervals` 只保留 **news_secs** 和 **price_secs** 两个真正时效敏感的 poller;原来的 5 个 24h `*_secs` 字段不再有效:

| 字段 | 状态 |
|------|------|
| `news_secs` | 保留,默认 900 (15 min) |
| `price_secs` | 保留,默认 300 (5 min) |
| `earnings_secs` | **已删除**,改 cron-align |
| `corp_action_secs` | **已删除**,改 cron-align |
| `macro_secs` | **已删除**,改 cron-align |
| `analyst_grade_secs` | **已删除**,改 cron-align |
| `earnings_surprise_secs` | **已删除**,改 cron-align |

旧 YAML 里残留的这 5 个字段**不会导致启动失败** —— serde 默认 tolerant,会悄悄忽略。但它们已经无效,看到后请清理。

## 没覆盖 / 已知限制

- `prefetch_offset_mins` 是一个全局值,没法按 poller 粒度配。如果以后发现 earnings_calendar 需要 60min 而 macro 只需要 10min,再拆字段
- `shift_hhmm_earlier` 不处理时区本身变化(夏令时切换日凌晨的 flush 可能错位 1 小时),MVP 阶段不修
- synth countdowns 只加到订阅该 symbol 的 actor —— 如果 actor 订阅了 `earnings_upcoming` kind 但没订阅该 symbol,不会看到倒计时。这是故意的:避免一个全关键词订阅的 actor 收到每个公司每天的倒计时 noise

# English

Release date: 2026-04-21

> The push-time vs fetch-time gap was the core pain point for digest flows. v0.1.46 ships an architectural refactor: the five 24h pollers become **cron-aligned**, firing N minutes before each digest window; earnings countdowns become **read-time derived**, computed at flush from the EventStore. Result: the 08:30 pre-market digest on your phone now reflects data pulled at 08:00, not whenever you last restarted Hone.

## User-facing

### Push data is always "just-fetched"

Before: 5 daily-cadence pollers (earnings_calendar / corp_action / macro / analyst_grade / earnings_surprise) used fixed `tokio::time::interval` at 24h. If you restarted Hone at 3pm yesterday, the 08:30 pre-market digest today was using data pulled 17 hours ago.

Now: those 5 pollers are **cron-aligned**:

- Each one fires `digest.prefetch_offset_mins` minutes before `digest.pre_market` AND before `digest.post_market` windows
- `prefetch_offset_mins` defaults to **30**
- Cold start still triggers each poller once immediately (so first-time users see yesterday's unread events)

So the 08:30 digest uses 08:00 data; the 09:00 post-market digest uses 08:30 data. The push/fetch gap drops from "up to 24h" to a consistent 30 minutes.

### Earnings countdowns no longer drift with poller ticks

In v0.1.45, T-3/T-2/T-1 countdown events were emitted by `EarningsPoller` as a side effect of calendar fetches. Problem: poller tick time and flush time are two distinct clocks — a tick drifting to 08:31 could cause an off-by-one in that day's countdown.

In v0.1.46:

- `EarningsPoller` emits only stable-id `earnings:{SYM}:{DATE}` teaser events (Medium, entering digest)
- T-3 / T-2 / T-1 countdowns are synthesized by `DigestScheduler` at each flush tick from the EventStore's upcoming earnings list
- Countdown events use a `synth:` id prefix — not stored, not deduped, pure per-flush overlays routed to subscribers of that symbol

User-visible change: T-1 is no longer auto-upgraded to High. The 08:30 pre-market window is itself the pre-market reminder; promoting T-1 above digest just adds noise. Want an earlier heads-up? Lower `digest.pre_market` to `07:30`.

Blocking still works the same way: `earnings_upcoming` in `blocked_kinds` silences the entire lifecycle (teaser + every countdown).

### `poll_intervals` breaking simplification (tolerant to stale YAML)

Only `news_secs` and `price_secs` remain. The 5 daily `*_secs` fields (earnings/corp_action/macro/analyst_grade/earnings_surprise) are **removed**.

Old YAML with those fields **will not crash** — serde silently ignores unknown fields. But they're dead, so clean them up when convenient.

3. Edit `config.yaml`:

## Known limits

- `prefetch_offset_mins` is global, not per-poller. If earnings_calendar ever needs 60min while macro only needs 10min, we'll split the field
- `shift_hhmm_earlier` is not DST-aware; the cutover day may mis-align by 1h in transition zones
- Synth countdowns only go to actors subscribed to that specific symbol, not to actors subscribed to the `earnings_upcoming` kind globally. This is intentional — a broad-kind subscriber would otherwise get every company's countdown every day
