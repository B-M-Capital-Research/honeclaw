# v0.11.2

发布日期：2026-05-12

English readers: please scroll down for the English version below.

## 用户应该知道的

- 公开用户端登录方式从“邀请码 + 密码”切换为“手机号 + 短信验证码”。公开 `/chat` 仍然是邀请制，用户手机号必须已经在管理端邀请码/白名单记录中，未加入白名单时会提示联系 `bm@hone-claw.com`。
- 管理端原有的邀请码记录现在作为用户端 SMS 登录白名单继续生效；历史邀请码不会作为第一登录方式展示给公开用户，但后台仍可用它维护允许访问的手机号。
- 用户端登录、聊天和个人页清理了旧密码流残留，删除了废弃的公开 modal / password setup 代码，并补充了 SMS 登录 E2E 覆盖。
- Feishu、事件推送、运行脚本、安装脚本和用户可见文案继续收敛，减少重复提示、误导性错误信息和旧配置口径。

## 谁应该尽快更新

- 正在开放 `hone-claw.com/chat` 或其它 public web 用户端入口的部署者。
- 需要把公开用户登录切到手机号短信校验，并继续复用管理端 invite whitelist 的团队。
- 依赖 Feishu 长连接渠道、事件 digest、scheduler heartbeat、CLI 安装/重启脚本或 public/admin 文案的维护者。

## 升级后会有什么变化

- 公开用户输入手机号后会先经过本地白名单查询；只有管理端已有 invite 用户记录的手机号才会触发阿里云 SMS 验证码发送。
- 阿里云短信认证通过环境变量配置：`ALIBABA_CLOUD_ACCESS_KEY_ID` / `ALIBABA_CLOUD_ACCESS_KEY_SECRET`，也兼容 `ALIYUN_*` 与 `HONE_ALIYUN_*` 前缀。默认国家码为 `86`，默认签名为 `速通互联验证码`，默认 template code 为 `100001`。
- `/api/public/auth/sms/send` 和 `/api/public/auth/sms/login` 成为 public web 登录链路；旧的公开邀请码密码登录 UI 被移除。
- 用户端 public bundle、管理端用户设置文案、Terms / Privacy 文案会同步说明 SMS 登录与邀请制白名单。

## 安装与更新

- `curl | bash` 用户：重新运行安装脚本即可切到新版本
- Homebrew 用户：`brew update && brew upgrade honeclaw`

## 已知注意事项

- 运行 public SMS 登录前，后端运行环境必须提供阿里云短信 AccessKey；不要把真实密钥提交到仓库。
- 架构 SVG 不需要更新：本次 release 没有新增 channel、runner、poller、EventKind 或存储布局；public web 仍沿用既有 admin / public dual-surface 架构，只是替换 public auth 登录方式。
- 远端 release workflow 在 tag 推送后由 GitHub Actions 触发；本地只能确认 tag 已推送，不能替代远端产物发布结果。

## English

Release date: 2026-05-12

### What Users Should Know

- Public user login has moved from “invite code + password” to “phone number + SMS code”. Public `/chat` remains invite-only: the phone number must already exist in the admin invite/whitelist records, and non-whitelisted users are told to contact `bm@hone-claw.com`.
- Existing admin invite records now act as the whitelist for SMS login. Historical invite codes are no longer the primary public login method, but the admin surface can still use them to manage allowed phone numbers.
- The public login, chat, and account pages removed stale password-flow code, deleted the old public modal/password setup surfaces, and added SMS-login E2E coverage.
- Feishu, event delivery, runtime scripts, install scripts, and user-facing copy were tightened to reduce repeated prompts, misleading errors, and stale configuration language.

### Who Should Update Soon

- Operators exposing `hone-claw.com/chat` or another public web user entrypoint.
- Teams that want public users to authenticate with SMS while continuing to reuse the admin invite whitelist.
- Maintainers of the Feishu long-connection channel, event digests, scheduler heartbeats, CLI install/restart scripts, or public/admin copy.

### What Will Change After Upgrading

- Public users are checked against the local whitelist before an SMS is sent. Only phone numbers with existing admin invite-user records can receive an Aliyun SMS verification code.
- Aliyun SMS verification is configured through `ALIBABA_CLOUD_ACCESS_KEY_ID` / `ALIBABA_CLOUD_ACCESS_KEY_SECRET`; the `ALIYUN_*` and `HONE_ALIYUN_*` prefixes are also supported. The default country code is `86`, the default sign name is `速通互联验证码`, and the default template code is `100001`.
- `/api/public/auth/sms/send` and `/api/public/auth/sms/login` are now the public web login path. The old public invite/password login UI has been removed.
- The public web bundle, admin user settings copy, Terms, and Privacy pages now describe SMS login and the invite-only whitelist model.

### Install and Update

- `curl | bash` users: rerun the install script to move to this version
- Homebrew users: `brew update && brew upgrade honeclaw`

### Known Notes

- The backend runtime must provide Aliyun SMS AccessKey credentials before public SMS login can send real codes. Do not commit real credentials to the repository.
- Architecture SVG does not need an update: this release does not add channels, runners, pollers, EventKinds, or storage layout changes. The public web still uses the existing admin / public dual-surface architecture; only the public auth method changed.
- The remote release workflow is triggered by the pushed tag in GitHub Actions. Local verification can confirm that the tag was pushed, but not that remote release assets finished publishing.
