# v0.1.34

发布日期：2026-04-17

English readers: please scroll down for the English version below.

## 用户应该知道的

- Feishu 直聊和定时任务这轮重点补的是“不要静默失败”：空回复伪成功、placeholder 后停住、scheduler 生成成功但发送阶段 `HTTP 400` 这几类高优先级问题，现在都增加了更强的发送回退和失败兜底。
- Feishu 出站链路现在会在 `update_message` 或 `reply_message` 命中 `HTTP 400 Bad Request` 时自动退回到 standalone send，避免消息卡死在旧 placeholder 或回复链路上。
- direct scheduler 面向 `open_id` 的多段消息不再默认依赖 reply 链路，降低“内容已生成但用户收不到”的概率。
- `AgentSession` 现在会把“净化后为空的 success 结果”改写成统一 fallback 文案，不再把空 assistant 当成成功持久化。
- `hone-cli` 对 Telegram / Discord / OpenCode 等凭据输入做了统一 `trim` 与更稳的必填收口，尤其 Discord token 的可见输入、格式校验和恢复提示更准确。

## 谁应该尽快更新

- 依赖 Feishu 私聊稳定回复的用户，尤其是最近遇到过 placeholder 后无后续消息、空答复或统一失败兜底过多的会话。
- 使用 Feishu 定时播报、盘前盘后提醒或晨报任务的维护者，尤其是近期在日志里见过 `HTTP 400 Bad Request` 发送失败的场景。
- 通过 `hone-cli onboard` 或 `hone-cli configure --section channels` 配置 Discord / Telegram / OpenCode 凭据，并希望减少空值误判、空白字符污染或 token 提示误导的维护者。

## 升级后会有什么变化

- Feishu 发送链路在 placeholder 更新失败、reply 失败或 handler panic 时，会更积极地回退到直接发送可见文本，而不是把一次处理留在半完成状态。
- 当 agent 结果表面 `success=true`，但对用户可见的正文在净化内部内容后变成空串时，系统会改发统一 fallback，而不是继续落空内容。
- scheduler 对 direct `open_id` 目标的多段发送顺序更稳，不再默认把后续分段都绑在脆弱的 reply 链路上。
- CLI 保存凭据前会统一裁掉首尾空白；一些此前会因为“非空但无效”而误走空值恢复菜单的场景，现在会给出更准确的下一步提示。

## 安装与更新

- `curl | bash` 用户：重新运行安装脚本即可切到新版本
- Homebrew 用户：`brew update && brew upgrade honeclaw`

## 已知注意事项

- 这次 release 先解决的是“失败时至少给出用户可见结果”和“发送链路更稳”的问题；导致 search 阶段反复调用 `cron_job` 的上游策略根因仍需继续观察和单独治理。
- 几个 Feishu `P1` 缺陷当前状态仍是 `Fixing` 而非 `Fixed`，因为还需要下一批真实直聊 / scheduler 样本确认回归已经稳定收口。
- 本次 compare 范围同时包含活跃 bug 文档和计划文档更新；它们用于记录当前修复状态，不代表所有问题都已完全关闭。

## English

Release date: 2026-04-17

### What Users Should Know

- This release focuses on stopping silent failures in Feishu direct chats and scheduled jobs. Several P1 cases now have stronger send fallbacks and explicit user-visible failure handling, including empty-success replies, placeholder-only stalls, and scheduler deliveries that previously failed with `HTTP 400`.
- The Feishu outbound path now falls back to standalone sends when `update_message` or `reply_message` returns `HTTP 400 Bad Request`, instead of getting stuck on an old placeholder or reply chain.
- Multi-segment direct scheduler messages sent to `open_id` targets no longer depend by default on the reply flow, which reduces the chance that content is generated successfully but never delivered.
- `AgentSession` now rewrites “successful but empty after sanitization” outputs into a consistent fallback message instead of persisting an empty assistant response.
- `hone-cli` also tightens credential input handling across Telegram, Discord, OpenCode, and related flows by trimming values consistently and improving required-field recovery, especially for Discord token entry and validation.

### Who Should Update Soon

- Users who rely on Feishu direct chats and have recently seen placeholder-only hangs, empty replies, or overly frequent generic fallback messages.
- Maintainers running Feishu scheduled briefings, market alerts, or direct reminders, especially if recent logs contained `HTTP 400 Bad Request` delivery failures.
- Maintainers who use `hone-cli onboard` or `hone-cli configure --section channels` to manage Discord, Telegram, or OpenCode credentials and want fewer empty-value misclassifications or whitespace-related config issues.

### What Will Change After Upgrading

- Feishu delivery now falls back more aggressively to direct visible sends when placeholder updates fail, reply sends fail, or the handler panics, rather than leaving the interaction half-finished.
- When an agent reports `success=true` but the user-visible content becomes empty after stripping internal-only output, the system now emits a fallback message instead of persisting an empty response.
- Scheduler delivery for direct `open_id` targets is more robust for multi-part messages because follow-up segments no longer default to the more fragile reply path.
- CLI credential flows now trim leading and trailing whitespace before persisting values, and some cases that previously misrouted non-empty invalid input into empty-value recovery now show clearer next-step prompts.

### Install and Update

- `curl | bash` users: rerun the install script to move to this version
- Homebrew users: `brew update && brew upgrade honeclaw`

### Known Notes

- This release primarily improves user-visible fallback behavior and send-path resilience. The upstream strategy issue that can cause repeated `cron_job` calls during search still needs separate follow-up work.
- The affected Feishu P1 bugs remain in `Fixing` rather than `Fixed` because the final close-out depends on fresh real-world direct-chat and scheduler samples.
- The compare range also includes updates to active bug and planning docs that capture current repair status; those docs should not be read as proof that every issue is fully closed.
