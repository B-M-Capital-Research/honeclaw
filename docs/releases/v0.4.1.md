# v0.4.1

发布日期：2026-04-28

English readers: please scroll down for the English version below.

> 这是 v0.4.0 之后两天的快速跟进版本，主线是公开端 chat 完整改版（白色主题 + 移动端可用性收口）+ 管理端新增 `/notifications` 推送日志页用于跨任务排查“到底有没有发出去”。同时移除了 `event_engine.dryrun` 配置项 —— 这是个隐藏地雷，配齐渠道的部署可能因为它而以为推送链路活着、实际只在写日志。其余是 global digest Pass1 模型替换、scheduler 跳过的 assistant turn 回滚、FMP 大 payload 卡死等一批 P1/P2 修复。

## 用户应该知道的

### 1. 管理端：新增 `/notifications` 推送日志页

跨任务汇总了 cron 推送的执行记录，专门用来排查“给某个用户的某条推送到底成没成功”。位于 sidebar “用户视角 / 推送日志”：

- **顶部 6 个汇总卡**：24h 总数 / 已发送 / 发送失败 / 主动跳过 / 重复拦截 / 覆盖用户。
- **24h 小时直方图**：按小时叠加 sent / failed / skipped 三色，一眼看出某个时段是不是集中失败。
- **扁平表 + 过滤器**：时间窗口（1h / 6h / 24h / 3d / 7d）、渠道、用户 ID、发送状态、执行状态。点行展开右侧抽屉看完整 detail / response_preview / error。
- **后端**：`GET /api/admin/notifications`（auth 受 `require_api_auth` 保护）跨 `cron_job_runs` 表聚合，复用现有索引，单次扫盘控制在 1k 行内。

之前要回答“某用户最近收到了什么推送”得分别去 `/sessions`（看气泡）和 `/tasks`（看每个任务的执行记录），现在一个页面解决。

### 2. Public 端：完整改版 + 移动端 UX 收口

- **整套 landing / roadmap / chat 改版**（#a67cc0e）：白色主题、本地化资源、信息架构整理。新部署默认即是新风格，无需迁移。
- **/me + /portfolio 重塑**：投资偏好、quota、密码修改、登出等账户操作收到 `/me`；`/portfolio` 仍然展示蒸馏的 thesis 卡片和投资风格。
- **移动端 UX 多轮迭代**（#1e31b7b → #d9b53f5 等）：composer reachability、表格横向溢出、状态条、sticky scroll、字号 + 主题控件、prefs popover 防越界、关闭操作（Esc / 点外部）等都按真机调过一轮。整体密度比照微信 chat。
- **若干 fix**：unlimited quota 被误判耗尽（#88c3ccf）、auth 刷新后未持久化（#14b5d62）、密码 guard 误触发（#6d30428）、checkbox 内嵌 ToS / Privacy 链接被吞（v0.4.0 已修，v0.4.1 进一步收尾 popover & invite 文案）。

### 3. Event engine：移除 `event_engine.dryrun` 配置项（破坏性，但低风险）

`event_engine.dryrun` 是早期“接 sink 之前怕骚扰用户”的过渡配置。配置 = `true` 时，event engine 一律装 `LogSink`，**渲染好的消息只写日志、不真发**。这个开关在 `config.example.yaml` 里默认 `true`，是个静默地雷：

> 用户跟着 `hone-cli onboard` 配齐 Discord token、加了 admins 白名单、bot 也起来了，但因为这一行没改，event-engine 推送一条都到不了用户那。

v0.4.1 直接把它删掉：

- `EventEngineConfig::dryrun` 字段移除；`config.yaml` / `config.example.yaml` 里如有 `dryrun:` 行会被 serde 静默忽略，不报错。
- `build_event_engine_sink` 一律按 `channel.enabled` + 凭据情况装真 sink；未注册的 actor.channel 仍 fall back 到 `LogSink`（这条 fallback 路径仍写 `delivery_log.status='dryrun'` 防止被 `count_high_sent_since` 误算成真 ack —— 是稳定契约，不变）。
- `engine.start` 启动 tracing 行也去掉了 `dryrun` 字段。

### 4. Global digest：Pass1 切到 grok-4.1-fast + 接纳 FMP trusted-Low

- **Pass1 模型切换**（#e29a898）：从 nova-lite 换到 `x-ai/grok-4.1-fast`，候选评分 / 聚类的判定一致性显著改善。
- **接纳 FMP trusted-Low**：原本 trusted-source 只看 High/Medium，FMP 的部分 trusted-Low 被无差别拒之门外；现在按规则放行进入候选池，再交 Pass1 评分判定。

### 5. Scheduler：被跳过的 assistant turn 回滚修复

`fix(scheduler): rollback skipped assistant turns`（#83dd922）：调度器在某些 race 下会让一条已跳过的 assistant turn 留在 session log 里，下一轮 compact summary 看到“说了一半”的回合而生成幻觉摘要。本次直接在跳过路径里把 turn 回滚掉，session log 与 conversation 状态保持一致。

### 6. FMP：开 gzip + timeout 提到 60s

`fix(fmp): enable gzip and raise timeout to 60s`（#0081b4d）：拉大 payload（earnings_calendar / corp_action 等）时偶发超时 + 明显大于必要的带宽。本次同时开 gzip 与放宽 timeout，`fmp ok` 节奏更稳，巡检里不再频繁出现 30s 超时降级。

### 7. 其它修复

- **Desktop invite capability**（#2f257a6）：desktop 端启动后的 invite 功能 capability 漏配，导致按钮不可用。
- **Public chat header deployment docs**（#ec7931a）：部署文档遗留旧 header 描述。
- **Skill：company_portrait profile 必须带 ticker frontmatter**（#5a38c17）：缺 ticker 的画像在 thesis 蒸馏链路里会被静默跳过；现在 skill 强制要求 frontmatter 里必须有 ticker。

## 谁应该尽快更新

- **任何在 `config.yaml` 里曾设过 `event_engine.dryrun` 的部署**：升级后字段会被忽略，**真实 sink 立即开始按 channel 配置投递**。如果之前是为了"测试不骚扰用户"而设 `dryrun: true`，请改为 `event_engine.enabled: false` 或临时把对应 channel `enabled: false`。
- **公开端有真实用户的部署**：移动端可用性大幅提升，建议升级。
- **使用 cron 推送任务的部署**：`/notifications` 页能省掉很多排查时间。
- **依赖 FMP 拉大 payload 的部署**：FMP 超时降级会显著减少。

## 升级后会有什么变化

- **`event_engine.dryrun` 字段失效**：保留在 yaml 里不会报错，但完全无效。如果之前用它兜底"不真发"，请改用 `event_engine.enabled: false`。
- **管理端 sidebar 多一项 "推送日志"**（受 `cron_jobs` capability 控制）：路径 `/notifications`。
- **公开端整套 UI 风格变白**：信息架构与原来一致，浏览器里直接刷新即可。
- **public chat 移动端密度变高**：更接近微信的对话密度，桌面端不受影响。
- **global digest 候选打分 / 聚类判定更稳**：模型从 nova-lite 切到 grok-4.1-fast；trusted-Low FMP 信号会进入候选池。
- **scheduler 跳过 assistant turn 后的下轮 compact summary 不再幻觉**：session log 状态保持一致。
- **FMP 拉数据更稳**：开 gzip + 60s timeout。

## 安装与更新

- `curl | bash` 用户：重新运行安装脚本即可切到新版本。
- Homebrew 用户：`brew update && brew upgrade honeclaw`。
- Admin console 用户：刷新页面；新页面在 sidebar "用户视角 / 推送日志"。
- **生产环境注意**：升级前确认 `config.yaml` 里如果设了 `event_engine.dryrun: true`，是出于真实"不想骚扰用户"目的还是历史遗留 —— 真有需要可改用 `event_engine.enabled: false` 或对应 channel `enabled: false`。

## 已知注意事项

- `LogSink` 仍以 `delivery_log.status='dryrun'` 标记其 fallback 输出（actor.channel 没注册到真 sink 的情况）。这是历史 5 处 SQL 查询的稳定契约，**不会**因本次 release 改名或迁移；只是 config 那一层的开关没了。
- tag 推送只代表 GitHub release workflow 被触发；最终 tarball / Homebrew formula 是否发布，请以 GitHub Actions 结果为准。

## English

Release date: 2026-04-28

> A two-day fast-follow on v0.4.0. Headline themes: a full white-theme redesign of the public landing / chat (with mobile usability hardening), a new admin `/notifications` push-log page for cross-task delivery investigation, and the removal of the `event_engine.dryrun` config flag — a silent landmine where deployments believed pushes were live while the engine was only logging. Plus a batch of P1/P2 fixes: global digest Pass1 model swap, scheduler skipped-turn rollback, FMP large-payload timeouts.

### What Users Should Know

#### 1. Admin: New `/notifications` Push Log Page

Cross-task aggregation of cron push execution records, built specifically to answer "did this push actually deliver?" Sidebar: 用户视角 / 推送日志.

- **Six summary cards**: 24h total / sent / failed / skipped / duplicate-suppressed / distinct users.
- **24h hour histogram**: stacked sent/failed/skipped per hour, makes time-of-day failure spikes obvious.
- **Flat table + filters**: window (1h / 6h / 24h / 3d / 7d), channel, user id, send status, exec status. Click a row to open a right-side drawer showing full detail / response_preview / error.
- **Backend**: `GET /api/admin/notifications` (under `require_api_auth`) aggregates over `cron_job_runs`, reuses existing indexes, caps single scan at ~1k rows.

Previously to answer "what pushes did user X recently receive" you had to look in `/sessions` (chat bubbles) and `/tasks` (per-task executions). Now one page.

#### 2. Public Surface: Full Redesign + Mobile UX Hardening

- **Landing / roadmap / chat redesign** (#a67cc0e): white theme, localized assets, IA cleanup. New deployments get the new style by default; no migration needed.
- **/me + /portfolio reshape**: account ops (preferences, quota, password, logout) consolidated under `/me`; `/portfolio` continues to show distilled theses + investment style.
- **Mobile UX iterations** (#1e31b7b → #d9b53f5, etc.): composer reachability, table overflow, status bar, sticky scroll, font-size + theme controls, prefs popover anti-overflow, dismissal (Esc / outside-click) — all dialed in against real devices. Density now closer to a WeChat chat.
- **Misc fixes**: unlimited quota was misread as exhausted (#88c3ccf), auth not persisted across refresh (#14b5d62), password guard misfire (#6d30428), polishing of popover & invite copy after the v0.4.0 link-eater fix.

#### 3. Event Engine: `event_engine.dryrun` Config Flag Removed (Breaking, Low Risk)

`event_engine.dryrun` was a "before we wired real sinks" transitional knob. When `true`, the event engine assembled `LogSink` and **the rendered messages only went to logs — nothing actually sent**. The flag defaults to `true` in `config.example.yaml`, which is a silent landmine:

> A user follows `hone-cli onboard`, configures the Discord token, adds themselves to admins, the bot starts up — but because this single line was never flipped, not one event-engine push reaches the end user.

v0.4.1 deletes it outright:

- `EventEngineConfig::dryrun` field is gone. Stale `dryrun:` lines in `config.yaml` / `config.example.yaml` are silently ignored by serde — no error.
- `build_event_engine_sink` always assembles real sinks based on `channel.enabled` + credentials; unregistered `actor.channel`s still fall back to `LogSink` (that fallback path keeps writing `delivery_log.status='dryrun'` so `count_high_sent_since` doesn't miscount it as a real ack — that's a stable contract, unchanged).
- The `engine.start` startup tracing line drops the `dryrun` field.

#### 4. Global Digest: Pass1 Switched to grok-4.1-fast + Admit FMP Trusted-Low

- **Pass1 model swap** (#e29a898): from nova-lite to `x-ai/grok-4.1-fast`. Candidate scoring / clustering judgment consistency is materially better.
- **Admit FMP trusted-Low**: trusted-source previously only accepted High/Medium and dropped FMP's trusted-Low items wholesale; they now enter the candidate pool and let Pass1 score them.

#### 5. Scheduler: Skipped Assistant Turn Rollback Fix

`fix(scheduler): rollback skipped assistant turns` (#83dd922): under certain races the scheduler left a skipped assistant turn in the session log, and the next compact summary saw a "half-spoken" turn and hallucinated a recap. v0.4.1 rolls the turn back inside the skip path, keeping session log and conversation state consistent.

#### 6. FMP: Enable gzip + Raise Timeout to 60s

`fix(fmp): enable gzip and raise timeout to 60s` (#0081b4d): pulling large payloads (earnings_calendar / corp_action / …) occasionally hit the 30s timeout and consumed more bandwidth than necessary. Enabling gzip and widening the timeout stabilizes `fmp ok` cadence — patrols no longer see frequent 30s timeout degradations.

#### 7. Other Fixes

- **Desktop invite capability** (#2f257a6): the desktop-side post-launch invite button was unusable due to a missing capability entry.
- **Public chat header deployment docs** (#ec7931a): deployment docs still referenced the old header.
- **Skill: company_portrait profile must carry ticker frontmatter** (#5a38c17): profiles missing the ticker field were silently skipped in the thesis-distill chain; the skill now enforces frontmatter ticker.

### Who Should Update Soon

- **Any deployment that has ever set `event_engine.dryrun` in `config.yaml`**: after upgrade, the field is ignored and **real sink delivery starts immediately based on channel config**. If you used `dryrun: true` to "test without spamming users," switch to `event_engine.enabled: false` or temporarily disable the corresponding channel.
- **Deployments with real public users**: mobile usability is materially improved; recommend upgrading.
- **Deployments using cron push tasks**: `/notifications` will save substantial investigation time.
- **Deployments that pull large FMP payloads**: FMP timeout degradations will drop significantly.

### What Will Change After Upgrading

- **`event_engine.dryrun` field is inert**: leaving it in yaml won't error but has no effect. If you relied on it as a safety net, switch to `event_engine.enabled: false`.
- **Admin sidebar gains "推送日志"** (gated on `cron_jobs` capability): path `/notifications`.
- **Public surface UI turns white**: same IA, refresh in the browser.
- **Public chat mobile density rises**: closer to a WeChat-style chat, desktop unaffected.
- **Global digest scoring/clustering becomes more stable**: model swapped to grok-4.1-fast; trusted-Low FMP signals now enter the candidate pool.
- **No more hallucinated compact summaries after a skipped assistant turn**: session log state stays consistent.
- **FMP fetches are more stable**: gzip + 60s timeout.

### Install and Update

- `curl | bash` users: rerun the install script.
- Homebrew users: `brew update && brew upgrade honeclaw`.
- Admin console users: refresh; the new page lives at sidebar 用户视角 / 推送日志.
- **Production note**: before upgrading, confirm whether `event_engine.dryrun: true` (if present in your `config.yaml`) is intentional safety or stale config — if intentional, switch to `event_engine.enabled: false` or per-channel `enabled: false`.

### Known Notes

- `LogSink` still tags its fallback writes (when `actor.channel` isn't registered to a real sink) as `delivery_log.status='dryrun'`. That's a stable contract for 5 historical SQL queries and **won't** be renamed or migrated by this release; only the config flag is gone.
- Pushing the tag only triggers the GitHub release workflow; whether the tarball / Homebrew formula publish must be confirmed via GitHub Actions.
