## Goal

Ship v0.14.1 with a fully visible desktop finance-calendar modal and a stable persistent WebKit data store for the standalone macOS user app so login cookies survive app restarts and upgrades.

## Scope

- Keep the finance-calendar dialog inside the desktop viewport, above the fixed chat shell, with internal scrolling where necessary.
- Give `hone-user-app` a deterministic macOS WebKit data-store identifier while preserving the default persistent fallback on macOS 12/13.
- Build and inspect the Universal macOS app and DMG.
- Align every formal-release version location, add bilingual release notes, push `main`, and publish annotated tag `v0.14.1`.

## Validation

- Completed: Web typecheck, 229 Web tests, public Web production build, and `1000x620` desktop viewport verification with internal modal scrolling.
- Completed: workspace Rust check, 4 macOS user-app tests, app-specific check, changed-file formatting, and the version-aware iOS remote-shell contract.
- Completed: Universal `arm64/x86_64` App/DMG build, strict bundle signature verification, DMG checksum verification, minimal bundle inspection, and two packaged-app launch smokes that reused the same named WebKit data store.
- Pending: pushed tag, GitHub Actions release run, and published Apple asset verification.

## Documentation Sync

- Add `docs/releases/v0.14.1.md` and update the macOS user-app runbook with the persistent WebKit store contract.
- Architecture SVG does not require an update because the remote-only client boundary, modules, runners, channels, and storage layout are unchanged.
- On completion, add/update the release handoff and archive index, move this plan to `docs/archive/plans/`, and remove it from `docs/current-plan.md`.

## Risks / Open Questions

- macOS 14+ supports named `WKWebsiteDataStore`; macOS 12/13 must continue using WebKit's default persistent store.
- The public DMG remains ad-hoc signed unless external Developer ID and notarization credentials are available.
- Tag publication triggers multiple platform artifacts; completion requires checking the remote workflow rather than assuming success after tag push.
