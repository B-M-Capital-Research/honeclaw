HONE 技术架构图

用一条主链路说明：用户消息从各通道进入后，如何在 hone-channels 里完成编排、由 runners 执行、经 hone-tools 与 memory 读写本地数据，最后回到通道或 Web 控制台。
另外还有一条独立于对话的链路 —— 主动事件 Feed，由 hone-event-engine 定时从外部数据源拉事件、按用户偏好路由，再经 MultiChannelSink 真实投递到各通道（见下方 ⑦）。

v0.2.0 · 2026-04-21 · feed 真实投递

▼ 进入 hone-channels：统一入口模型
2
编排核心：ingress → AgentSession
hone-channels

crates/hone-channels 中的 ingress 把「谁在说话、写到哪条会话、是否重复、会话锁」收敛成 IncomingEnvelope。 AgentSession::run 负责加载会话、拼装提示词（含 prompt.rs 全局金融域约束）、注册工具、选择沙箱，并把 runner 产出写回会话。

🏗 HoneBotCore

依赖工厂：读配置、初始化 LLM、存储、调度器、审计等单例。

🪪 ActorIdentity

「谁在执行」——配额、审计归档、工具可见性等按 Actor 隔离。

📎 SessionIdentity

「历史写到哪」——群聊共享会话等场景与 Actor 解耦。

🔧 ToolRegistry

按角色动态装配工具；ToolExecutionGuard 约束危险操作。

附件进 KB：在 attachments.rs 统一门禁（例如通用附件 5MB、图片 3MB、极端长宽比/分辨率拒绝），未通过的不进入 prompt 与知识库。
  • 会话压缩：消息条数 / 体积累积后摘要，控制上下文体积（与实现中的阈值一致）。
  • 出站：outbound 与各通道适配器消费统一事件，处理占位符、推理展示、分块与流式。

▼ 另一条链路：主动事件 Feed（独立于 AgentSession）
📡
主动事件 Feed
hone-event-engine · proactive push（v0.2.0 首次真实投递）

用户不发消息也能收到的那条链路。hone-event-engine 每隔固定窗口从外部数据源拉事件、去重落盘、按订阅与用户偏好路由，最终经 MultiChannelSink 真实投递到 Telegram / Discord / 飞书 / iMessage。整条流水线独立于 AgentSession，只借 HoneConfig 共享渠道凭据。

Feed 链路 → 外部数据源 → Pollers → EventStore（dedup） → Router（prefs + severity） → 立即推 / Digest Buffer → Renderer → MultiChannelSink → 通道 API
① 采集 · Pollers

📰 news

FMP stock_news，15 min tick。

💹 price

FMP quote 按 watch pool，5 min tick；阈值触发 price_alert / 52W / VolumeSpike。

📅 cron-aligned

earnings_calendar · earnings_surprise · corp_action · sec_filings · macro_calendar · analyst_grade；在 digest flush 窗口前 30 min 各跑一次（v0.1.46 起）。

▼ 落盘 + 路由决策
② 入库 & 路由

🗄 EventStore

sqlite（主存）+ events.jsonl（mirror）；按稳定 event id 去重。

🧭 Router

按订阅匹配 actor；逐个对照 NotificationPrefs（blocked_kinds / min_severity / portfolio_only）；阈值决定 severity。

🚦 分流

severity=High → 立即推（同 ticker 60 min cooldown、per-actor 日配额）；Low / Medium → Digest Buffer。

▼ Digest 调度（pre/post market）
③ Digest 调度 · Renderer

⏰ DigestScheduler

北京时区 cron；两个 flush 窗口 digest.pre_market / digest.post_market。

🧮 Read-time 衍生

财报 T-3 / T-2 / T-1 倒计时等在 flush 时刻从 EventStore 现算，不入库（v0.1.46 起）。

📝 Renderer

RenderFormat::Plain 统一渲染；max_items_per_batch 防溢洪。

▼ 真实投递（v0.2.0）
④ OutboundSink · v0.2.0 首次真实投递

v0.1.46 及之前只有 LogSink（主动事件仅落日志、不投递）。v0.2.0 起 hone-web-api 按 HoneConfig 各渠道的 enabled + 凭据就绪条件装配 MultiChannelSink，并按 ActorIdentity.channel 派发到具体 sink。

🧩 MultiChannelSink

路由表 channel → Arc<dyn OutboundSink>；未注册 / sink 抛错自动 fallback 到 LogSink。

📱 TelegramSink

Bot API sendMessage；私聊用 user_id、群聊剥 chat_ 前缀。

🎮 DiscordSink

/users/@me/channels 解 DM channel（进程内缓存）+ /channels/{id}/messages。

🪶 FeishuSink

缓存 tenant_access_token（提前 5 min 过期）+ /open-apis/im/v1/messages。

💬 IMessageSink

osascript + Messages.app；非 macOS 直接 bail 走 fallback。

🗒 LogSink（fallback）

所有未装配渠道的兜底；event_engine.dryrun=true 时强制全员走 LogSink。

用户偏好尊重：每个 actor 的 data/notif_prefs/<actor_key>.json 决定 min_severity、blocked_kinds、portfolio_only。把 earnings_upcoming 塞进 blocked_kinds 即可一键静音整条财报生命周期。
