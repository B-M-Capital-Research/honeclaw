> 前言：我们从 2 月开始启动 Hone 项目，经历了从 Python + Node.js 到 Rust 的全重构转型。这个过程里积累了很多实践经验，也越来越确信 Rust 在 AI Coding 时代是更优解，所以写下这篇文章做一次分享，希望大家能在技术选型的最左侧拥抱更多技术红利。

# TLDR

1. **上下文寸土寸金：** Rust 可以用更小的上下文表达业务，在上下文成为稀缺资源时节省 token；面对异步和并发，也更安全、更直观。相比之下，Java 往往引入大量依赖、样板和人为规约。
2. **不需要混合技术栈：** Rust 可以写后端、前端和客户端。它性能接近 C++，部署和启动却简单得多，适合搭建前后端一体、甚至带端侧的服务。
3. **内存安全，编译即解决大部分问题：** Rust 通过所有权和强类型系统，把大量问题暴露在编译期，而不是让它们在运行时反复出现。
4. **资源占用低，稳定性高：** Rust 重写的 Agent / CLI / 桌面工具通常能显著降低内存、启动延迟和二进制体积。
5. **AI Coding 改变了学习曲线：** Rust 学习成本高，但 AI 已经能承担大部分语法和样板生成；团队更应该把精力放在 CR、安全和工程判断上。

# 1. 寸土寸金的上下文

Anthropic Skills 现在是一个很火的概念，各种 Agent 系统都在走 Skill 插件化，而“渐进式披露”的核心就是解决上下文问题：上下文是稀缺资源，要合理利用，不能乱塞。早期 MCP 时代，动辄几百个工具和元数据塞进上下文；现在又返璞归真，回到 Skill 化治理，这些都属于 Context Engineering。

上下文过长会带来很多问题。最明显的是算力成本，算力目前依然很贵；其次是模型效果会变差。真正跑复杂的大规模项目时，会发现 AI Coding 的一次性完成能力会明显下降，经常要么改一半就停，要么考虑不全，需要大量人工兜底。

我们的实践经验是，采用 Rust 可以很大程度缓解这个问题。语言和技术栈对项目工程的影响非常巨大，在最左侧做好技术选型，可以更早享受复利。

先看 Java 会遇到哪些痛点。Java 是开发者使用最多的语言之一，长期遵循单文件公共类原则，并鼓励深层包结构和设计模式。当 AI 需要理解一个完整核心业务链路时，必须把数十个离散的接口定义、实现类、DTO/DO、配置文件一起加载进上下文。最直观的是 Spring Boot 的 Hello World，也需要多个文件夹和一批文件配合，这对 Function Calling 和 Agent 多轮 ReAct 很不友好。

> 案例 1：Java 经常出现文件夹多、文件多、命名冗长的问题。比如 Spring AI 这种相对新的 Java 项目，已经比很多旧项目清爽，但依然绕不开深包结构、长命名和面向对象规约带来的上下文冗余。

![Java 项目结构上下文膨胀示例](/blog/why-hone-uses-rust-java-files.png)

> 案例 2：大量用于协作的声明和描述文本。写过 Java 的朋友都熟悉 DO/DTO 这类模型，一个简单对象可能需要接近 100 行代码和一个完整文件去描述。

![Java DTO 样板代码示例](/blog/why-hone-uses-rust-java-dto.png)

相比之下，Rust 在语法设计和语言特性上展现出很高的 token 效率。它融合了底层系统编程的严谨性和现代函数式编程的高级抽象，让 AI 能在较短上下文窗口里捕捉系统业务的深层语义。

> 案例 1：Rust 项目通常没有大量长命名文件、长文件和冗长规约。比如 OpenAI Codex，很多代码能看出 AI 参与的痕迹，但整体清爽简洁，异步代码也没有太多多余结构。

> 案例 2：模型定义、处理逻辑、组装逻辑乃至单元测试，都可以在一个文件里高密度组织。比如 Qdrant，相关函数、枚举、trait 和测试可以保持局部性，让 AI 的注意力更聚焦。

现代软件工程里，异步和并发是 AI 最容易出错的领域。Java 的传统多线程模型和并发框架往往伴随线程池管理、锁分配和回调脚手架；人容易写错，AI 更容易写错。很多 Agent 的 SSE 代码也容易出现死锁和性能问题。

Rust 的 async/await 模型配合所有权检查提供零成本抽象。写大量网络连接代码时，AI 只需要使用简练的闭包和 `await`，编译器会把它展开成高效状态机。AI 不需要在上下文里推理复杂的底层状态转移和线程调度，代码逻辑更纯粹，也更适合有限 token 配额下解决复杂并发问题。

# 2. 各端一套技术栈

软件项目在演进过程中，架构复杂度往往随着跨平台、多端支持需求指数级上升。很多 AI 工具现在需要同时提供 Web 服务、Mac 和 Windows 桌面端，例如 OpenClaw、Codex、Qwen、Seed 等。AI 工具多端适配已经成为趋势。

传统模式是后端、前端、客户端各玩一套。现在强调 AI 原生和全栈开发，技术栈融合一直在发生，只是语言暂时还没有统一。一个项目可能后端是 Java，前端是 JavaScript，客户端是 Swift/C#，再混入各自生态。这对人类团队意味着高沟通成本和招聘门槛；对 AI 来说，也意味着它必须同时理解多套技术栈、协议框架、构建工具和部署工具。

Rust 很有趣的一点是：它既能写后端，也越来越适合前端和客户端。更重要的是，它性能高、产物轻、部署直观。

先看客户端。Electron 构建跨平台桌面应用一直有性能痛点：每个应用都捆绑完整 Chromium 内核和 Node.js 运行时。一个简单 Hello World 应用安装包也可能超过 150MB，启动慢，空闲内存轻易突破 300MB 到 500MB。

Rust 的 Tauri 摒弃内置浏览器内核，转而使用操作系统原生 WebView，同时把高消耗业务逻辑和系统 API 调用交给轻量 Rust 宿主进程。这让 Rust + Tauri 项目体积可以缩到 3MB 到 15MB，运行时内存通常也更低，性能更适合端侧长期运行。

在 Web 层面，Rust 占有率还不高，主要是生态支持还没完全成熟。但前端打包和构建工具已经大量使用 Rust；Rust 通过 WASM 也在前端侧持续跃迁。Dioxus、Leptos 等现代全栈框架已经能提供接近 React / Next.js 的同构渲染、服务端渲染和细粒度响应式体验。

如果激进一点，采用 Rust 可以更容易搭建前后端一体、覆盖端侧桌面的统一服务架构：

1. **数据模型复用：** 核心数据结构、验证规则和领域逻辑可以一次性编写，同时在后端、前端渲染和 Tauri 桌面端复用。
2. **消灭上下文断裂：** AI 不需要学习复杂跨语言通信协议，可以在统一强类型空间里完成从数据库查询、中间件逻辑到前端页面的垂直生成。

# 3. 编译即测试，强大的编译系统

前端页面直接白屏，是大家在 Vibe Coding 中经常遇到的问题。打开页面白屏，Console 里是 TypeError，然后只能把报错复制给 AI 反复修。点一会儿，又出现类似白屏。这就是典型运行时异常，尤其是类型崩溃导致的运行时问题。

![运行时 TypeError 导致白屏示例](/blog/why-hone-uses-rust-runtime-error.png)

在完全依赖 AI 多轮对话的开发模式下，最可怕的是 AI 的幻觉：它会生成语法结构和编码风格看似完美，但在深层语义、资源生命周期或并发状态上完全错误且隐蔽的代码。根据体感，很多问题都要到运行时才触发。

每一次 AI 陪伴的新功能迭代，都可能在未知角落引入新的运行时崩溃，项目被迫陷入“AI 生成 -> 运行时崩溃 -> 排查困难 -> 重新生成测试用例 -> 反复回归测试”的泥潭。这个问题不能完全靠换语言解决，但可以显著缓解。

Rust 的优势是把大量技术性运行时问题提前到编译期。Rust 的基础范式就是编译期安全护栏。GitHub 在《Why AI is pushing developers toward typed languages》中提到的核心观点也类似：

1. **AI 没有亲手编写代码时的隐式契约感。** 函数接不接受 null、返回 string 还是 buffer，AI 很容易断裂。
2. **类型错误占比很高，而且动态语言里很多问题要到运行时才出现。** Rust 的所有权和强类型系统能在模型生成代码后给出更早、更精确的反馈。
3. **Rust 正在成为底层复杂模块的基石。** AI 生成 Rust 代码时，一旦通过编译，稳定性就有更强保障。
4. **AI Coding 工具越来越依赖强类型反馈。** 类型错误能给模型提供最清晰的重试信号。

Rust 社区常说：如果它能通过编译，基本就已经正确了一大半。这种编译期防御力带来了现实红利。Google Android 团队的经验也显示，新代码迁移到 Rust 后，内存安全漏洞密度显著下降。在 AI Coding 时代，这种复利会更明显。

# 4. 性能与资源的提升

最近很火的 OpenClaw 被很多人吐槽：项目用 TS 写，代码库迅速膨胀，底层安全隔离不足，还曝出 RCE 和供应链攻击问题。性能上也有明显压力，比如单个 Agent 基础内存消耗高、启动延迟高，让很多开发者没法在普通硬件上舒服部署。

NEAR AI 团队随后推出了 Rust 重写的 IronClaw，聚焦安全和资源效率。

| 核心评估指标 | OpenClaw（TS/Node 架构） | IronClaw（Rust/WASM 架构） | 优化幅度 |
| --- | --- | --- | --- |
| 内存基线占用 | >1000MB | 约 5MB-7.8MB | 接近 120 倍提升 |
| 冷启动延迟 | >500ms | <10ms | 接近 50 倍提升 |
| 核心二进制体积 | 约 28MB，附加庞大运行时 | 3.4MB，单文件独立运行 | 极度精简 |
| 执行代码隔离机制 | Docker / 进程级隔离，高开销且权限蔓延风险高 | WebAssembly 沙箱无状态隔离，低开销、基于能力控制 | 安全与性能双赢 |
| 核心安全边界 | 依赖膨胀代码库的应用层逻辑判断 | 极小攻击面，网络层白名单拦截，多层防御 | 架构级跃升 |

# 5. 在最左侧大胆拥抱

在传统软件工程评估体系中，Rust 以内存安全、零成本抽象和机器码性能闻名，但陡峭学习曲线长期被视为快速迭代和开源社区采用的短板。

现在 AI Coding 已经非常普及，代码的经济学和效率模型发生了变化。当项目里大量基础代码、样板架构和复杂业务逻辑都可以由 AI 生成时，Rust 语法复杂度被大幅抹平，而编译器带来的安全特性被历史性放大。

在当前开源项目和企业级架构的初始技术选型中，果断采用 Rust 不是语言偏好之争，而是一种前瞻性的工程决策。它能带来更少、更精简的代码，更安全的运行时，更适合 AI 长上下文聚焦的工程结构，云、边、端统一的技术栈，以及极低的系统资源消耗。
